# Logs And Audit File Tracking Test Scenarios

This document is proposal-level architecture. It lists scenarios that should become acceptance tests, contract tests, integration tests, and operational review tests during implementation planning.

## Audit Record Scenarios

- Audit record requires source module.
- Audit record requires event/action type.
- Audit record includes actor/user/service where available.
- Audit record includes retention class.
- Audit record includes redaction class.
- Audit record links related record references without copying full source records.
- Sensitive audit record access is itself audited.

## File Tracking Scenarios

- File tracking record requires file name.
- File tracking record requires file type.
- File tracking record requires import/export direction.
- File tracking record captures source module.
- File tracking record captures vendor and buyer/entity scope where applicable.
- File tracking record captures date range where applicable.
- File tracking record captures generated by / uploaded by and generated at / uploaded at.
- File tracking record captures row count and failed row count.
- File tracking record captures validation status and processing status.
- File tracking record captures file version and checksum/hash.

## Vendor Operational Flow Scenarios

- Manual order export to vendor creates file tracking record.
- Manual return export to vendor creates file tracking record.
- Vendor shipping import creates file tracking and validation result records.
- Vendor return outcome import creates file tracking and processing result records.
- Duplicate vendor file creates duplicate detection record.
- Corrected vendor file creates correction/reupload history.
- Logs & Audit does not mutate fulfillment, return, routing, or invoice records.

## API Transmission Scenarios

- Failed API transmission creates transmission log.
- Retried API transmission appends retry/failure history.
- Completed API transmission records completion status.
- Sensitive request/response uses masked payload reference.
- Missing correlation id creates validation or review state.

## Payload And Privacy Scenarios

- Full payload storage denied by redaction policy.
- Masked payload reference accepted.
- Large file uses storage reference and metadata only.
- Customer-sensitive record is hidden from unauthorized user.
- Vendor user cannot view unrelated vendor payload references.

## Search Scenarios

- Search by date range returns scoped records.
- Search by company/entity applies authorization.
- Search by vendor excludes unrelated vendors.
- Search by source module returns audit and file records.
- Search by file type and direction returns import/export records.
- Search by error type returns failed processing or API transmissions.
- Search by redaction class requires permission.

## Retention Scenarios

- Retention class is required.
- Invalid retention class is rejected.
- Retention review required event is emitted.
- Retention review does not delete records by itself.
- AI cannot bypass retention rules.

## Boundary Scenarios

- Logs & Audit cannot create Product Catalog records.
- Logs & Audit cannot change Device Catalog identity.
- Logs & Audit cannot calculate Pricing values.
- Logs & Audit cannot choose routes or create suborders.
- Logs & Audit cannot mark shipment delivered.
- Logs & Audit cannot generate or finalize invoices.
- Logs & Audit cannot approve warranty claims.
- Logs & Audit cannot manage Tenant Company eligibility.
- Logs & Audit emits notification hooks but does not deliver notifications.
- Logs & Audit emits Analytics signals but does not define metrics.

## Open Test Questions

- Which source modules require contract tests first?
- Which vendor file workflows require end-to-end tests before launch?
- Which payload/redaction classes need security test coverage?
- Which searches are buyer/vendor-facing versus internal-only?

## Scheduled System Admin Activity Summary Evidence Test Scenarios (Cross-Module PR)

This section adds proposal-level test/review scenarios for the Logs & Audit File Tracking side of the cross-module summary email hardening pass. Scenarios are architecture-level walk-throughs for spec review.

### Scenario PR-C-LA-1: Activity Summary Generated Evidence recording

1. Analytics Workflow 5 completes with non-empty aggregation for Reporting Window W1 (Aggregation Record A1 produced).
2. Analytics signals Logs & Audit Workflow 10.
3. Workflow 10 creates Activity Summary Generated Evidence E1 with: `activity_summary_configuration_reference` (C1), `activity_summary_reporting_window_reference` (W1), `activity_summary_aggregation_record_reference` (A1), `generation_timestamp`, `actor_reference` (system service identity), `carry_forward_window_reference_collection` (mirrored from W1; may be empty), retention/redaction/access classes from existing patterns.
4. Workflow 10 also creates a canonical Audit Record via the existing Audit Record Creation Workflow, referencing E1.
5. Emits `audit.activity-summary-evidence.recorded`.

Expected: E1 is immutable; references resolve to canonical entities (C1 at Notification Platform Service, W1 and A1 at Analytics / Reporting); Audit Record retention pattern engaged.

### Scenario PR-C-LA-2: No-Activity Summary Suppression Evidence recording

1. Analytics Workflow 6 completes for Reporting Window W2 with zero activity. Analytics transitions W2 to `suppressed_no_activity` terminal state. **Analytics does NOT advance the configuration cursor.**
2. Analytics signals Logs & Audit Workflow 10.
3. Workflow 10 creates No-Activity Summary Suppression Evidence S1 with: `activity_summary_configuration_reference` (C1), `activity_summary_reporting_window_reference` (W2), `evaluation_timestamp`, `suppression_reason = no_source_facts_in_window`, `window_start_timestamp`, `window_end_timestamp`, `source_fact_zero_activity_assertion_reference`, retention/redaction/access classes. **S1 does NOT carry a cursor-advancement timestamp; the cursor mutation has not yet occurred at S1 creation time.**
4. Workflow 10 creates canonical Audit Record AR1 referencing S1. Emits `audit.activity-summary-suppression-evidence.recorded` (payload does NOT include a cursor-advancement timestamp). Analytics emits `analytics.activity-summary-window.evaluated` with `resultDiscriminator = suppressed_no_activity` (payload does NOT include a cursor-advancement timestamp).
5. Notification Platform Service consumes the suppression-outcome event and/or S1's reference via NPS Workflow 9 Trigger B path. NPS validates the suppression-outcome and advances the configuration cursor on C1 to W2's `window_end_timestamp`. NPS captures the cursor-advancement timestamp at the moment of mutation.
6. NPS signals Logs & Audit Workflow 10 again (trigger (e) Trigger B variant). Workflow 10 creates a NPS-side Audit Record AR2 via the existing Audit Record Creation Workflow with `source_module = notification-platform-service`, `event_action_type = activity_summary_cursor_advanced_on_consumed_no_activity_outcome`, `related_record_references` linking back to C1 (Activity Summary Configuration), W2 (Reporting Window), and S1 (No-Activity Summary Suppression Evidence). AR2 carries the `cursor_advancement_timestamp` value captured at NPS-side mutation.

Expected: S1 is immutable and does not require a field that did not exist at its creation time; AR2 retains the cursor-advancement audit trail; the absence of a delivery attempt is consistent with the no-activity outcome (no Activity Summary Delivery Attempt created for W2); the two retention surfaces (S1 + AR2) together form the full audit trail.

### Scenario PR-C-LA-3: Activity Summary Delivery Attempt retained by reference

1. NPS Workflow 7 dispatches Activity Summary Delivery Attempt D1 for Aggregation Record A1.
2. Notification Platform Service signals Logs & Audit Workflow 10.
3. Workflow 10 uses the existing Audit Record Creation Workflow to create an Audit Record with `source_module = notification-platform-service`, `event_action_type = activity_summary_delivery_attempted`, `related_record_references` pointing to D1.
4. No new Logs & Audit-side entity is created. D1 remains canonical at Notification Platform Service.
5. When D1 later transitions to `acknowledged` (via NPS Workflow 9) or `failed` (via NPS Workflow 8), additional Audit Records are created via Workflow 10 in the same reference pattern.

Expected: delivery attempt lifecycle traceable from Logs & Audit search via the Audit Record entity; no duplication of lifecycle ownership.

### Scenario PR-C-LA-4: Configuration lifecycle retained by reference

1. CIXCI System Admin creates an Activity Summary Configuration C1 (NPS Workflow 1).
2. Notification Platform Service signals Logs & Audit Workflow 10.
3. Workflow 10 creates an Audit Record with `source_module = notification-platform-service`, `event_action_type = activity_summary_configuration_created`, `related_record_references` pointing to C1.
4. Subsequent updates, pauses, retirements all produce additional Audit Records via the same pattern.

Expected: configuration lifecycle traceable from Logs & Audit search; no duplication of lifecycle ownership.

### Scenario PR-C-LA-5: Multiple consecutive failures and carry-forward audit trail

1. W1 (09:00-13:00) fails delivery. Workflow 10 creates Audit Record for `notification.activity-summary-delivery.failed`. W1 in `delivery_failed`.
2. W2 (09:00-18:00 via carry-forward of W1) fails delivery. Workflow 10 creates Audit Record. W2 in `delivery_failed`. W1 remains in `delivery_failed`.
3. W3 (09:00-09:00-next-day via carry-forward of W1 and W2) succeeds delivery. Workflow 10 creates:
   - Activity Summary Generated Evidence E3 referencing W3 and Aggregation Record A3. E3's `carry_forward_window_reference_collection` includes W1 and W2.
   - Audit Record for `notification.activity-summary-delivery.succeeded`.
   - Audit Records for the supersession transitions (W1 -> superseded, W2 -> superseded).

Expected: full audit trail across three windows preserved; carry-forward visibility on E3 enables direct Logs & Audit search without requiring Analytics queries.

### Scenario PR-C-LA-6: Amendment of an evidence record

1. Activity Summary Generated Evidence E1 was created with a typo in `actor_reference` (implementation detail; PR-C does not enforce specific actor reference format).
2. An authorized actor requests amendment via the existing Audit Amendment Workflow.
3. Existing Audit Amendment Workflow records the amendment as a new evidence record (or amendment record per existing patterns). E1's original content remains unchanged. The amendment record references E1 and the corrected value.

Expected: immutability preserved; correction is via amendment, not in-place edit; existing Logs & Audit amendment patterns continue to apply.

### Scenario PR-C-LA-7: Search across new evidence types

1. CIXCI System Admin uses existing Logs & Audit search to find all No-Activity Summary Suppression Evidence records for the past month.
2. Existing Logs & Audit search/filtering applies tenant, entity, role, redaction class, retention class, access class, source-module, and date-range filters.
3. Results return E2 records with references to the corresponding Reporting Windows and Configurations.

Expected: existing search patterns sufficient for new evidence types; cross-tenant search denied by default per existing rules; sensitive access of suppression evidence is itself auditable.

### Cross-module verification checklist for Logs & Audit File Tracking side

For each scenario, reviewers should confirm:

- Activity Summary Generated Evidence and No-Activity Summary Suppression Evidence are immutable per existing Logs & Audit rules.
- Activity Summary Delivery Attempt lifecycle is NOT duplicated as a Logs & Audit-side entity; retention is via existing Audit Record entity pattern.
- Activity Summary Configuration lifecycle is NOT duplicated as a Logs & Audit-side entity; retention is via existing Audit Record entity pattern.
- Existing retention class, redaction class, and access class patterns apply unchanged.
- No new retention, redaction, or access class is introduced.
- Source-module records are never mutated by Logs & Audit (PR-C reaffirms this for the summary domain).
- The No-Activity Summary Suppression Evidence record is created **before** NPS performs cursor mutation; it does NOT carry a cursor-advancement timestamp. The cursor-advancement audit trail lives on a separate NPS-side Audit Record created via the existing Audit Record entity pattern after NPS Workflow 9 Trigger B path performs the cursor mutation. Together the two retention surfaces form the full audit trail across the choreography; neither requires a field that did not exist at its own creation time. Logs & Audit retains both records; NPS is the canonical writer for the cursor mutation and its post-mutation Audit Record.
- The `carry_forward_window_reference_collection` field on Activity Summary Generated Evidence mirrors the Analytics-side field for direct Logs & Audit search traceability.

## PR-A Test Scenarios - Core Evidence Spine

This section defines proposal-level acceptance scenarios for PR-A. All scenarios are architecture-level. Concrete test implementation is deferred. Scenarios are organized by concept area.

### Audit Record hardening scenarios

**Scenario AR-1: Audit Record created with formalized reference triad.**

- Given a source-module service emits an audit-worthy action.
- When the source module submits an Audit Record creation request.
- Then Logs & Audit creates the Audit Record with `source_module_reference`, `source_record_reference`, `source_snapshot_reference` (when applicable), `actor_reference` (when human-actor), `service_trigger_reference` (when service-initiated), and `company_scope_reference` populated.
- And `audit.record.created` is emitted with the same references in payload.
- And all existing baseline fields (status, timestamp, retention_class, redaction_class, access_class, correlation_id, idempotency_key, etc.) remain present.

**Scenario AR-2: Audit Record created without source_record_reference (pure observation event).**

- Given a source-module service emits a read-access event with no operational record to reference.
- When the source module submits an Audit Record creation request without `source_record_reference`.
- Then Logs & Audit creates the Audit Record successfully.
- And `source_record_reference` is null on the Audit Record.
- And `audit.record.created` is emitted with `source_record_reference` absent or null.

**Scenario AR-3: Audit Record with both actor_reference and service_trigger_reference.**

- Given a service action is authorized by a human user (e.g., System Admin triggers a scheduled export).
- When the source module submits an Audit Record creation request with both `actor_reference` (the System Admin) and `service_trigger_reference` (the scheduled-export service).
- Then Logs & Audit creates the Audit Record with both references populated.
- And subscribers may distinguish the human authorization from the service execution.

**Scenario AR-4: Audit Record with optional evidence_record_reference back-link.**

- Given an Audit Record is created and an Evidence Record is attached.
- When PR-A Workflow 1 completes Evidence Record creation.
- Then the Audit Record's `evidence_record_reference` is populated with the Evidence Record's identifier.
- And the Audit Record can be queried to navigate to the attached Evidence Record.

### Evidence Record creation scenarios

**Scenario ER-1: Evidence Record created with all PR-A metadata fields.**

- Given a source module emits evidence intent with `evidence_type`, source references, actor reference, scope reference, hash, attachment reference, retention/redaction/access classes.
- When Logs & Audit creates the Evidence Record per PR-A Workflow 1.
- Then the Evidence Record carries `evidence_record_id`, `evidence_schema_version`, `evidence_type`, `audit_record_reference`, source references, optional external reference, actor and scope references, `evidence_hash_reference`, `captured_at`, `created_at`, `created_by_actor_reference` or `created_by_service_trigger_reference`, `correlation_reference`, optional `trace_reference`, optional `idempotency_key`, optional `replay_safe_dedupe_reference`, `evidence_status = active`, classification fields, optional `restricted_evidence`, and `audit_reference`.
- And `audit.evidence.recorded` is emitted with the `evidence_type` discriminator and the same fields in payload.

**Scenario ER-2: Evidence Record `captured_at` distinct from `created_at`.**

- Given a source module observes an event at time T1 and submits evidence to Logs & Audit at time T2 (where T2 > T1).
- When Logs & Audit creates the Evidence Record.
- Then `captured_at` = T1 (source-module-observed time).
- And `created_at` = T2 (Logs & Audit-recorded time).
- And both timestamps travel on the `audit.evidence.recorded` event.

**Scenario ER-3: Evidence Record idempotent submission.**

- Given a source module submits an Evidence Record with `idempotency_key = K1`.
- When Logs & Audit creates the Evidence Record successfully.
- And the source module retries the same submission with `idempotency_key = K1`.
- Then Logs & Audit returns the previously created Evidence Record (no duplicate creation).
- And `audit.evidence.recorded` is emitted exactly once for the original creation.

**Scenario ER-4: Evidence Record replay_safe_dedupe_reference dedup.**

- Given a source module submits an Evidence Record with `replay_safe_dedupe_reference = R1` (a structured business-key dedupe reference, e.g., import-job-id + step-id + attempt-number).
- When the same business key is submitted again (potentially with a different idempotency_key due to retry by a different actor).
- Then Logs & Audit recognizes the replay-safe dedupe match and returns the previously created Evidence Record.
- And no duplicate event is emitted.

### Source Module / Source Record / Source Snapshot Reference scenarios

**Scenario SR-1: Source Module Reference required at creation.**

- Given a source module attempts to create an Audit Record or Evidence Record without `source_module_reference`.
- When Logs & Audit receives the submission.
- Then the submission is rejected with a missing-required-reference error (concrete error code is implementation-level).
- And no Audit Record or Evidence Record is created.

**Scenario SR-2: Source Snapshot Capture follows minimization rule.**

- Given a source module elects to attach a source snapshot.
- When the source module prepares the snapshot.
- Then the snapshot contains selected metadata, hashes, and references (NOT a full operational record copy).
- And the snapshot hash is computed.
- And Logs & Audit records the `source_snapshot_reference` on the Evidence Record.
- And the snapshot payload shape is architecture-level (not OpenAPI-specific).

**Scenario SR-3: Full snapshot requires Full Payload Exception Record.**

- Given a source module needs to store a full operational record copy as evidence (exceptional case).
- When the source module submits the snapshot.
- Then Logs & Audit requires the existing baseline Full Payload Exception Record workflow (approval, reason, retention class, redaction class, access class, payload size checks).
- And without exception approval, the submission is rejected per the existing baseline `FULL_PAYLOAD_EXCEPTION_REQUIRED` error code.

### External Evidence Reference scenarios

**Scenario EER-1: External Evidence Reference sub-structure populated on Evidence Record.**

- Given Integration Management performs an external system call that produces external coordination evidence.
- When the source-module service (or Integration Management) submits an Evidence Record with `external_evidence_reference` sub-structure.
- Then the Evidence Record carries the sub-structure with `provider_response_reference`, `external_id_reference`, optional `external_file_reference`, optional `external_task_project_reference`, `external_timestamp`, optional `webhook_receipt_reference`, optional `retry_failure_evidence_reference`, optional `external_evidence_hash`.
- And `audit.evidence.recorded` is emitted with the sub-structure in payload.
- And NO separate `audit.external-evidence-reference.recorded` event is emitted.

**Scenario EER-2: External Evidence Reference does not make external system source of truth.**

- Given an Evidence Record carries an `external_evidence_reference` with `external_id_reference` pointing to QuickBooks (or other external system).
- When an investigator queries the Evidence Record for canonical business state.
- Then the investigator must resolve to the CIXCI source-module record (via `source_record_reference`), NOT to the external system.
- And the External Evidence Reference is coordination/proof only.

### Evidence Amendment scenarios

**Scenario EA-1: Evidence Amendment appended without mutating target.**

- Given an Evidence Record E1 exists with `evidence_status = active`.
- When an authorized actor submits an Evidence Amendment against E1 (e.g., typo correction, additional context).
- Then Logs & Audit creates an Evidence Amendment Record A1 referencing E1 as target.
- And A1 is appended to E1's `amendment_record_reference_collection`.
- And E1's content is NOT mutated.
- And E1's `evidence_status` transitions from `active` to `amended_with_amendments` (soft transition).
- And `audit.evidence-amendment.recorded` is emitted.

**Scenario EA-2: Multiple amendments on the same Evidence Record.**

- Given an Evidence Record E1 has one prior amendment A1.
- When a second amendment A2 is submitted against E1.
- Then Logs & Audit creates A2 and appends it to E1's `amendment_record_reference_collection`.
- And E1's collection now contains [A1, A2].
- And A1 is NOT mutated by the creation of A2.
- And E1 is NOT mutated.

**Scenario EA-3: Amendment carries actor and reason references.**

- Given an Evidence Amendment is submitted.
- When Logs & Audit creates the Evidence Amendment Record.
- Then the record carries `amendment_actor_reference` (or `amendment_service_trigger_reference`), `amendment_reason_reference`, `tenant_company_authority_reference`, optional `amendment_payload_reference`, `created_at`, and `audit_record_reference` (the audit for the amendment action itself).

### Evidence Supersession scenarios

**Scenario ES-1: Supersession creates new Evidence Record and lineage record.**

- Given an Evidence Record E1 exists with `evidence_status = active` from a prior import.
- When the source module corrects the operational record and submits an Evidence Supersession intent with new evidence content.
- Then Logs & Audit creates a new Evidence Record E2 via PR-A Workflow 1 with `evidence_status = active`.
- And Logs & Audit creates an Evidence Supersession Record S1 linking E1 (prior) to E2 (new).
- And E1's `evidence_status` transitions from `active` to `superseded`.
- And E1's `supersession_reference` is populated pointing to S1.
- And both E1 and E2 remain in storage and remain queryable.
- And `audit.evidence-supersession.recorded` is emitted.
- And `audit.evidence.recorded` is emitted for E2 creation.

**Scenario ES-2: Supersession lineage preserves prior evidence.**

- Given an investigator queries the historical evidence for a corrected source-module record.
- When the investigator follows the supersession lineage from E2 backward.
- Then the investigator can navigate via `supersession_reference` to E1 (the prior superseded evidence).
- And E1's full content is retrievable (`evidence_status = superseded` does NOT remove it from storage).

**Scenario ES-3: Supersession-of-already-superseded (chain).**

- Given Evidence Record E1 was superseded by E2.
- When the source module performs a second correction and submits a supersession of E2 by E3.
- Then Logs & Audit creates E3 with `evidence_status = active`.
- And E2's `evidence_status` transitions to `superseded`.
- And a new Supersession Record S2 links E2 to E3.
- And the chain E1 -> E2 -> E3 is fully queryable via supersession_reference chains.

**Scenario ES-4: Supersession is distinct from amendment.**

- Given an Evidence Record E1 exists.
- When the underlying source-module record is corrected (vs. when only the evidence record needs metadata clarification).
- Then the source module emits a Supersession (creates new E2) rather than an Amendment (appends to E1).
- And the choice is the source module's decision; Logs & Audit records the chosen lineage.

**Scenario RRA-1: retention_class assigned at creation.**

- Given an Evidence Record is created with `retention_class = standard`.
- When Logs & Audit creates the record.
- Then `retention_class = standard` is locked at creation.
- And PR-A does NOT modify the class post-creation.
- And future PR-D review workflows MAY upgrade or downgrade with audit trail.

**Scenario RRA-2: redaction_class assigned at creation with baseline values.**

- Given an Evidence Record is created with `redaction_class = pricing_sensitive_restricted`.
- When Logs & Audit creates the record.
- Then `redaction_class = pricing_sensitive_restricted` is locked at creation.
- And the value matches one of the existing baseline redaction classes (preserved verbatim).
- And the value travels on `audit.evidence.recorded` event.

**Scenario RRA-3: access_class assigned at creation.**

- Given an Evidence Record is created with `access_class = system_admin_only`.
- When Logs & Audit creates the record.
- Then `access_class = system_admin_only` is locked at creation.
- And PR-D will define the full access matrix for which roles can read which access_class values.

**Scenario RRA-4: restricted_evidence flag set at creation.**

- Given an Evidence Record is created with `restricted_evidence = true` (because the evidence content is operationally sensitive).
- When Logs & Audit creates the record.
- Then `restricted_evidence = true` is recorded on the Evidence Record.
- And PR-D will define the full gating matrix for restricted_evidence reads.

**Scenario RRA-5: legal_hold_reference placeholder remains null at PR-A creation.**

- Given an Evidence Record is created.
- When Logs & Audit creates the record.
- Then `legal_hold_reference = null` (PR-A placeholder; populated by future PR-D workflows).
- And `evidence_status` is NOT `legal_hold` at PR-A creation.

### evidence_type discriminator scenarios

**Scenario ET-1: Evidence Record carries evidence_type.**

- Given a source module submits evidence for an import workflow.
- When Logs & Audit creates the Evidence Record.
- Then `evidence_type` is populated with one of the proposal-level starter set values (e.g., `import_evidence`).
- And `audit.evidence.recorded` carries the discriminator.

**Scenario ET-2: Subscribers filter by evidence_type.**

- Given multiple Evidence Records are created with different `evidence_type` values across multiple source modules.
- When a subscriber filters `audit.evidence.recorded` events by `evidence_type = export_evidence`.
- Then only the export evidence events are delivered to the subscriber.
- And NO separate event stream (e.g., no `audit.export-evidence.recorded`) is required.

**Scenario ET-3: PR-A does not pre-define comprehensive evidence_type taxonomy.**

- Given PR-A's spec documents a starter set of foundation values.
- When a source module needs to emit evidence of a type not in the starter set.

### Evidence-Per-Lifecycle-Step Rule scenarios (canonical, new in PR-A)

**Scenario EPLS-1: Canceled product import creates evidence.**

- Given a vendor uploads a product import file and reaches validation but cancels before commit.
- When Product Catalog completes the cancel-before-commit transition.
- Then the source module emits evidence (file upload evidence + validation evidence + canceled-before-commit evidence) per the Evidence-Per-Lifecycle-Step Rule.
- And the evidence is retained per `retention_class` (PR-D defines duration; PR-A defaults to indefinite).

**Scenario EPLS-2: Failed buyer API product export creates evidence.**

- Given a buyer API product export attempt fails before delivery (e.g., network error, validation failure, no eligible products).
- When the source module records the failure.
- Then evidence is created for the failed export even though no buyer system received the payload.
- And the evidence carries `export_evidence` with a PR-C-cataloged sub-type for "failed_export".

**Scenario EPLS-3: Failed vendor email export creates evidence.**

- Given a vendor email export is generated and Notification Platform attempts delivery.
- When the email delivery fails (e.g., vendor mailbox unreachable, bounce, rejection).
- Then evidence is created for the delivery attempt and the failure.
- And the evidence carries the External Evidence Reference sub-structure with `retry_failure_evidence_reference` and the delivery-attempt context.

**Scenario EPLS-4: Canceled vendor import retains file and validation evidence.**

- Given a vendor uploads a shipping import file and the import is canceled (by user action or timeout).
- When Fulfillment / Returns marks the import canceled.
- Then file upload evidence and validation evidence are retained per `retention_class`.
- And subsequent retry / reprocess attempts link back to the original evidence via Evidence Supersession Record lineage.

**Scenario EPLS-5: Retry of canceled import links via supersession.**

- Given a canceled vendor shipping import has retained file and validation evidence.
- When the vendor or System Admin retries the import successfully.
- Then a new Evidence Record is created for the retry attempt.
- And an Evidence Supersession Record links the original (now superseded) evidence to the new (active) evidence.

### Source-of-truth boundary scenarios

**Scenario SOT-1: Logs & Audit does not validate source-module business logic.**

- Given a source module submits an Evidence Record with potentially incorrect business semantics (e.g., invalid status value per the source module's own validation rules).
- When Logs & Audit receives the submission.
- Then Logs & Audit creates the Evidence Record as received (no business-logic validation).
- And source modules own validation; Logs & Audit observes.

**Scenario SOT-2: Logs & Audit does not mutate source-module records.**

- Given an attempt to invoke Logs & Audit in a way that would mutate a source-module record (e.g., to "correct" an order line item via audit amendment).
- When Logs & Audit receives the request.
- Then Logs & Audit rejects the request as a boundary violation (existing baseline `SOURCE_BOUNDARY_VIOLATION` error code).
- And only the source module can mutate its own records.

**Scenario SOT-3: Logs & Audit does not become operational source of truth.**

- Given investigators are tempted to query Logs & Audit as the source of truth for operational state.
- When they need canonical business state.
- Then they must resolve to the source module (via `source_record_reference`).
- And Logs & Audit holds evidence, not operational state.

### External-Tool-Not-Source-of-Truth Rule scenarios

**Scenario ETNSOT-1: External system reference does not promote external system to source of truth.**

- Given an Evidence Record has `external_evidence_reference` with `external_id_reference` pointing to a vendor portal.
- When a downstream consumer queries for the canonical state of the underlying CIXCI record.
- Then the consumer resolves to the CIXCI source module (via `source_record_reference`), NOT to the vendor portal.
- And the vendor portal is coordination/evidence only.

**Scenario ETNSOT-2: External tool downtime does not invalidate evidence.**

- Given an Evidence Record was created with external_evidence_reference at time T1.
- When the external tool is later unavailable or decommissioned.
- Then the Evidence Record remains valid (the references are stored; the external tool resolution is implementation-level).
- And the CIXCI source-module record continues to be the canonical source of truth.

### Immutable Evidence Rule scenarios

**Scenario IER-1: Evidence Record is append-only.**

- Given an Evidence Record exists.
