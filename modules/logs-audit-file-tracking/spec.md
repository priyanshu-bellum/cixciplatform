# Logs And Audit File Tracking Specification

This document is proposal-level architecture. It defines the initial Logs & Audit File Tracking module shape without finalizing implementation behavior or moving business ownership into this platform service.

## Purpose

Logs & Audit provides platform-wide traceability for imports, exports, API transmissions, file processing, validation results, retries, failures, user actions, manual vendor file workflows, and operational audit evidence.

The module tracks:

- Audit record model.
- File tracking record model.
- API transmission log model.
- Import/export tracking.
- Validation result tracking.
- Processing result tracking.
- Retry/failure history.
- User-trigger history.
- Source file references.
- Payload reference or masked payload reference.
- Retention class.
- Redaction class.
- Search/filtering.
- Duplicate file detection.
- Correction/reupload history.
- Manual vendor file workflows.

## Scope

### In Scope

- Record audit events from platform modules.
- Track import/export files, manual uploads/downloads, and API transmission attempts.
- Preserve file names, file types, directions, source modules, vendors, buyer/entity scopes, date ranges, actors, timestamps, row counts, failed row counts, accepted row counts, validation status, processing status, error summaries, file versions, checksum/hash values, duplicate detection, and correction/reupload history.
- Track manual export of order data to vendors.
- Track manual export of return data to vendors.
- Track vendor import of shipping information.
- Track vendor import of return outcome information.
- Store masked payload references by default and full payload references only by approved exception.
- Provide audit search and filtering by date range, company, child entity, vendor, buyer, source module, file type, API event type, status, actor/user, error type, and direction.
- Emit audit, file, and API transmission events.
- Provide signals to AI Agent Services and future Notification hooks.

### Out of Scope

- Product Catalog source-record ownership.
- Device Catalog source-record ownership.
- Pricing calculation, interpretation, or snapshot ownership.
- Order Routing decisions.
- Fulfillment/Returns operational state.
- Invoice records, invoice generation, or reconciliation decisions.
- Warranty claim state or warranty approval.
- Tenant eligibility, hierarchy, roles, or permissions ownership.
- AI recommendation ownership or agent action decisions.
- Notification delivery.
- Analytics reporting definitions.
- Vendor Integration ownership, including vendor credentials, transports, API endpoints, schedules, file mappings, vendor-specific transforms, delivery mechanics, or vendor integration setup/configuration.
- Unrestricted sensitive payload data lake behavior.

## Core Rule

Logs & Audit owns traceability evidence only. Source modules remain the owners of the business records, operational workflows, and decisions that Logs & Audit observes.

Logs & Audit must not mutate source records, correct business data, resolve operational exceptions, approve workflow outcomes, or bypass source-module ownership boundaries.

## Audit Immutability Model

Proposal-level immutability rules:

- Audit records are append-only.
- File tracking records are append-only.
- API transmission logs are append-only.
- Validation results are append-only.
- Processing results are append-only.
- Corrections use amendment records, supersession references, correction references, or reupload history records.
- No silent rewrite of audit evidence is allowed.
- Sensitive audit access/view activity should itself be auditable.
- Review/status metadata may be added only through explicit amendment or review records.

## Payload And Privacy Guardrails

Masked payload reference is the default.

Full payload reference is exception-based and should require approval, reason, retention class, redaction class, access class, and payload size checks.

Sensitive customer, order, pricing, invoice, warranty, tenant, media, licensing, commercial, user, and operational data must not appear in events unless explicitly allowed by policy and consumer class.

Full payload storage is proposal-level and subject to privacy, contract, file size, retention, redaction, encryption/access class, and security rules.

Logs & Audit should prefer references, masked payload references, checksums, row counts, validation summaries, and error summaries over full sensitive payload copies where possible.

Logs & Audit must not become an unrestricted data lake of sensitive payloads.

## PR-A Specification - Core Evidence Spine

This section specifies the Core Evidence Spine layered on top of the existing Logs & Audit File Tracking baseline. All baseline content (Audit Record, File Tracking Record, API Transmission Log, Validation Result Record, Row-Level Validation Error Record, Partial File Processing Record, Audit Access Record, vendor operational flow records, signal records, immutability rules, payload guardrails, retention governance, search guardrails, etc.) is preserved without rename or removal.

### Scope summary

PR-A addresses five concept areas as a single cohesive spine:

1. Audit Record hardening (formalizing the reference triad and the Actor / Service Trigger / Company Scope reference types).
2. Generic Evidence Record introduction (one entity with `evidence_type` discriminator, plus the PR-A metadata field set).
3. Source Module / Source Record / Source Snapshot Reference and External Evidence Reference formalization.
4. Evidence Amendment Record (standardized from baseline) and Evidence Supersession Record (new) as the two distinct lineage mechanisms.
5. Retention / Redaction / Access foundation (class enumerations only, plus `legal_hold_reference` placeholder).

PR-A does NOT define the comprehensive evidence_type taxonomy (PR-C), retention duration matrix (PR-D), redaction transformation workflow (PR-D), legal hold lifecycle (PR-D), access matrix (PR-D), or search workflows (PR-E). PR-A does NOT modify `openapi-contracts.md` or any source-module file.

### Audit Record hardening

Logs & Audit File Tracking owns the Audit Record. PR-A preserves the existing baseline field set and adds the following formalizations.

- **Source Module Reference, Source Record Reference, Source Snapshot Reference triad.** Previously expressed as the loose "Source module" and "Related record references" fields on Audit Record. PR-A formalizes the three references as a canonical triad. The triad is REQUIRED on Audit Record for any evidence-bearing observation; it is OPTIONAL for pure observation events (read-only access logs, permission checks) where there is no operational record to reference. Source modules populate the triad at Audit Record creation. The Source Snapshot Reference is governed by the Source Snapshot Minimization Rule (see below).
- **Actor Reference and Service Trigger Reference as distinct.** Previously combined in "Actor/user/service" on Audit Record. PR-A separates them. An Audit Record carries `actor_reference` when the originator is a human user, or `service_trigger_reference` when the originator is a non-human service (scheduler, webhook handler, agent runtime, internal service). Both may be populated when a service action was authorized by a human; only one is required.
- **Company Scope Reference.** Previously "Company/entity scope" on Audit Record. PR-A formalizes the reference type. The reference is REQUIRED on every Audit Record. The reference captures vendor / buyer / parent / child company scope by reference to a Tenant Company scope record.
- **Optional `evidence_record_reference` back-link.** PR-A adds an optional field on Audit Record pointing to the attached Evidence Record (when one exists). Some Audit Records have no attached Evidence Record (e.g., a read-access event captures only the Audit Record); for those, `evidence_record_reference` is null.
- **All existing baseline fields preserved.** Status, timestamp, before/after summary placeholder, validation result reference, retry count, error code/message, retention owner, retention class, redaction class, access class, legal/contract hold, purge eligibility, expiration behavior placeholder, review status, correlation id, idempotency key. PR-A does NOT remove or rename any baseline field.

### Generic Evidence Record introduction

Logs & Audit File Tracking introduces a single new entity, **Evidence Record**, with an `evidence_type` discriminator. Every future cross-module evidence record (product import / export, buyer download, vendor email, vendor shipping import, media versioning, AI agent action, SLA exception, handoff consumption, etc.) conforms to this entity.

- **Audit Record and Evidence Record separation rule (canonical).** Audit Record is the observation: "an audit-worthy thing happened." Evidence Record is the artifact: "here is the proof of what happened." The two are distinct entities; Audit Record CAN exist without Evidence Record (pure observation events); Evidence Record ALWAYS references its parent Audit Record via `audit_record_reference`. The Audit Record's optional `evidence_record_reference` may point back to the Evidence Record when one exists.
- **`evidence_type` discriminator.** Every Evidence Record carries a non-empty `evidence_type`. PR-A documents a small proposal-level starter set of foundation values as examples only: `module_action_evidence`, `file_evidence`, `api_transmission_evidence`, `import_evidence`, `export_evidence`, `summary_evidence`. The comprehensive taxonomy (Product Import Evidence, Buyer API Product Export Evidence, Vendor Email Export Evidence, Vendor Shipping Import Evidence, Media Evidence subtypes, AI Agent Evidence subtypes, SLA Evidence subtypes, Handoff Evidence subtypes) is **PR-C scope**; PR-A does NOT lock the taxonomy.
- **`evidence_type` is NOT a stand-alone entity for any subtype in PR-A.** PR-A explicitly does NOT create separate top-level entities for Product Import Evidence, API Product Export Evidence, Vendor Email Export Evidence, Vendor Shipping Import Evidence, Media Evidence, SLA Evidence, Handoff Evidence, AI Agent Evidence. Each future evidence subtype is an `evidence_type` value on the Generic Evidence Record.
- **Evidence Record metadata field set.** See `data-model.md` PR-A section for the complete field list. The PR-A metadata fields added per Codex cleanup directive are: `evidence_schema_version`, `captured_at`, `source_event_reference`, `correlation_reference`, `trace_reference`, `idempotency_key`, `replay_safe_dedupe_reference`.

### Source Module / Source Record / Source Snapshot Reference

PR-A formalizes the three references as reference TYPES, NOT stored entities.

- **Source Module Reference.** Identifies the originating module. Proposal-level structured form: module name selected from a Logs & Audit-maintained source module catalog. The concrete catalog values are implementation-level; PR-A documents the architectural surface only.
- **Source Record Reference.** Identifies the operational record this evidence pertains to. Proposal-level structured form: module identifier + record type + record id. The reference does NOT carry the record content; it carries the identifier.
- **Source Snapshot Reference.** Reference to a minimized snapshot of the source record at the time the evidence was created. The snapshot itself may be stored as an attachment via `evidence_attachment_reference`; the Source Snapshot Reference is the identifier and shape descriptor.

**Source Snapshot Minimization Rule (canonical, new in PR-A).** Source snapshots should use selected metadata, hashes, and references when possible. Source snapshots should NOT blindly copy full operational records. Snapshots are evidence, not operational truth. Full operational record copies are exceptional and gated by the existing baseline Full Payload Exception Record mechanism (which requires approval, reason, retention class, redaction class, access class, and payload size checks).

Source-module decision: each source module is responsible for the minimization at evidence emission time. Logs & Audit documents the expected shape but does not enforce minimization at write time (enforcement is implementation-level and outside PR-A scope).

### External Evidence Reference

PR-A formalizes External Evidence Reference as a **sub-structure on Evidence Record**, NOT a standalone entity.

The sub-structure carries:

- `provider_response_reference` - reference to a provider response from an external system call.
- `external_id_reference` - the external system's own identifier for the coordinated record.
- `external_file_reference` - reference to a file stored or delivered externally.
- `external_task_project_reference` - reference to an external task, project, or workflow item.
- `external_timestamp` - the timestamp the external system reported.
- `webhook_receipt_reference` - reference to a webhook receipt event from the external system.
- `retry_failure_evidence_reference` - reference to retry / failure transport evidence captured by Integration Management.
- `external_evidence_hash` - hash of external evidence content where available.

**External-Tool-Not-Source-of-Truth Rule (canonical, new in PR-A).** Integration Management owns connector execution and transport. External evidence references are coordination and proof only. External systems must NEVER become CIXCI operational source of truth. Investigators and source-module logic must trace back to CIXCI source-module records, not to external system records, for canonical business state.

### Actor Reference, Service Trigger Reference, Company Scope Reference

PR-A formalizes the three as reference TYPES to Tenant Company records, NOT stored Logs & Audit entities.

- **Actor Reference.** Identifies the human user actor. Reference to a Tenant Company user record. Tenant Company is NOT modified by PR-A. The reference value is opaque to Logs & Audit; investigators resolve to Tenant Company for the canonical user identity.
- **Service Trigger Reference.** Identifies the non-human service actor (scheduler, webhook handler, agent runtime, internal service, integration connector, scheduled policy actor). Distinct from Actor Reference. The reference value is opaque to Logs & Audit; investigators resolve to an internal service registry (not part of PR-A).
- **Company Scope Reference.** Captures vendor / buyer / parent / child company scope. Reference to a Tenant Company scope record. Tenant Company is NOT modified by PR-A.

**No new role, capability flag, or scope.** PR-A authority-bearing actions (evidence creation, amendment submission, supersession authorization) flow through existing Tenant Company `check_access` patterns and existing baseline permission families.

### Evidence Amendment Record (formalized from baseline)

The existing baseline entity "Audit Evidence Amendment Record" is referenced in PR-A under the standardized canonical name **Evidence Amendment Record**.

**Promotion-of-Naming Rule (one-time clarification).** PR-A does NOT rename or remove the existing baseline "Audit Evidence Amendment Record" entity. PR-A documents once that the canonical PR-A name for the concept is "Evidence Amendment Record" and uses it forward. Existing baseline workflow and data-model language referring to "Audit Evidence Amendment Record" is preserved without rewrite.

Evidence Amendment Record is append-only small correction to an existing Evidence Record. Use cases include: typo in evidence metadata, missing field added, additional context recorded after the fact, actor name disambiguated, redaction class clarified post-creation.

- The original Evidence Record retains its `evidence_status = active`.
- The amendment is attached to the Evidence Record via `amendment_record_reference_collection`.
- The Evidence Record is NEVER mutated.
- The amendment carries its own actor reference, reason reference, timestamp, audit reference.

### Evidence Supersession Record (new)

Evidence Supersession Record is append-only record that a new Evidence Record (with its own id and full field set) supersedes a prior one because the underlying source-module record was corrected, reissued, or replaced.

- The prior Evidence Record transitions to `evidence_status = superseded`.
- The new Evidence Record is `evidence_status = active`.
- Both records remain in storage and remain queryable.
- The Evidence Supersession Record links the two via `prior_evidence_record_reference` and `new_evidence_record_reference`.
- The source-module correction reference is captured for traceability.

**Amendment vs Supersession Distinction Rule (canonical, new in PR-A).** Amendment is small correction to existing evidence (no new "current" evidence; the amendment is metadata attached to the existing record). Supersession is creation of a new Evidence Record that supersedes a prior one (prior transitions to superseded; both remain in storage). They are different lifecycles. Source modules choose which mechanism applies based on whether the underlying source-module record was corrected (supersession) or whether the evidence record itself needs clarification (amendment).

Logs & Audit records lineage; source modules own the business correction decision.

### Retention / Redaction / Access foundation

PR-A formalizes the class enumerations at the *class* level. The existing baseline values are preserved.

- **`retention_class`** - enumeration on Audit Record and Evidence Record. Proposal-level values: `transient` (short-term operational logs), `standard` (default), `extended` (longer retention for compliance), `regulatory` (formal regulatory hold), `legal_hold` (under legal hold), `audit_critical` (immutable for full audit trail). PR-D will define the full duration matrix; PR-A captures only the enumeration shape.
- **`redaction_class`** - enumeration on Audit Record and Evidence Record. Existing baseline values are preserved verbatim: `public_metadata_placeholder`, `buyer_visible_audit`, `vendor_visible_audit`, `internal_operations`, `customer_sensitive_restricted`, `pricing_sensitive_restricted`, `invoice_sensitive_restricted`, `warranty_sensitive_restricted`, `tenant_security_restricted`, `audit_only`. PR-D will define redaction transformations; PR-A captures only the enumeration shape.
- **`access_class`** - enumeration on Audit Record and Evidence Record. Proposal-level values: `public_metadata`, `buyer_visible`, `vendor_visible`, `internal_operations`, `system_admin_only`, `compliance_only`. PR-D will define the full access matrix (tenant / parent / child); PR-A captures only the enumeration shape.
- **`restricted_evidence` flag** - boolean on Evidence Record. When true, the evidence is restricted from normal access (typically System-Admin-only or stricter). The full gating matrix is PR-D.
- **`raw_evidence_reference`** - optional reference to the raw, unredacted evidence artifact. Stored alongside any future `redacted_view_reference` projections; redaction creates a view, NOT a destructive edit.
- **`redacted_view_reference`** - optional reference to a redacted projection of the raw evidence. Populated by future PR-D redaction workflows. PR-A defines only the field.
- **`legal_hold_reference` placeholder** - field on Evidence Record. The Legal Hold entity, lifecycle, application workflow, and lift workflow are PR-D. PR-A introduces only the placeholder field.

**At-creation classification rule (canonical).** Source modules and Logs & Audit assign `retention_class`, `redaction_class`, and `access_class` AT EVIDENCE CREATION. PR-D may upgrade or downgrade the class via review workflows (with audit trail), but PR-A captures only the at-creation assignment. Misassignment is a source-module concern; PR-D provides the review surface.

### Evidence integrity hash recording

PR-A's Evidence Record carries `evidence_hash_reference` capturing the hash of the evidence payload (or attachment) at creation. The hash algorithm is implementation-level (likely SHA-256 or comparable; PR-A does NOT specify).

**Periodic integrity verification is future phase.** PR-A does NOT introduce a workflow to re-hash and compare stored evidence against the captured hash. The verification workflow is future (likely PR-E or beyond). PR-A's job is to capture the hash at creation only.

### Canonical rules introduced or reaffirmed by PR-A

The following rules are documented in PR-A's `spec.md` and `boundary-contracts.md` PR-A sections and enforced throughout the PR-A workflows.

- **Source-of-Truth Boundary Rule (canonical).** Logs & Audit retains immutable evidence; source modules remain canonical for operational records; external tools are coordination/evidence only. Reaffirms and tightens the existing baseline boundary.
- **External-Tool-Not-Source-of-Truth Rule (canonical, new in PR-A).** External evidence references are coordination/proof only; external systems must NEVER become CIXCI operational source of truth.
- **Immutable Evidence Rule (canonical, reaffirms baseline).** Evidence is append-only. Corrections are amendments or supersessions. No destructive edits. Reaffirms the existing baseline "no silent rewrite of audit evidence is allowed."
- **Source Snapshot Minimization Rule (canonical, new in PR-A).** Source snapshots are selected metadata + hashes + references; full operational record copies are exceptional and gated by Full Payload Exception Record.
- **Evidence-Per-Lifecycle-Step Rule (canonical, new in PR-A).** Any source-module workflow that reaches validation, preview, commit, cancel, fail, or supersession creates evidence regardless of final outcome. Canceled imports retain uploaded file and validation evidence. Failed exports create evidence even if no buyer system receives the payload. Failed email delivery attempts create evidence even if the vendor never receives the file.
- **Audit-Record-and-Evidence-Record Separation Rule (canonical, new in PR-A).** Audit Record is the observation; Evidence Record is the artifact. The two are distinct entities.
- **Amendment vs Supersession Distinction Rule (canonical, new in PR-A).** Amendment is small correction; supersession is replacement after source-module correction. Different lifecycles.
- **Promotion-of-Naming Rule (one-time clarification, new in PR-A).** Evidence Amendment Record is the canonical name; existing baseline language preserved.
- **At-Creation Classification Rule (canonical, new in PR-A).** retention_class, redaction_class, access_class assigned at evidence creation; PR-D adds review workflows.

- **file_tracking_record_reference required or typical only for file-backed evidence.** Non-file-backed evidence types have `file_tracking_record_reference` = not-applicable.
- **external_evidence_reference required or typical only for external / API / notification / transport evidence.** Pure operational_state / decision evidence types have `external_evidence_reference` = optional or not-applicable.
- **company_scope_reference is required for ALL evidence_type values.** Tenant scope governance applies universally.
- **One of actor_reference or service_trigger_reference is required for ALL evidence_type values.** Source modules choose which based on whether the action is human-triggered or service-triggered.

### Evidence Backing Classification

PR-C defines 8 non-exclusive backing classifications:

- **file_backed** - evidence whose primary substance is a file artifact tracked via File Tracking Record (PR-B).
- **api_backed** - evidence whose primary substance is an API call (request / response payloads, transmission lifecycle).
- **notification_backed** - evidence whose primary substance is a notification (delivery attempt, suppression, fanout).
- **external_backed** - evidence whose primary substance is an external system interaction (provider response, webhook receipt, third-party action).
- **ai_backed** - evidence whose primary substance is an AI agent execution, recommendation, or output. PR-C reserves this classification for future use; no evidence_type values are catalogued under `ai_backed` in PR-C because AI Agent Services module does not exist on origin/main.
- **operational_state** - evidence that captures a source-module operational state value or state transition at a point in time.
- **decision** - evidence that records a source-module decision (commit / cancel / accept / reject / approve / deny / route).
- **transport_delivery** - evidence about delivery mechanics distinct from business outcomes (delivery attempts, retries, idempotency, dedupe).

Classifications are non-exclusive. A single evidence_type can carry multiple backing classifications. For example:

- `vendor_shipping_import` is BOTH `file_backed` AND `operational_state`.
- `import_commit` is BOTH `operational_state` AND `decision`.
- `vendor_export_delivery` is BOTH `transport_delivery` AND `notification_backed` (or `external_backed`).
- `api_response` is BOTH `transport_delivery` AND `api_backed` AND `external_backed`.

### Transport-delivery vs business-outcome separation (canonical)

Transport evidence (`transport_delivery` + `api_backed` / `notification_backed` / `external_backed`) is explicitly distinct from business-outcome evidence (`operational_state` + `decision`). Subscribers distinguishing transport from business outcomes filter on the backing classification.

For example: a vendor email export attempt that succeeded at the transport layer (`vendor_export_delivery` with `transport_delivery` + `notification_backed`) does NOT imply the vendor took business action on the export. The business outcome is a separate evidence emission from the source module (Order Routing's operational records).

### External-Tool-Not-Source-of-Truth Rule (PR-A) reaffirmed for PR-C external_backed values

All `external_backed` evidence_type values in PR-C are coordination evidence only. External provider responses, webhook receipts, email delivery acknowledgements, and external system actions are NOT CIXCI authoritative source. CIXCI source modules remain the source of truth for the underlying operational state. PR-A's External-Tool-Not-Source-of-Truth Rule applies fully.

### Canonical rules introduced by PR-D (19 rules)

**Retention rules (5):**

- **Legal-Hold-Overrides-Purge Rule.** Legal hold MUST override purge. Any disposition state that would delete payload is blocked when Legal Hold Record is active for the evidence's scope. Workflow MUST record `purge_blocked_by_hold` instead of `purge_eligible` / `purged_reference_only`.
- **Immutable-Evidence-Retention Rule.** Evidence metadata MUST remain append-only even when payload / file references expire or are purged. Retention actions do NOT mutate the Evidence Record.
- **Retention-Disposition-Append-Only Rule.** Retention Disposition Record is append-only. Each disposition is a new record; reversal of `purge_eligible` to `retain` creates a NEW Retention Disposition Record; reversal of `purged_reference_only` is impossible (payload deleted).
- **File-Metadata-Outlives-Payload Rule.** When file-backed evidence reaches `purged_reference_only`, the File Tracking Record metadata and `file_hash_reference` MUST be preserved. `file_storage_reference` becomes a tombstone.
- **Source-Module-Deletion-Independence Rule.** Source-module operational record deletion does NOT auto-delete corresponding Evidence Records. Source-module deletion decisions are recorded as new Evidence Records.

**Redaction rules (5):**

- **Redaction-Never-Overwrites-Raw Rule.** Raw evidence is immutable. Redaction MUST create a separate redacted view; MUST NOT overwrite or mutate `raw_evidence_reference`.
- **Redacted-View-Default Rule.** When a redacted view is available, broad access defaults to the redacted view. Raw evidence access requires explicit justification.
- **Redaction-Transformation-Append-Only Rule.** Redaction Transformation Record is append-only. Re-redaction creates a NEW record with incremented `redaction_version`.
- **Redacted-Views-Per-Audience Rule.** PR-D supports multiple redacted views per Evidence Record, one per audience. The Redaction Transformation Record carries the `redaction_audience` discriminator.
- **Amendment-vs-Redaction-Distinction Rule.** Evidence Amendment Record (PR-A) is used when evidence CONTENT requires clarification. Redaction Transformation Record (PR-D) is used when evidence content is unchanged but a REDACTED VIEW must be created for audience-specific access.

**Legal Hold rules (3):**

- **Legal-Hold-Does-Not-Mutate-Evidence Rule.** Applying / releasing a Legal Hold MUST NOT mutate any Evidence Record. Holds are referenced; Evidence Record content is unchanged.
- **Append-Only-During-Hold Rule.** During an active Legal Hold, amendments (PR-A) and supersessions (PR-A) ARE STILL ALLOWED for Evidence Records under the hold. The hold blocks PURGE / DELETION, not append-only correction.
- **Legal-Hold-Apply-Append-Only Rule.** Legal Hold Record is append-only. Status transitions (`applied` -> `released` / `lapsed`) are append-to-record (release_timestamp, release_authority populated); no field deletion or re-applying-overwrites.

**Access rules (4):**

- **Raw-Evidence-Access-Exceptional Rule.** Raw evidence access is EXCEPTIONAL. Default access is redacted view. Raw access requires `access_reason_reference` and is logged via hardened Audit Access Record.
- **All-Access-Logged Rule.** Every access attempt (granted, denied, or attempted-and-incomplete) MUST be logged via hardened Audit Access Record. No silent access.
- **Service-Identity-Access-Logged Rule.** Service identity access (machine actors, service triggers) is logged identically to human access via hardened Audit Access Record. `service_trigger_reference` populated instead of `actor_reference`.
- **Tenant-Company-Owns-Authority Rule.** Tenant Company remains canonical authority for users, roles, permissions, parent / child scope, capability checks, and service identities. Logs & Audit DOES NOT become permission authority. Logs & Audit documents access expectations; Tenant Company evaluates `check_access`.

**Governance / policy rules (2):**

- **Governance-Policy-Locked-By-PR-D Rule.** PR-A / PR-B / PR-C default class guidance is suggestion-only; PR-D locks the policy. PR-A's At-Creation Classification Rule remains in force.
- **No-New-Tenant-Roles-In-PR-D Rule.** PR-D does NOT introduce new Tenant Company roles, capability flags, scopes, or permission families. PR-D documents access expectations; Tenant Company hardening (future PR) creates roles if needed.

## PR-E Specification - Search / Query / Review / Investigation / Audit Report Export

This section specifies the Search / Query / Review / Investigation / Audit Report Export governance layered on top of the existing baseline, PR-A Core Evidence Spine, PR-B File Tracking Foundation, PR-C Cross-Module Evidence Catalog, and PR-D Retention / Redaction / Legal Hold / Access Governance. All prior content is preserved without rename or removal.

### Scope summary

PR-E introduces 5 new entities (Evidence Search Session, Evidence Review Session, Evidence Collection Record, Review Note / Annotation, Audit Report Export Record), adds 4 events (`audit.search.executed`, `audit.review-session.recorded`, `audit.review-note.recorded`, `audit.evidence-export.recorded`), adds 13 numbered workflows, and locks 25 canonical rules. PR-E is documentation-and-architecture; PR-E does NOT modify source modules; does NOT modify `openapi-contracts.md`; does NOT introduce new Tenant Company roles; does NOT introduce new filterable fields beyond existing PR-A through PR-D fields.

### Sequence completion claim

### PR-E enumerations and discriminators

#### Evidence Search Session `query_target` (3 values)

- `evidence_records` - search targets Evidence Records (PR-A).
- `file_tracking_records` - search targets File Tracking Records (PR-B).
- `pr_d_governance_records` - search targets PR-D governance records (Legal Hold Records, Retention Disposition Records, Redaction Transformation Records, hardened Audit Access Records).

#### Evidence Search Session `sensitive_filter_used` (boolean)

`true` when any of: `restricted_evidence = true` filter, `legal_hold_reference != null` filter, `redaction_class IN (customer_sensitive_restricted, pricing_sensitive_restricted, invoice_sensitive_restricted, warranty_sensitive_restricted, tenant_security_restricted)` filter, `view_type = raw` filter, `access_class IN (compliance_only, system_admin_only)` filter is in the filter set. Otherwise `false`.

#### Evidence Review Session `review_session_status` (3 values)

- `open` - session active; reviewer is actively working.
- `closed` - session formally closed; review_disposition populated; further notes attached create new records but session itself is terminal.
- `suspended` - session paused; may transition back to `open` or to `closed`.

#### Review Note / Annotation `review_note_target_kind` (3 values)

- `evidence_record` - note targets a specific Evidence Record.
- `evidence_collection` - note targets an Evidence Collection Record.
- `review_session` - note targets a Review Session as a whole (e.g., session-level summary).

#### Review Disposition (enumeration on Review Session and Review Note)

- `informational` - record-keeping only.
- `escalated` - escalated for further review.
- `dismissed` - reviewed and dismissed.
- `referred_to_compliance` - escalated to compliance / audit reviewer.
- `referred_to_legal` - escalated to legal.
- `closed_no_action` - reviewed; no action taken.

Concrete downstream consumer behavior for each disposition is future Tenant Company coordination.

#### Audit Report Export Record `export_status` (4 values; for `audit.evidence-export.recorded` discriminator)

- `generated` - export package generated successfully; `export_file_tracking_record_reference` populated.
- `failed` - export attempt failed; `export_file_tracking_record_reference` NULL; failure reason captured.
- `canceled` - export canceled (e.g., by reviewer, by authority change, by hold-blocked content); `export_file_tracking_record_reference` NULL.
- `metadata_only` - export was a metadata-only preview / inventory listing; NO export file generated; `export_file_tracking_record_reference` NULL.

`audit.evidence-export.recorded` carries `export_status` as a discriminator. NO separate `audit.evidence-export.generated` event.

#### Audit Report Export Record `export_redaction_audience`

Default audiences: `internal`, `audit_only`, `compliance_only`. (May not be `buyer` or `vendor` by default; buyer / vendor-facing exports require explicit authority and audience-specific Redaction Transformation Record.)

### PR-E canonical rules (25 rules introduced)

#### Search safety rules (5)

- **Search-Result-Redacted-By-Default Rule.** Search results default to the redacted view per PR-D Redacted-View-Default Rule. Raw retrieval requires explicit Raw Evidence Retrieval Request (PR-E Workflow 6) which invokes PR-D Workflow 9 (Raw Evidence Access) per PR-D Raw-Evidence-Access-Exceptional Rule.

- **Search-Defers-To-PR-D-Access-Governance Rule.** All search result access flows through PR-D Workflow 8 (Evidence Access Recording) which creates the hardened Audit Access Record. PR-E does NOT introduce a parallel access logging entity. Every result access (granted / denied / attempted-non-terminal) is logged via PR-D.

- **Hidden-Denied-Result Rule.** Denied results are hidden from the result set by default, INCLUDING from result counts and pagination indicators. Unauthorized requesters MUST NOT learn of the existence of evidence they are denied to access. Only authorized audit / compliance reviewers MAY request visible-denied metadata (see Visible-Denied-Metadata-Minimized Rule); that request itself is governed by PR-D Access Policy Matrix and PR-D hardened Audit Access Record.

- **Visible-Denied-Metadata-Minimized Rule.** When visible-denied metadata is allowed (authorized audit / compliance reviewers per PR-D Access Policy Matrix), the metadata MUST be minimized: it MUST NOT expose raw evidence content; it MUST NOT expose restricted details (PII, pricing values, customer-sensitive content, source snapshot bodies); it SHOULD limit disclosure to: presence-of-evidence indicator, `evidence_type`, `evidence_family`, `captured_at`, `redaction_class`, `access_class`, and reason-for-denial reference. Visible-denied access is itself logged via PR-D hardened Audit Access Record with `access_result = denied` and `restricted_evidence_flag` reflecting the underlying evidence's flag.

- **Sensitive-Search-Logged Rule.** Searches with `sensitive_filter_used = true` REQUIRE `search_initiated_purpose_reference` on the Evidence Search Session AND emit `audit.search.executed` with `sensitive_filter_used = true` discriminator for anomaly detection / abusive-search detection.

#### Result rendering rules (6)

- **Per-Audience-Result-Selection Rule.** When multiple Redaction Transformation Records exist per Evidence Record (one per audience per PR-D Redacted-Views-Per-Audience Rule), search result rendering selects the appropriate redacted view per requester's audience. Selection flows through PR-D Workflow 13 (Tenant / Parent / Child Evidence Access Evaluation).

- **Purged-Reference-Only-Metadata-View Rule.** Search results matching Evidence Records with `retention_disposition_state = purged_reference_only` return metadata / hash / tombstone view only. Payload is unavailable; File Tracking Record `file_hash_reference` + metadata + tombstone `file_storage_reference` are preserved per PR-D File-Metadata-Outlives-Payload Rule. Result MUST be clearly marked as `purged_reference_only`; result rendering MUST NOT 404.

- **Archived-Result-Availability-State Rule.** Search results matching Evidence Records with `retention_disposition_state = archive` indicate availability state (e.g., archived; retrieval-latency-expected) rather than treating the record as fully available with zero-latency retrieval. Concrete archive retrieval mechanics are implementation-level.

- **File-Backed-Result-Masked-Default Rule.** File-backed search results (Evidence Records with `file_tracking_record_reference`) default to masked / redacted payload view via PR-B `masked_payload_reference`. Raw file payload retrieval requires PR-E Workflow 7 (File-Backed Evidence Retrieval) which invokes PR-D Workflow 9 escalation. Downloaded files are typically the served view (masked / redacted); raw download requires escalation.

- **Source-Snapshot-Minimization-In-Preview Rule.** Search result previews of source snapshots respect PR-A Source Snapshot Minimization Rule: previews show minimized content; full snapshot retrieval requires explicit raw retrieval workflow.

- **Full-Payload-Exception-No-Raw-Preview Rule.** Search results matching Evidence Records with PR-B Full Payload Exception Record default to masked / redacted view per PR-D Full Payload Exception Redaction Requirement. Raw full-payload preview is NOT default; raw retrieval requires PR-D Workflow 11 (Full Payload Exception Access Review) escalation.

#### Sensitivity flag visibility rules (2)

- **Legal-Hold-Flag-Visibility-Scoped Rule.** Legal hold flags (i.e., the indicator that a result is under active Legal Hold) on search results are visible ONLY to authorized audiences (Compliance / Audit Reviewer, CIXCI System Admin, source-module service identities with appropriate access_class). Unauthorized audiences see no legal hold indicator in result rendering. Visibility is bound by PR-D Access Policy Matrix.

- **Restricted-Flag-Visibility-Scoped Rule.** `restricted_evidence` flags on search results are visible ONLY to authorized audiences (consistent with Hidden-Denied-Result Rule). Unauthorized audiences see no restricted_evidence indicator.

#### Review rules (3)

- **Review-Note-Append-Only Rule.** Review Note / Annotation entities are append-only. Corrections create NEW Review Note entities that reference the prior note via `prior_review_note_reference`; the prior note is NOT modified.

- **Review-Note-Is-Not-Evidence-Amendment Rule.** Review Notes are reviewer commentary; they are NOT PR-A Evidence Amendment Records. Review Notes do NOT modify Evidence Record content. When evidence CONTENT requires clarification / correction, the source module emits an Evidence Amendment Record (PR-A); review notes only document the reviewer's commentary.

- **Evidence-Collection-References-Only Rule.** Evidence Collection Record groups Evidence Records by reference; it does NOT copy or embed evidence content. Membership additions / removals create new collection_version (architectural pattern; concrete versioning shape implementation-level).

#### Export rules (3)

- **Export-Default-Redacted Rule.** Audit Report Export Record content defaults to redacted / masked rendering per `export_redaction_audience`. Each evidence item in the export uses the appropriate Redaction Transformation Record (PR-D). Raw export requires per-item PR-D Workflow 9 escalation.

- **Export-File-Tracking-Only-When-Artifact-Exists Rule.** Audit Report Export Record's `export_file_tracking_record_reference` is populated ONLY when an export file / package artifact is generated (`export_status = generated`). When `export_status = metadata_only` (preview / inventory listing) OR `export_status = failed` OR `export_status = canceled`, `export_file_tracking_record_reference` is NULL. Metadata-only report-preview activity does NOT require PR-B File Tracking Record.

- **Export-Access-Logged-Via-PR-D Rule.** Every download / access of an Audit Report Export package is logged via PR-D Workflow 8 -> PR-D hardened Audit Access Record. PR-E does NOT introduce a separate Audit Export Download Record. `view_type` is `redacted` by default; raw export item access requires elevated authority.

#### Indexing rules (4)

- **Index-Is-Not-Source-of-Truth Rule.** The Search Index Projection is a derived projection over PR-A / PR-B / PR-C / PR-D records. PR-A Evidence Records (and PR-B / PR-C / PR-D records) are the canonical source of truth. Index loss / corruption MUST be recoverable via rebuild from canonical records.

- **Index-Default-Redacted Rule.** Index entries that contain any evidence content (beyond references and discriminators) default to redacted content per PR-D Redaction Policy Matrix. Sensitive evidence content is NEVER indexed in raw form.

- **No-Raw-Payload-Indexing Rule.** Raw evidence payloads are NEVER indexed. File payloads, raw_evidence_reference content, Source Snapshot bodies, Full Payload Exception content are NOT included in the Search Index Projection. The index contains references, classes, statuses, dates, identifiers, and redacted text only.

- **Index-Stale-Acceptable Rule.** PR-E acknowledges that the Search Index Projection is eventually consistent. Stale results are acceptable IF disclosed via a freshness indicator (implementation-level). Reviewers MUST NOT treat search results as canonical source of truth.

#### Boundary rules (2)

- **Search-Not-Source-of-Truth Rule.** Logs & Audit search returns Evidence Records (audit trail of events and decisions); source modules retain canonical operational source-of-truth. PR-E search filter dimensions filter on PR-A Evidence Record fields and PR-D governance fields; NOT on source-module operational state. Logs & Audit search is NOT a substitute for source-module operational queries.

- **Audit-Export-Not-Analytics Rule.** Audit Report Export Records are compliance / investigation / regulatory artifacts. Audit exports are NOT BI / analytics dashboards / metrics / trend reports / business-performance reporting (Analytics module owns BI surface). Audit exports do NOT produce aggregated KPIs or business intelligence outputs.
