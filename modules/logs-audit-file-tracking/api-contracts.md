# Logs And Audit File Tracking API Contracts

This document is proposal-level architecture. It describes domain-specific API contract concepts without finalizing endpoint design or implementation schema.

## API Purpose

Logs & Audit APIs should support audit record creation, file tracking, API transmission logging, validation/processing result capture, retry/failure history, user-trigger history, duplicate detection, correction/reupload history, reprocess request evidence, search/filtering, and controlled payload references.

## Service Ownership

Logs & Audit owns API contracts for traceability evidence only.

The API must not expose operations that mutate source records, calculate pricing, choose routes, mark fulfillment state, generate invoices, approve warranty claims, manage tenant eligibility, execute AI actions, deliver notifications, define Analytics metrics, or configure Vendor Integration behavior.

## Contract Families

### Audit Record APIs

Purpose: create and retrieve append-only audit records for platform actions.

Inputs should include:

- Source module.
- Event/action type.
- Actor/user/service.
- Company/entity scope.
- Related record references.
- File reference where applicable.
- Masked payload reference by default.
- Full payload reference by approved exception only.
- Status.
- Timestamp.
- Before/after summary placeholder.
- Validation result reference.
- Retry count.
- Error code/message.
- Retention owner.
- Retention class.
- Redaction class.
- Access class.
- Legal/contract hold where applicable.
- Review status.

### File Tracking APIs

Purpose: record append-only imports, exports, uploads, downloads, duplicate detection, corrections, and reuploads.

Required concepts:

- File name.
- File type.
- Direction: import/export.
- Source module.
- Vendor.
- Buyer/entity scope.
- Date range.
- Generated by / uploaded by.
- Generated at / uploaded at.
- Row count.
- Accepted row count.
- Failed row count.
- Validation status.
- Processing status.
- Error summary.
- File version.
- Checksum/hash.
- Duplicate detection.
- Correction/reupload history.

### API Transmission Log APIs

Purpose: record append-only external or module-to-module API attempts, retries, failures, and completions.

Inputs should include:

- Source module.
- Target system/module.
- Transmission direction.
- Request reference or masked request reference.
- Response reference or masked response reference.
- Status.
- Retry count.
- Error code/message.
- Correlation id.
- Idempotency key.
- Retention class.
- Redaction class.
- Access class.

### Validation And Processing Result APIs

Purpose: attach append-only validation and processing outcomes to file tracking or API transmission records.

Inputs should include:

- Validation status.
- Processing status.
- Row count.
- Accepted row count.
- Failed row count.
- Error summary.
- Rule set/version placeholder.
- Processing timestamps.
- Row-level validation error references.
- Partial file accepted/rejected state.

### Row-Level Error APIs

Purpose: record row-level validation and processing errors without mutating source records.

Inputs should include:

- Row identity.
- Row number.
- Source row hash.
- Validation error code.
- Validation error message.
- Severity.
- Retryability.
- Blocking state.
- Correction link/reference.
- Source-module disposition.

### Vendor Operational File APIs

Purpose: track manual vendor file workflows where APIs are unavailable or not configured.

Supported proposal-level flows:

- Manual export of order data to vendors.
- Manual export of return data to vendors.
- Vendor import of shipping information.
- Vendor import of return outcome information.

These APIs must not change routing, fulfillment, return, invoice, vendor source records, or Vendor Integration configuration.

### Reprocess / Retry Request APIs

Purpose: record reprocess or retry requests and outcomes as traceability evidence.

Rules:

- Logs & Audit may record request status and source-module outcome.
- Source modules own execution and any business-state mutation.
- Logs & Audit must not directly reprocess source business records.

### Search And Filtering APIs

Purpose: retrieve audit/file/transmission records for operational investigation and accountability.

Filters may include:

- Date range.
- Company.
- Child entity.
- Vendor.
- Buyer.
- Source module.
- File type.
- API event type.
- Status.
- Actor/user.
- Error type.
- Import/export direction.
- Correlation id.
- Related record reference.
- Retention class.
- Redaction class.
- Access class.

Rules:

- Cross-tenant search is denied by default.
- Buyer/vendor projections must be scoped.
- Global search is limited to approved internal roles.
- Sensitive search queries are auditable.
- Result export controls are required.
- Search rate limits are required.
- Source module/service identity may constrain service access.
- No unrestricted audit browsing.

## Error Models

Proposal-level error families:

- Missing source module.
- Missing event/action type.
- Missing actor/service reference.
- Missing file direction.
- Missing file type.
- Missing checksum/hash where required.
- Invalid retention class.
- Invalid redaction class.
- Invalid access class.
- Payload storage not permitted.
- Full payload exception approval missing.
- Sensitive payload requires masked reference.
- Payload size limit exceeded.
- Duplicate file detected.
- Cross-tenant search denied.
- Search access denied.
- Result export denied.
- Source module boundary violation.
- Reprocess execution not owned by Logs & Audit.

## Idempotency Rules

Idempotency is required for:

- Audit record creation.
- File tracking record creation.
- API transmission attempt logging.
- Validation result creation.
- Processing result creation.
- Row-level error creation.
- Retry/failure history updates.
- Reprocess/retry request creation.
- Correction/reupload history creation.

Dedupe keys should include source module, event/action type, related record references, file checksum/hash, file direction, vendor/buyer scope, date range, row identity/source row hash where applicable, correlation id, and idempotency key where available.

## Authentication / Authorization

Authorization should be scoped by tenant, company/entity, actor role, source module, source module/service identity, action type, redaction class, retention class, access class, and consumer class.

Sensitive audit records should be visible only to roles explicitly authorized for the relevant redaction and access class.

Sensitive audit search, result export, and payload reference access should themselves create audit records.

- Which audit searches are buyer/vendor-facing versus internal-only?
- Which file reprocess actions belong in Logs & Audit versus source modules?
- Which full payload exception workflows require Compliance or Security approval?

The concrete deduplication semantics are implementation-level; PR-A documents the architectural requirement only.

### Correlation and tracing on PR-A API surfaces

### Failure handling on PR-A API surfaces

PR-A's architectural surfaces inherit existing baseline failure handling:

- Missing required references (source_module_reference, source_record_reference for evidence-bearing audits, company_scope_reference) reject the submission at the source module's call site. Concrete error handling is implementation-level.

- Source-of-truth violations (attempts to mutate a source-module record through Logs & Audit) are rejected as boundary violations per the existing baseline `SOURCE_BOUNDARY_VIOLATION` error code.
