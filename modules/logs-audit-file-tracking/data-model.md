## Audit Immutability Model

Proposal-level rules:

- Audit records are append-only.
- File tracking records are append-only.
- API transmission logs are append-only.
- Validation results are append-only.
- Processing results are append-only.
- Corrections use amendment records, supersession references, or correction references.
- No silent rewrite of audit evidence is allowed.
- Audit access/view activity should itself be auditable for sensitive records.
- Status or review changes should be captured as new records or explicit amendments.

## Scheduled System Admin Activity Summary Evidence Entities (Cross-Module PR)

This section introduces two new entities owned by Logs & Audit File Tracking for the Scheduled System Admin Activity Summary Email cross-module hardening pass. Two additional entities are owned by Notification Platform Service (Activity Summary Configuration, Activity Summary Delivery Attempt), and two additional entities are owned by Analytics / Reporting (Activity Summary Reporting Window, Activity Summary Aggregation Record). All concepts are additive. Existing Logs & Audit File Tracking entities (Audit Record, File Tracking Record, API Transmission Log, Validation Result Record, Processing Result Record, Row-Level Validation Error Record, Partial File Processing Record, and so on) are not redefined; this PR layers on top.

### Cross-module boundary discipline reaffirmed

- Logs & Audit File Tracking owns the Activity Summary Generated Evidence entity and the No-Activity Summary Suppression Evidence entity introduced below.
- Notification Platform Service owns the Activity Summary Configuration and the Activity Summary Delivery Attempt entities (see notification-platform-service/data-model.md PR-C section). Logs & Audit File Tracking retains the Activity Summary Delivery Attempt records by reference; it does not duplicate the lifecycle entity. This decision is intentional: Notification Platform Service is canonical for delivery attempt state; Logs & Audit File Tracking owns retention of the record via the existing audit-record retention pattern.
- Analytics / Reporting owns the Activity Summary Reporting Window and Activity Summary Aggregation Record entities (see analytics-reporting/data-model.md PR-C section).
- Source modules (Order Routing, Fulfillment / Returns) and their facts are read-only inputs to Analytics / Reporting; Logs & Audit File Tracking does not consume source-module events directly for this PR. The evidence retention pattern operates on the Cross-Module PR entities only.
- Tenant Company owns recipient scope authority and `check_access` patterns; not modified by this PR.
- Integration Management owns transport-layer records; not modified by this PR.

### Phase 1 scope guardrails

- This PR's Logs & Audit additions are scoped to the summary-specific evidence surfaces required by the Cross-Module Summary Emails PR. This PR is not the full Logs & Audit File Tracking module hardening pass; that remains a later standalone hardening area.
- The two new entities introduced here follow the existing Logs & Audit immutability rule: append-only, no silent rewrite, amendment via amendment records where applicable.
- Retention duration, storage mechanics, search semantics, payload reference rules, and access-class enforcement for the new evidence types follow the existing Logs & Audit retention and redaction patterns; no new retention class is introduced by this PR.
- The new entities use references to the canonical owning entities (Activity Summary Configuration, Activity Summary Reporting Window, Activity Summary Aggregation Record, Activity Summary Delivery Attempt); they do not embed source-module record content or aggregation record content.

### Activity Summary Generated Evidence

The immutable record that a summary aggregation record was generated for a specific reporting window. Created by Workflow 10 (Summary Audit Evidence Recording) when triggered by Analytics / Reporting Workflow 5 completing with non-empty aggregation.

**Ownership:** Logs & Audit File Tracking.

**Identity:** referenced via `activity_summary_generated_evidence_reference` from operator surfaces and from the existing Logs & Audit audit-record search patterns.

**Lifecycle:** Created once per successful (non-empty) window aggregation. Immutable. Amendments use the existing Logs & Audit amendment workflow if correction is needed.

**Required fields and references (proposal-level):**

- `activity_summary_generated_evidence_reference` - canonical identifier.
- `activity_summary_configuration_reference` - the originating Notification Platform Service Activity Summary Configuration.
- `activity_summary_reporting_window_reference` - the Analytics / Reporting Reporting Window that produced the aggregation.
- `activity_summary_aggregation_record_reference` - the Analytics / Reporting Aggregation Record that was generated.
- `generation_timestamp` - moment Analytics / Reporting Workflow 5 completed (matches the aggregation record's `generated_at_timestamp`).
- `actor_reference` - typically a system service identity for scheduled aggregation; reference resolves through Tenant Company `check_access` patterns.
- `carry_forward_window_reference_collection` - optional; references to prior `delivery_failed` windows whose intervals were subsumed by this aggregation. Mirrors the same field on the Analytics / Reporting Reporting Window for direct Logs & Audit search traceability.
- `retention_class` - follows existing Logs & Audit retention class patterns; specific retention duration is implementation-level and deferred.
- `redaction_class` - follows existing Logs & Audit redaction class patterns; Activity Summary Generated Evidence is internal-scope (CIXCI System Admin only; no buyer-or-vendor exposure in Phase 1).
- `access_class` - follows existing Logs & Audit access class patterns.
- `audit_record_reference` - reference to the Logs & Audit canonical audit record created alongside this evidence (using the existing Audit Record entity pattern).
- `created_at` - record-management timestamp.

**Boundary discipline:**

- This evidence does not embed the Activity Summary Aggregation Record content. The aggregation record is referenced; aggregation content lives in Analytics / Reporting.
- This evidence does not embed source-module records.
- Amendment of this evidence record follows the existing Logs & Audit amendment workflow. Direct mutation is not permitted (per the Logs & Audit immutability rule).
- Retention duration for Activity Summary Generated Evidence follows the existing retention-class policy for internal-scope evidence; PR-C does not introduce a new retention duration.
- This evidence is not a substitute for the Analytics / Reporting aggregation record. The aggregation record is the canonical Analytics-owned snapshot; this evidence is the Logs & Audit-owned retention surface.

### No-Activity Summary Suppression Evidence

The immutable record that a reporting window was evaluated and produced zero activity, and therefore no email was sent. Created by Workflow 10 when triggered by Analytics / Reporting Workflow 6 (No-Activity Suppression).

**Ownership:** Logs & Audit File Tracking.

**Identity:** referenced via `no_activity_summary_suppression_evidence_reference` from operator surfaces and from existing Logs & Audit audit-record search patterns. The Analytics / Reporting Reporting Window's `suppression_evidence_reference` field points to this record.

**Lifecycle:** Created once per window that reaches `suppressed_no_activity` terminal state. Immutable.

**Required fields and references (proposal-level):**

- `no_activity_summary_suppression_evidence_reference` - canonical identifier.
- `activity_summary_configuration_reference` - the originating Notification Platform Service Activity Summary Configuration.
- `activity_summary_reporting_window_reference` - the Analytics / Reporting Reporting Window that was evaluated and suppressed.
- `evaluation_timestamp` - moment Analytics / Reporting Workflow 5 evaluated the window and produced zero activity.
- `suppression_reason` - enumeration; Phase 1 value is `no_source_facts_in_window`. Future values may include `all_source_facts_excluded_by_filter` (not implemented in Phase 1).
- `window_start_timestamp` and `window_end_timestamp` - mirror the window's boundaries for direct Logs & Audit search traceability.
- `source_fact_zero_activity_assertion_reference` - reference or assertion that the source-fact aggregation pass produced zero counts across all sections for the effective window interval. Captures the basis for suppression without requiring source-record content.
- `retention_class` - follows existing Logs & Audit retention class patterns.
- `redaction_class` - follows existing Logs & Audit redaction class patterns; internal-scope.
- `access_class` - follows existing Logs & Audit access class patterns.
- `audit_record_reference` - reference to the Logs & Audit canonical audit record created alongside this evidence.
- `created_at` - record-management timestamp.

**Boundary discipline:**

- This evidence preserves the audit trail for the suppression decision itself (that a reporting window was evaluated and produced zero activity). The cursor-advancement-on-no-activity action is performed downstream by Notification Platform Service (NPS Workflow 9 Trigger B path) after it consumes this evidence; the cursor-advancement audit trail lives on the NPS-side existing Audit Record entity pattern (with `source_module = notification-platform-service`, `event_action_type = activity_summary_cursor_advanced`, and `related_record_references` pointing back to this Suppression Evidence record and to the Activity Summary Configuration). The two retention surfaces together (this Suppression Evidence record plus the NPS-side cursor-advancement Audit Record) provide the full audit trail across the choreography; neither requires a field that does not yet exist at its own creation time.
- No delivery attempt is created in association with this evidence. The window did not produce content for delivery.
- Amendment of this evidence record follows the existing Logs & Audit amendment workflow.
- This evidence distinguishes "window was evaluated and produced zero activity" from "window was never evaluated". Without an explicit suppression evidence record, the absence of an aggregation record would be ambiguous.

### Activity Summary Delivery Attempt retention (reference pattern, not separate entity)

PR-C does not introduce a separate Logs & Audit-side entity for delivery attempt lifecycle. The Notification Platform Service Activity Summary Delivery Attempt is the canonical record for delivery attempt state (`pending`, `dispatched`, `acknowledged`, `failed`).

**Retention pattern:**

- Each Activity Summary Delivery Attempt is referenced by an audit record created via the existing Logs & Audit Audit Record entity pattern.
- The existing Audit Record carries `related_record_references` pointing to the Activity Summary Delivery Attempt's identifier.
- The existing Audit Record's source-module field is `notification-platform-service`; the event/action type discriminates between `delivery_attempted`, `delivery_succeeded`, and `delivery_failed`.
- Retention duration, redaction class, and access class follow the existing Logs & Audit patterns for Notification Platform Service delivery audit references.

**Boundary discipline:**

- No new Logs & Audit entity is required for delivery attempt retention. The existing Audit Record entity is sufficient.
- Delivery attempt lifecycle queries route to Notification Platform Service via the Activity Summary Delivery Attempt entity; retention queries route to Logs & Audit via the Audit Record entity referencing the delivery attempt.

### Missed Window Carry-Forward Evidence retention (reference pattern, not separate entity)

PR-C does not introduce a separate Logs & Audit-side entity for missed-window carry-forward state. The carry-forward references live on the Analytics / Reporting Reporting Window (in the `carry_forward_window_reference_collection` field) and are mirrored on the Activity Summary Generated Evidence record above (via its own `carry_forward_window_reference_collection` field).

**Retention pattern:**

- The Activity Summary Generated Evidence record's `carry_forward_window_reference_collection` field provides the Logs & Audit-side retention surface for carry-forward traceability.
- Each subsumed window's prior `delivery_failed` state remains observable on the Analytics / Reporting Reporting Window entity until it transitions to `superseded`. The transition itself produces an audit record via the existing Logs & Audit Audit Record entity pattern.

**Boundary discipline:**

- No new Logs & Audit entity is required for carry-forward retention.
- Multiple consecutive failures carry forward across multiple windows; each carry-forward event is auditable via the Audit Record pattern and via the carry-forward reference collection on the eventual successful Activity Summary Generated Evidence record.

### Evidence Record (PR-A) (new entity)

The Generic Evidence Record is the artifact spine. Every cross-module evidence record in CIXCI (current and future) conforms to this entity.

**Ownership:** Logs & Audit File Tracking.

**Identity:** referenced via `evidence_record_reference` from Audit Record (back-link), Evidence Amendment Record (target), Evidence Supersession Record (prior or new), and audit retention records.

**Required fields (proposal-level):**

**Identity and discrimination:**

- `evidence_record_id` - canonical identifier.
- `evidence_schema_version` - the schema version under which this evidence was created. Allows future schema migrations to coexist with historical evidence. Required.
- `evidence_type` - discriminator distinguishing evidence subtypes. Required. Proposal-level starter set of foundation values (examples only; comprehensive taxonomy is PR-C):
  - `module_action_evidence` - generic source-module action evidence (default fallback).
  - `file_evidence` - evidence relating to a file (upload, generated file, download). File-specific evidence types are PR-B / PR-C.
  - `api_transmission_evidence` - evidence relating to an API transmission. Refines into PR-C subtypes.
  - `import_evidence` - evidence relating to an import workflow. Refines into PR-C subtypes (manual product import, API product import, vendor shipping import, vendor delivery import, vendor return import, etc.).
  - `export_evidence` - evidence relating to an export workflow. Refines into PR-C subtypes (buyer API product export, vendor order export, vendor return export, vendor email export, etc.).
  - `summary_evidence` - evidence relating to a summary generation or suppression (refines into Cross-Module Summary Emails PR-C subtypes).
- `audit_record_reference` - reference to the parent Audit Record. Required. Every Evidence Record references an Audit Record.

**Source references:**

- `source_module_reference` - formal Source Module Reference. Required.
- `source_record_reference` - formal Source Record Reference. Required.
- `source_snapshot_reference` - formal Source Snapshot Reference. Optional. Governed by the Source Snapshot Minimization Rule.
- `source_event_reference` - optional. Reference to the source-module broker event that triggered evidence creation, when applicable.

**External coordination:**

- `external_evidence_reference` - optional sub-structure. NOT a standalone entity. See the External Evidence Reference Sub-Structure section below.

**Actor and scope:**

- `actor_reference` - optional. Populated for human-actor events.
- `service_trigger_reference` - optional. Populated for service-triggered events.
- `company_scope_reference` - required.

**Integrity:**

- `evidence_hash_reference` - hash of the evidence payload or attachment. Required. Hash algorithm is implementation-level.
- `evidence_attachment_reference` - optional. Reference to a stored artifact (file, snapshot blob, masked payload).

**Storage and view:**

- `raw_evidence_reference` - optional. Reference to the raw, unredacted evidence artifact.
- `redacted_view_reference` - optional. Reference to a redacted projection of the raw evidence. Populated by future PR-D workflows.

**Timestamps:**

- `captured_at` - required. Timestamp at which the source module observed the event being evidenced. Distinct from `created_at`.
- `created_at` - required. Timestamp at which Logs & Audit recorded the evidence.
- `created_by_actor_reference` or `created_by_service_trigger_reference` - one required.

**Correlation and deduplication:**

- `correlation_reference` - required. Cross-module correlation id for tracing related evidence across modules.
- `trace_reference` - optional. Distributed trace identifier (where applicable).
- `idempotency_key` - optional but recommended. For deduplication on retried evidence submissions.
- `replay_safe_dedupe_reference` - optional. Structured key for replay-safe deduplication beyond idempotency key (e.g., business-key tuple unique to the evidence event).

**Lifecycle and classification:**

- `evidence_status` - enumeration: `active`, `superseded`, `amended_with_amendments`, `legal_hold`. Initial value on creation is `active`. ### Amendment vs Supersession Distinction Rule (canonical, new in PR-A)

The two lineage mechanisms are distinct lifecycles.

- **Amendment** is small correction to existing evidence metadata or context. No new "current" Evidence Record is created. The original Evidence Record retains effective active status; the amendment is metadata attached to it. Use when: typo in evidence metadata, missing field added, additional context recorded after the fact, actor name disambiguated, redaction class clarified post-creation, source_snapshot_reference link added.
- **Supersession** is creation of a new Evidence Record that supersedes a prior one because the underlying source-module record was corrected. The prior Evidence Record transitions to `superseded`; the new Evidence Record is `active`. Use when: source-module operational record was updated and the new state needs new evidence; vendor re-uploaded a file with corrected rows; Media Asset Version was superseded by re-ingestion (the Media Asset Version evidence supersedes the prior version evidence); product import was re-run after correction.

Source modules choose the mechanism. Logs & Audit records lineage; source modules own outcomes.

### Evidence-Per-Lifecycle-Step Rule (canonical, new in PR-A)

Any source-module workflow that reaches validation, preview, commit, cancel, fail, or supersession creates evidence regardless of final outcome.

**Specific applications (architectural; specific evidence_types are PR-C):**

- A vendor product import that reaches validation creates evidence (file upload evidence + validation evidence) even if the vendor cancels before commit.
- A vendor product import that reaches preview creates evidence (preview generated evidence) even if the vendor abandons after preview.
- A buyer API product export that fails before delivery creates evidence (failed export evidence) even if no buyer system receives the payload.
- A vendor email export that fails delivery creates evidence (delivery attempt evidence + failure evidence) even if the vendor never receives the file.
- A vendor shipping import that is canceled retains uploaded file and validation evidence.
- A retry / replay attempt links back to the original evidence via supersession lineage.

**Rationale:** The audit trail must be complete. Investigators must be able to trace what happened even when the operational outcome was "canceled" or "failed". The evidence retention is governed by `retention_class` at creation; PR-D defines the duration matrix.

### Audit Record and Evidence Record Separation Rule (canonical, new in PR-A)

- Audit Record is the observation: "an audit-worthy thing happened."
- Evidence Record is the artifact: "here is the proof of what happened."
- The two are distinct entities.
- Audit Record CAN exist without Evidence Record (pure observation events: read-access, permission checks, status queries).
- Evidence Record ALWAYS references its parent Audit Record via `audit_record_reference`.
- Audit Record may reference Evidence Record via optional `evidence_record_reference` when proof is attached.
- Audit Record MUST NOT be merged with Evidence Record. The two-entity model preserves the meaning while remaining cohesive.

---

### Immutable Evidence Rule (canonical, reaffirms baseline)

- Evidence is append-only.
- Audit Records, Evidence Records, Evidence Amendment Records, Evidence Supersession Records are all append-only.
- File Tracking Records, API Transmission Logs, Validation Result Records, Processing Result Records, Row-Level Validation Error Records, Partial File Processing Records (existing baseline entities) remain append-only.
- Corrections are amendments (Evidence Amendment Record) or supersessions (Evidence Supersession Record + new Evidence Record).
- No destructive edits.
- No silent rewrite of audit evidence is allowed (reaffirms existing baseline rule).

---

### Source-of-Truth Boundary Rule (canonical, reaffirms baseline)

- Logs & Audit retains immutable evidence.
- Source modules remain canonical for operational records, business decisions, validation rules, correction decisions, state transitions, and source-module APIs and contracts.
- External tools are coordination and evidence only.
- Logs & Audit must NOT become the operational source of truth for orders, fulfillment, returns, media, product catalog, pricing, invoices, notifications, analytics, integrations, AI agents, or tenant/company permissions.
- Logs & Audit must NOT mutate source-module records.
- Logs & Audit must NOT decide business outcomes.

- `downloaded` as a lifecycle status. Download is treated as an audit-worthy action (new File Tracking Record with `file_direction = downloaded`), not as a lifecycle state of a source file. The source file remains `stored`.

### Evidence Type Reference Requirements Summary table

This table consolidates the per-evidence_type reference requirements above. Use the family-specific tables above for the authoritative per-evidence_type detail. The summary patterns are:

- **`actor_reference` OR `service_trigger_reference`** is REQUIRED for every catalogued evidence_type. Source modules populate one based on whether the action is human-triggered or service-triggered.
- **`company_scope_reference`** is REQUIRED for every catalogued evidence_type. Tenant scope governance applies universally.
- **`source_record_reference`** is REQUIRED for every catalogued evidence_type. PR-A's Audit-Record-and-Evidence-Record Separation Rule and Evidence-Per-Lifecycle-Step Rule apply.
- **`source_snapshot_reference`** ranges from REQUIRED (for snapshot-anchored values like `routing_decision`, `handoff_source_snapshot`, `export_payload_snapshot`, `buyer_product_export_baseline`, `pricing_snapshot`, `pricing_validation`, `pricing_exception`, `commission_revshare_snapshot`, `invoice_generation`, `invoice_adjustment`, `vendor_order_export_batch`, `vendor_return_export_batch`, `vendor_export_content_snapshot`) to TYPICAL for most others. PR-A's Source Snapshot Minimization Rule applies.
- **`file_tracking_record_reference`** is REQUIRED for file-backed evidence (`product_import_received`, `device_import_received`, `media_upload_session`, `media_upload_job`, `media_asset_version`, `source_url_reingestion`, `vendor_order_export_batch`, `vendor_return_export_batch`, `vendor_shipping_import`, `vendor_delivery_import`, `vendor_return_import`, `correction_reupload`, `analytics_export`, `report_file_generation`, `buyer_product_export_batch`); TYPICAL for partially file-backed evidence; NOT-APPLICABLE for non-file-backed evidence.
- **`evidence_attachment_reference`** is OPTIONAL across most evidence_type values. Pairs with PR-A's `evidence_attachment_reference` on Evidence Record.
- **`external_evidence_reference`** is REQUIRED for `external_backed` / `api_backed` integration evidence; TYPICAL for transport_delivery + external_backed; NOT-APPLICABLE for pure `operational_state` / `decision` business-outcome evidence.

### Entity 1 - Legal Hold Record (NEW PR-D entity)

**Purpose.** Records the legal hold scope, reason, authority, owner, applied / released timestamps, and status. Multi-dimensional scope. One Legal Hold Record per hold.

**Fields (architectural):**

- `hold_id` - canonical identifier.
- `hold_scope_discriminators` (sub-structure):
  - `evidence_type_scope` - list of evidence_type values from PR-C catalog (optional; at least one scope field populated).
  - `evidence_family_scope` - list of evidence_family values from PR-C catalog.
  - `company_scope_reference` - REQUIRED; ONE company_scope_reference per Legal Hold Record. Cross-tenant holds NOT supported.
  - `file_scope` - list of File Tracking Record references.
  - `source_module_scope` - list of source module references.
- `reason` - hold reason (e.g., regulatory investigation, court order, internal investigation).
- `authority` - reference to legal authority (court order ID, regulator, etc.). Compliance / legal owns authority semantics.
- `owner` - hold owner (Logs & Audit / Compliance role; NOT a Tenant Company role per Tenant-Company-Owns-Authority Rule).
- `applied_timestamp` - hold apply timestamp.
- `applied_actor_reference` OR `applied_service_trigger_reference` - who applied (one populated).
- `released_timestamp` - nullable; populated when status transitions to `released` or `lapsed`.
- `release_authority` - nullable; populated when released.
- `released_actor_reference` OR `released_service_trigger_reference` - who released.
- `status` - one of `applied` / `released` / `lapsed`.
- `hold_audit_reference` - parent Audit Record (PR-A).
- `correlation_reference` - PR-A envelope field.
- `evidence_schema_version` - PR-A envelope field.

**Append-only.** Status transitions populate release fields; do NOT delete fields or rewrite.

**Boundary discipline:**

- Logs & Audit owns the Legal Hold Record entity.
- Compliance / legal owns the authority decision; PR-D documents expectation; Logs & Audit logs the application.
- Tenant Company owns user / role identity of the applying actor; Tenant Company `check_access` evaluates whether the actor has authority to apply.
- Per Legal-Hold-Does-Not-Mutate-Evidence Rule: Evidence Records are NOT modified at apply / release time.
- Per Legal-Hold-Apply-Append-Only Rule: Legal Hold Record is append-only; status transitions append release fields.
- Per Append-Only-During-Hold Rule: amendments (PR-A) and supersessions (PR-A) ARE STILL ALLOWED for Evidence Records under hold.

### Search Index Projection (architecture concept; NOT an entity)

PR-E documents Search Index Projection as an ARCHITECTURE CONCEPT, NOT an entity. The projection is the derived view over Evidence Records / File Tracking Records / PR-D governance records that supports search query evaluation.

**Indexable fields (architecture-level list):**

- PR-A envelope fields: `captured_at`, `correlation_reference`, `trace_reference`, `evidence_schema_version`.
- PR-A references: Source Module Reference, Source Record Reference, Actor Reference, Service Trigger Reference, Company Scope Reference, External Evidence Reference identifiers.
- PR-A class fields: `retention_class`, `redaction_class`, `access_class`, `restricted_evidence`.
- PR-A `evidence_type` (PR-C catalog identifier).
- PR-A `evidence_family` (PR-C family identifier).
- PR-A `evidence_hash_reference`, `evidence_attachment_reference`.
- PR-B File Tracking Record fields: `file_purpose`, `file_direction`, `file_lifecycle_status`, `file_hash_reference`, `file_integrity_reference`.
- PR-D fields: Legal Hold Record `status` and scope discriminators; Retention Disposition Record `disposition_state` and `proof_class`; Redaction Transformation Record `redaction_class` / `redaction_audience` / `redaction_version`; hardened Audit Access Record `access_result` / `view_type` / `access_class_evaluated` / `redaction_class_evaluated` / `restricted_evidence_flag` / `legal_hold_state_at_access`.
- PR-E fields: Evidence Search Session identifiers + `query_target` + `sensitive_filter_used`; Evidence Review Session `review_session_status`; Evidence Collection Record `collection_status` + `collection_version`; Review Note / Annotation `review_note_target_kind`; Audit Report Export Record `export_status` + `export_redaction_audience`.

**NOT indexable (per No-Raw-Payload-Indexing Rule):**

- Raw evidence payloads.
- `raw_evidence_reference` content.
- Source Snapshot bodies.
- Full Payload Exception content.
- File payloads.

**Index discipline rules (PR-E canonical rules):**

- Per Index-Is-Not-Source-of-Truth Rule: PR-A Evidence Records (and PR-B / PR-C / PR-D records) are canonical source of truth; index is derived; rebuild is possible.
- Per Index-Default-Redacted Rule: index entries containing evidence content default to redacted content per PR-D Redaction Policy Matrix.
- Per No-Raw-Payload-Indexing Rule: raw payloads NEVER indexed.
- Per Index-Stale-Acceptable Rule: eventually-consistent index acceptable IF disclosed via freshness indicator (implementation-level).

**What PR-E does NOT define (indexing):**

- Database schema for the index.
- Runtime index implementation (search engine / FTS engine / etc.).
- Search engine choice.
- Concrete query execution engine.
- Concrete index rebuild mechanics.
- Index storage tier.
- Index versioning.
