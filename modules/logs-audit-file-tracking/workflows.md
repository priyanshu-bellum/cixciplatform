# Logs And Audit File Tracking Workflows

This document is proposal-level architecture. It outlines initial workflows without finalizing implementation behavior, storage policy, retention policy, or source-module contracts.

## Audit Record Creation Workflow

1. Source module emits or calls to create an audit record.
2. Logs & Audit validates source module, action type, actor/service, scope, redaction class, access class, and retention class.
3. Logs & Audit stores append-only audit metadata and permitted references.
4. Logs & Audit emits `audit.record.created` where configured.

Boundary note: creating an audit record does not approve or mutate the source business action.

## Audit Amendment Workflow

1. Authorized user, service, or source module requests an amendment, supersession reference, or correction reference.
2. Logs & Audit records the amendment as a new evidence record.
3. Original evidence remains unchanged.
4. Sensitive amendment access is auditable.

Boundary note: no silent rewrite of audit evidence is allowed.

## File Tracking Workflow

1. Source module or user creates a file tracking record for an import/export/upload/download.
2. Logs & Audit records file name, file type, direction, source module, vendor, buyer/entity scope, date range, actor, timestamp, checksum/hash, retention class, redaction class, and access class.
3. Validation and processing result records are attached as append-only evidence.
4. Duplicate detection and correction/reupload history are linked where applicable.

Boundary note: Logs & Audit tracks the file and processing history only. Source modules own the business effects of the file.

## API Transmission Logging Workflow

1. Source module records an API transmission attempt.
2. Logs & Audit stores target system/module, direction, request/response references or masked references, status, retry count, error details, correlation id, and idempotency key.
3. Retries append retry/failure history.
4. Completion or failure events are emitted.

Boundary note: Logs & Audit records transmission evidence; it does not own retry business decisions unless separately assigned.

## Validation And Processing Result Workflow

1. Source module provides validation or processing result.
2. Logs & Audit links the result to file tracking or API transmission evidence.
3. Row count, accepted row count, failed row count, validation status, processing status, error summary, and rule set/version placeholders are recorded.
4. Row-level errors capture row identity, row number, source row hash, error code/message, severity, retryability, blocking state, correction reference, and source-module disposition where permitted.
5. AI signals or notification hooks may be emitted for repeated failures or audit gaps.

## Manual Vendor Order Export Workflow

1. Authorized user or service exports order data to a vendor.
2. Logs & Audit records file tracking metadata, vendor, buyer/entity scope, date range, generated by, generated at, row count, accepted row count, failed row count, checksum/hash, and file version.
3. File validation and processing status are tracked where applicable.
4. Correction/reupload history is linked if the export is replaced.

Boundary note: Order Routing owns routing/suborder decisions. Logs & Audit only tracks the exported file evidence. Vendor Integration, if introduced, owns vendor credentials, transports, schedules, mappings, transforms, and delivery mechanics.

## Manual Vendor Return Export Workflow

1. Authorized user or service exports return data to a vendor.
2. Logs & Audit records file tracking metadata, vendor, buyer/entity scope, date range, generated by, generated at, row count, checksum/hash, and file version.
3. Error summaries and retry/failure history are attached where needed.

Boundary note: Fulfillment/Returns owns return operational state.

## Vendor Shipping Import Workflow

1. Vendor uploads shipping information file.
2. Logs & Audit records file name, file type, direction, vendor, buyer/entity scope, uploaded by, uploaded at, checksum/hash, and file version.
3. Validation result captures row count, accepted row count, failed row count, validation status, row-level errors, and error summary.
4. Processing result captures status, partial file accepted/rejected state, and failures.
5. Fulfillment/Returns consumes the file or source-module result through its own contract.

Boundary note: Logs & Audit must not mark shipments delivered or mutate fulfillment state.

## Vendor Return Outcome Import Workflow

1. Vendor uploads return outcome file.
2. Logs & Audit records import metadata, checksum/hash, row counts, failed rows, validation status, processing status, partial-file state, and correction/reupload history.
3. Fulfillment/Returns owns any operational return updates after consuming approved source-module data.

## Duplicate Detection Workflow

1. New file tracking record is compared against checksum/hash, file name, vendor, buyer/entity scope, date range, file type, and source module.
2. Duplicate detection record is created where a possible duplicate is found.
3. Review state is captured.
4. Source module decides whether to ignore, reject, reprocess, or accept according to its own rules.

Cost guardrail: duplicate detection should use scoped comparisons and cost controls so high-volume file activity does not create unbounded comparison work.

## Reprocess / Retry Request Workflow

1. Authorized user or service records a reprocess or retry request.
2. Logs & Audit records request reason, related evidence, source module responsible for execution, request status, and audit reference.
3. Source module decides whether and how to execute.
4. Logs & Audit records source-module response and outcome reference.

Boundary note: Logs & Audit records request and outcome evidence only. Source modules own execution and business-state mutation.

## Search / Investigation Workflow

1. Authorized user searches audit records by approved filters.
2. Logs & Audit applies tenant, entity, redaction, retention, access class, source-module, and role filters.
3. Cross-tenant search is denied by default.
4. Buyer/vendor projections are scoped.
5. Results return metadata and references, not unrestricted payloads.
6. Sensitive search, result export, or payload access is itself audited.

## Payload Exception Workflow

1. Full payload reference is requested.
2. Logs & Audit validates approval, reason, retention class, redaction class, access class, payload size limits, and legal/contract constraints.
3. If approved, Logs & Audit records full payload exception evidence.
4. If not approved, masked payload reference remains the only allowed reference.

## Retention Review Workflow

1. Retention review placeholder is created by policy, age, file type, risk, legal/contract hold, expiration behavior, or manual request.
2. Logs & Audit records retention owner, retention class, redaction class, access class, record references, and review reason.
3. Future retention behavior is applied only according to approved retention policy.

Boundary note: AI agents, users, or source modules must not delete audit logs or bypass retention rules without approved governance.

## Open Questions

- Which source module events create audit records synchronously?
- Which file processing steps are owned by Logs & Audit versus source modules?
- Which manual vendor files can be reprocessed from backup?
- Which workflows need buyer/vendor-facing status views?
- Which reprocess requests should be blocked unless source module approves first?

## Scheduled System Admin Activity Summary Evidence Workflow (Cross-Module PR)

This section adds one architecture-level workflow owned by Logs & Audit File Tracking for the scheduled summary email cross-module hardening pass. Six additional workflows are owned by Notification Platform Service (see notification-platform-service/workflows.md PR-C section), and three additional workflows are owned by Analytics / Reporting (see analytics-reporting/workflows.md PR-C section), for a total of ten workflows across the three modules. Existing Logs & Audit File Tracking workflows (Audit Record Creation, Audit Amendment, File Tracking, API Transmission Logging, Validation And Processing Result, Manual Vendor Order Export, Duplicate Detection, Reprocess / Retry Request, Search / Investigation, Payload Exception, Retention Review) are not modified by this PR.

### Cross-module workflow choreography reaffirmed

Notification initiates schedule. Analytics aggregates. Logs & Audit records evidence.

Logs & Audit File Tracking participates by recording evidence in response to triggers from Notification Platform Service (delivery-side events) and Analytics / Reporting (aggregation and suppression). Logs & Audit File Tracking does not initiate the summary cycle and does not consume source-module events directly.

### Logs & Audit Workflow 10 - Summary Audit Evidence Recording

- **Trigger:** One of the following events from the other two modules:
  - **(a)** Analytics Workflow 5 completes with non-empty aggregation, signaling Logs & Audit to record Activity Summary Generated Evidence.
  - **(b)** Analytics Workflow 6 completes with no-activity suppression, signaling Logs & Audit to record No-Activity Summary Suppression Evidence.
  - **(c)** NPS Workflow 7 dispatches a delivery attempt, signaling Logs & Audit to retain the Activity Summary Delivery Attempt by reference (via the existing Audit Record entity pattern; no new entity).
  - **(d)** NPS Workflow 8 records a delivery failure, signaling Logs & Audit to retain the failure outcome by reference (existing Audit Record pattern).
  - **(e)** NPS Workflow 9 records a cursor advancement, signaling Logs & Audit to retain the cursor-advancement outcome by reference (existing Audit Record pattern). NPS Workflow 9 has two trigger paths: **Trigger A path** (delivery acknowledged on an Activity Summary Delivery Attempt) signals retention of the delivery success and any carry-forward supersession transitions; **Trigger B path** (consumed no-activity-suppression-outcome from Analytics Workflow 6) signals retention of the cursor-advancement-after-suppression-consumption decision. In both cases, Logs & Audit creates an existing-pattern Audit Record on the NPS side of the choreography (no new Logs & Audit entity); the Audit Record references the Activity Summary Configuration, the Reporting Window, and (for Trigger B) the corresponding No-Activity Summary Suppression Evidence record. This NPS-side cursor-advancement Audit Record carries the actual `cursor_advancement_timestamp`, which only exists post-mutation.
  - **(f)** NPS Workflow 1 lifecycle transitions (configuration created, updated, paused, retired), signaling Logs & Audit to retain by reference (existing Audit Record pattern).
- **Steps:**
  1. **For trigger (a):** create an Activity Summary Generated Evidence record. Set `activity_summary_configuration_reference`, `activity_summary_reporting_window_reference`, `activity_summary_aggregation_record_reference`, `generation_timestamp`, `actor_reference`, `carry_forward_window_reference_collection` (mirroring the window's collection), `retention_class`, `redaction_class`, `access_class`. Create the canonical Audit Record cross-reference via the existing Audit Record Creation Workflow. Emit `audit.activity-summary-evidence.recorded`.
  2. **For trigger (b):** create a No-Activity Summary Suppression Evidence record. Set `activity_summary_configuration_reference`, `activity_summary_reporting_window_reference`, `evaluation_timestamp`, `suppression_reason = no_source_facts_in_window`, `window_start_timestamp`, `window_end_timestamp`, `source_fact_zero_activity_assertion_reference`, `retention_class`, `redaction_class`, `access_class`. **The record is created BEFORE Notification Platform Service consumes the suppression outcome and advances its own cursor; therefore the record does NOT carry a cursor-advancement timestamp.** The cursor-advancement audit reference is recorded separately on the Notification Platform Service side, after NPS Workflow 9 Trigger B path performs the cursor mutation, via the existing Audit Record entity pattern. Create the canonical Audit Record cross-reference for this evidence. Emit `audit.activity-summary-suppression-evidence.recorded`.
  3. **For triggers (c), (d), (e), (f):** use the existing Logs & Audit Audit Record entity pattern. Create an Audit Record with `source_module = notification-platform-service`, `event_action_type` discriminating the specific trigger (delivery attempt dispatched, delivery succeeded, delivery failed, **activity summary cursor advanced on delivery acknowledgement, activity summary cursor advanced on consumed no-activity-suppression-outcome**, configuration created, configuration updated, configuration paused, configuration retired), `related_record_references` pointing to the Activity Summary Delivery Attempt, Activity Summary Configuration, Activity Summary Reporting Window, and/or No-Activity Summary Suppression Evidence as applicable, and appropriate retention/redaction/access classes. The cursor-advancement Audit Record variants (both NPS Workflow 9 trigger paths) carry the `cursor_advancement_timestamp` value produced at NPS-side cursor mutation time; this value exists only post-mutation and is recorded only on the NPS-side Audit Record, not on the Analytics suppression-outcome event nor on the Logs & Audit Suppression Evidence record. No new Logs & Audit-side entity is created for triggers (c)-(f); the existing Audit Record entity is the canonical retention surface for these events.
  4. **In all cases:** the evidence record is immutable per the existing Logs & Audit immutability rule. Amendments use the existing Audit Amendment Workflow.
- **Discipline:**
  - Logs & Audit File Tracking does not modify source records, Activity Summary Configuration records, Activity Summary Reporting Window records, Activity Summary Aggregation Records, or Activity Summary Delivery Attempt records. It creates evidence records that reference these canonical records.
  - Logs & Audit File Tracking does not consume source-module events directly for this PR. The summary-related source events (PR #91, PR #92, PR #94) are consumed by Analytics / Reporting; Logs & Audit File Tracking is involved only when Analytics or Notification Platform signals it.
  - Activity Summary Delivery Attempt lifecycle and Activity Summary Configuration lifecycle remain canonical at Notification Platform Service; Logs & Audit File Tracking retains by reference, not by duplicate lifecycle.
  - Activity Summary Reporting Window lifecycle and Activity Summary Aggregation Record lifecycle remain canonical at Analytics / Reporting; the new Logs & Audit entities (Generated Evidence, Suppression Evidence) are retention surfaces, not lifecycle owners.
  - Retention duration, storage location, search semantics, and access-class enforcement for the new entities follow existing Logs & Audit patterns; PR-C does not introduce new retention classes, redaction classes, or access classes.
  - The Activity Summary Generated Evidence record's `carry_forward_window_reference_collection` mirrors the Reporting Window's collection. This mirroring provides Logs & Audit search traceability without requiring Logs & Audit to query Analytics for the collection at search time.
  - The No-Activity Summary Suppression Evidence record is created **before** Notification Platform Service consumes the suppression outcome and performs the cursor mutation; the Suppression Evidence record therefore does NOT carry a cursor-advancement timestamp. Cursor-advancement audit traceability lives on the NPS-side Audit Record (created later via the existing Audit Record entity pattern, with `event_action_type = activity_summary_cursor_advanced_on_consumed_no_activity_outcome` and `related_record_references` linking back to the Suppression Evidence and Activity Summary Configuration). Together the two retention surfaces (this Logs & Audit Suppression Evidence record plus the NPS-side post-mutation Audit Record) form the full audit trail; neither requires a field that does not exist at its own creation time. Logs & Audit does not perform the cursor mutation.

## PR-A Workflows - Core Evidence Spine

This section defines PR-A workflows layered on the existing baseline. Baseline workflows (Audit Record Creation Workflow, Audit Amendment Workflow, File Tracking Workflow, API Transmission Logging Workflow, Validation And Processing Result Workflow, Manual Vendor Order Export Workflow, Manual Vendor Return Export Workflow, Vendor Shipping Import Workflow, Vendor Return Outcome Import Workflow, Duplicate Detection Workflow, Reprocess/Retry Request Workflow, Search/Investigation Workflow, Payload Exception Workflow, Retention Review Workflow) are preserved without rename or removal. PR-A introduces 8 numbered workflows plus 1 folded behavior.

### Evidence Integrity Hash Recording (folded into PR-A Workflow 1)

Evidence Integrity Hash Recording is folded into PR-A Workflow 1 (Generic Evidence Record Creation). At creation, the hash of the evidence payload (or attachment) is captured into `evidence_hash_reference` on the Evidence Record. Hash algorithm is implementation-level. Periodic integrity verification (re-hash and compare) is future phase, NOT in PR-A.

### PR-A Workflow 1 - Generic Evidence Record Creation

**Trigger:** source module emits an evidence intent (with `evidence_type`, source references, optional snapshot, optional attachment, optional external reference, retention/redaction/access classes, metadata fields).

1. Logs & Audit receives the evidence intent input.
2. If no parent Audit Record exists for the originating action, Logs & Audit creates one via the existing baseline Audit Record Creation Workflow, populating `source_module_reference`, `source_record_reference` (when applicable), `source_snapshot_reference` (when applicable), `actor_reference` or `service_trigger_reference`, `company_scope_reference`, retention/redaction/access classes, and the existing baseline fields.
3. Logs & Audit creates the Evidence Record with all required fields:
   - `evidence_record_id` (canonical identifier).
   - `evidence_schema_version` (the schema version under which this evidence is created).
   - `evidence_type` (the discriminator).
   - `audit_record_reference` (parent Audit Record).
   - `source_module_reference`, `source_record_reference`, `source_snapshot_reference` (where applicable).
   - `source_event_reference` (optional).
   - `external_evidence_reference` sub-structure (optional, populated via PR-A Workflow 4 if applicable).
   - `actor_reference` and/or `service_trigger_reference`.
   - `company_scope_reference`.
   - `evidence_hash_reference` (computed at creation; see folded Integrity Hash Recording behavior).
   - `evidence_attachment_reference` (optional).
   - `raw_evidence_reference` (optional; populated when raw evidence is stored alongside any future redacted view).
   - `redacted_view_reference` (null at PR-A creation; populated by future PR-D workflows).
   - `captured_at` (timestamp from source module).
   - `created_at` (timestamp from Logs & Audit).
   - `created_by_actor_reference` or `created_by_service_trigger_reference`.
   - `correlation_reference`.
   - `trace_reference` (optional).
   - `idempotency_key` (optional but recommended).
   - `replay_safe_dedupe_reference` (optional).
   - `evidence_status = active`.
   - `retention_class` (assigned by source module per PR-A Workflow 7).
   - `redaction_class` (assigned by source module per PR-A Workflow 8).
   - `access_class` (assigned by source module).
   - `restricted_evidence` (default false; source module may set true).
   - `legal_hold_reference` (null at PR-A creation; populated by future PR-D workflows).
   - `audit_reference` (Logs & Audit retention reference).
4. Logs & Audit links the Evidence Record to its parent Audit Record by populating the Audit Record's optional `evidence_record_reference` (back-link).
5. Logs & Audit emits `audit.record.created` (if a new Audit Record was created in step 2) and `audit.evidence.recorded`.

**Folded integrity hash recording:** the `evidence_hash_reference` is computed at step 3 from the evidence payload or attachment. Hash algorithm is implementation-level. The hash is captured once at creation; periodic verification is future phase.

**Boundary discipline:**

- Logs & Audit does NOT validate source-module business logic.
- Logs & Audit does NOT decide whether the evidence is "correct"; it records what the source module emitted.
- Logs & Audit does NOT mutate source-module records.

### PR-A Workflow 2 - Source Module Evidence Reference Capture

**Trigger:** PR-A Workflow 1 step 3 requires source references.

1. The source module populates `source_module_reference` (the module identifier from the Logs & Audit-maintained source module catalog).
2. The source module populates `source_record_reference` (module identifier + record type + record id of the operational record this evidence pertains to). Required for evidence-bearing audits; optional for pure observation events.
3. The source module optionally populates `source_snapshot_reference` (governed by PR-A Workflow 3 minimization rule).
4. Logs & Audit validates that the required references are present per the `evidence_type` (validation rules per evidence_type are PR-C decisions; PR-A documents the architectural requirement only).
5. Missing required references cause evidence creation to be rejected at the source module's call site (architectural rule; concrete error handling is implementation-level).

### PR-A Workflow 3 - Source Snapshot Capture with Minimization Rule

**Trigger:** PR-A Workflow 2 step 3, when the source module elects to attach a source snapshot.

1. The source module prepares a minimized snapshot per the Source Snapshot Minimization Rule: selected metadata + hashes + references, NOT a full operational record copy.
2. The source module computes the snapshot's own hash (distinct from the Evidence Record's `evidence_hash_reference`).
3. The source module stores the snapshot (typically via `evidence_attachment_reference` mechanism) and produces the `source_snapshot_reference` identifier.
4. Logs & Audit records the Source Snapshot Reference on the Evidence Record.
5. If the source module needs to store a full operational record copy (exceptional), it must go through the existing baseline Full Payload Exception Record workflow (which requires approval, reason, retention class, redaction class, access class, payload size checks).
6. PR-A does NOT enforce the minimization at write time; enforcement is implementation-level. The minimization rule is documented in the architecture; misuse (full operational record copy without exception approval) is a source-module bug, reviewable via PR-D surfaces.

**Source Snapshot Minimization Rule (reaffirmation):**

- Source snapshots should use selected metadata, hashes, and references when possible.
- Source snapshots should NOT blindly copy full operational records.
- Snapshots are evidence, not operational truth.
- Snapshot payload shape remains architecture-level, not OpenAPI-specific.

### PR-A Workflow 4 - External Evidence Reference Capture

**Trigger:** an external-system coordination event needs to attach to an Evidence Record (typically driven by Integration Management transport activity).

1. Integration Management (or another transport-owning module) prepares the external coordination evidence: provider response reference, external ID, external file reference, external task/project reference, external timestamp, webhook receipt reference, retry/failure transport evidence reference, external evidence hash where available.
2. Logs & Audit records the External Evidence Reference as a sub-structure on the Evidence Record (NOT a separate entity).
3. The Evidence Record's `external_evidence_reference` field is populated with the sub-structure values.
4. Logs & Audit emits `audit.evidence.recorded` with the External Evidence Reference included in the payload (no separate `audit.external-evidence-reference.recorded` event in PR-A).

**External-Tool-Not-Source-of-Truth Rule (reaffirmation):**

- Integration Management owns connector execution and transport.
- External evidence references are coordination and proof only.
- External systems must NEVER become CIXCI operational source of truth.
- Investigators and source-module logic must trace back to CIXCI source-module records, not to external system records, for canonical business state.

### PR-A Workflow 5 - Evidence Amendment

**Trigger:** authorized actor or source-module service submits an amendment intent against an existing Evidence Record (typo, missing field, additional context, actor disambiguation, redaction class clarification, source_snapshot_reference link added post-creation).

1. Logs & Audit receives the amendment input: target Evidence Record reference, amendment reason, amendment payload (optional, may be null when the amendment is purely a metadata clarification), amendment actor reference.
2. Tenant Company `check_access` is invoked for the amendment actor. PR-A introduces no new role or capability flag; authority flows through existing patterns.
3. Logs & Audit creates the Evidence Amendment Record with:
   - `evidence_amendment_record_id`.
   - `target_evidence_record_reference`.
   - `amendment_reason_reference`.
   - `amendment_actor_reference`.
   - `amendment_service_trigger_reference` (optional).
   - `tenant_company_authority_reference`.
   - `amendment_payload_reference` (optional).
   - `created_at`.
   - `audit_record_reference` (the amendment itself is an audit-worthy action; an Audit Record is created for it).
   - `audit_reference`.
4. Logs & Audit attaches the Evidence Amendment Record to the target Evidence Record's `amendment_record_reference_collection` (append; existing collection is preserved).
5. The target Evidence Record's `evidence_status` transitions from `active` to `amended_with_amendments` (a soft transition; downstream consumers continue to read the evidence with amendments alongside).
6. **The target Evidence Record is NEVER mutated.** Only the amendment record collection grows.
7. Logs & Audit emits `audit.evidence-amendment.recorded`.

**Boundary discipline:**

- Append-only.
- The Evidence Amendment Record is NEVER mutated after creation.
- The target Evidence Record's content is NEVER mutated.
- Source modules choose between amendment (Workflow 5) and supersession (Workflow 6) based on whether the underlying source-module record was corrected (supersession) or whether the evidence record itself needs clarification (amendment).

### PR-A Workflow 6 - Evidence Supersession

**Trigger:** source module recognizes that an operational record has been corrected, reissued, or replaced, and the new state needs new evidence.

1. The source module emits a supersession intent: prior Evidence Record reference, supersession reason, source_module_correction_reference (the operational record correction that triggered the supersession), and the new evidence content (which becomes a full new Evidence Record).
2. Tenant Company `check_access` is invoked for the supersession actor.
3. Logs & Audit creates a NEW Evidence Record via PR-A Workflow 1 with all required fields (new `evidence_record_id`, new `audit_record_reference` for the supersession audit action, new full field set). The new Evidence Record's `evidence_status = active`.
4. Logs & Audit creates an Evidence Supersession Record with:
   - `evidence_supersession_record_id`.
   - `prior_evidence_record_reference` (the Evidence Record being superseded).
   - `new_evidence_record_reference` (the new Evidence Record).
   - `supersession_reason_reference`.
   - `supersession_actor_reference`.
   - `supersession_service_trigger_reference` (optional).
   - `tenant_company_authority_reference`.
   - `source_module_correction_reference` (optional but recommended).
   - `created_at`.
   - `audit_record_reference`.
   - `audit_reference`.
5. The prior Evidence Record's `supersession_reference` is populated, pointing to the Evidence Supersession Record.
6. The prior Evidence Record's `evidence_status` transitions from `active` (or `amended_with_amendments`) to `superseded`.
7. **Both the prior Evidence Record and the new Evidence Record remain in storage and remain queryable.** Neither is mutated beyond the supersession_reference and evidence_status field updates on the prior record (which are the supersession transitions themselves; PR-A treats these as the ONLY mutations allowed on the prior Evidence Record, and they are themselves part of the supersession lineage record).
8. Logs & Audit emits `audit.evidence-supersession.recorded` and `audit.evidence.recorded` for the new Evidence Record creation.

**Boundary discipline:**

- The supersession lineage is append-only.
- Source modules own the business correction decision; Logs & Audit records the lineage.
- The prior Evidence Record's evidence_status and supersession_reference field updates are the only mutations permitted under PR-A on a superseded Evidence Record, and these are the supersession transition itself (not a generic mutation surface).

### PR-A Workflow 7 - Evidence Retention Classification at Creation

**Trigger:** PR-A Workflow 1 step 3 requires `retention_class` assignment.

1. The source module assigns `retention_class` based on the evidence type, sensitivity, business context, and existing baseline retention class semantics.
2. Logs & Audit records the `retention_class` on the Evidence Record.
3. The class is locked at creation.
4. PR-D will introduce review workflows that may upgrade or downgrade the class (with audit trail via Evidence Amendment Record). PR-A captures only the at-creation assignment.

**Proposal-level `retention_class` values (PR-A introduces enumeration shape; PR-D defines durations):**

- `transient` - short-term operational logs (e.g., transient API transmission attempts).
- `standard` - default retention class.
- `extended` - longer retention for compliance-sensitive evidence.
- `regulatory` - formal regulatory hold class.
- `legal_hold` - under legal hold (overlays with `legal_hold_reference` field).
- `audit_critical` - immutable for full audit trail (the longest standard class).

PR-A documents the enumeration shape; concrete durations per class are PR-D.

### PR-A Workflow 8 - Evidence Redaction Classification at Creation

**Trigger:** PR-A Workflow 1 step 3 requires `redaction_class` assignment.

1. The source module assigns `redaction_class` based on the evidence content's sensitivity to buyers, vendors, internal operations, customers, pricing, invoices, warranties, tenant security, or audit-only.
2. Logs & Audit records the `redaction_class` on the Evidence Record.
3. The class is locked at creation.
4. PR-D will introduce redaction transformation workflows that produce `redacted_view_reference` projections while preserving `raw_evidence_reference`. PR-A captures only the at-creation classification.

**Existing baseline `redaction_class` values (preserved verbatim by PR-A):**

- `public_metadata_placeholder`
- `buyer_visible_audit`
- `vendor_visible_audit`
- `internal_operations`
- `customer_sensitive_restricted`
- `pricing_sensitive_restricted`
- `invoice_sensitive_restricted`
- `warranty_sensitive_restricted`
- `tenant_security_restricted`
- `audit_only`

PR-A also formalizes `access_class` (separate from `redaction_class`):

**Proposal-level `access_class` values (PR-A introduces enumeration shape; PR-D defines access matrix):**

- `public_metadata`
- `buyer_visible`
- `vendor_visible`
- `internal_operations`
- `system_admin_only`
- `compliance_only`

PR-A documents the enumeration shape; concrete access matrices per class (tenant / parent / child / role projection) are PR-D.

PR-A also formalizes `restricted_evidence` (boolean flag) and the `legal_hold_reference` placeholder field. PR-D defines the Legal Hold entity, lifecycle, application workflow, and lift workflow.

## PR-D Workflows - Retention / Redaction / Legal Hold / Access Governance

PR-D adds exactly **13 numbered workflows** on top of PR-A's 8 workflows, PR-B's 10 workflows, and PR-C's architectural usage patterns (zero new numbered workflows). PR-D workflows formalize the governance lifecycle for retention, redaction, legal hold, and access. All existing workflows are preserved without rename.

### PR-D Workflow 1 - Retention Review Trigger Workflow

**Purpose.** Detects when an Evidence Record reaches its retention review point based on `retention_class` and named retention policy reference from the Retention Policy Matrix.

**Steps (architectural):**

1. Periodic / event-driven review process scans Evidence Records (or batches) approaching the review point per `retention_class` + named policy reference.
2. For each Evidence Record (or batch) at review point, emit `audit.retention-review.required` event.
3. Downstream Retention Policy Evaluation Workflow consumes the event.

**Inputs:** Evidence Record `captured_at` + `retention_class`; Retention Policy Matrix; named retention policy reference.
**Outputs:** `audit.retention-review.required` event.
**Boundary:** Does NOT take disposition action; does NOT mutate Evidence Record. Per Immutable-Evidence-Retention Rule.

### PR-D Workflow 2 - Retention Policy Evaluation Workflow

**Purpose.** Evaluates Retention Policy Matrix against an Evidence Record (or batch); produces disposition recommendation; mandatory Legal Hold Record check.

**Steps (architectural):**

1. Receive `audit.retention-review.required` event (or direct invocation).
2. Look up Retention Policy Matrix row for the evidence's (family, evidence_type, retention_class).
3. Determine policy-recommended disposition state (retain / archive / purge_eligible / preserved).
4. Invoke Legal Hold Blocks Purge Workflow (PR-D Workflow 7) for any purge-related recommendation.
5. If Legal Hold matched -> recommended disposition state is overridden to `purge_blocked_by_hold`.
6. Pass disposition recommendation to Retention Disposition Workflow (PR-D Workflow 3).

**Inputs:** Evidence Record; Retention Policy Matrix; Legal Hold Records.
**Outputs:** Disposition recommendation.
**Boundary:** Does NOT execute disposition; does NOT mutate Evidence Record. Per Legal-Hold-Overrides-Purge Rule, MUST invoke Legal Hold check.

### PR-D Workflow 3 - Retention Disposition / Archive / Purge Decision Workflow

**Purpose.** Creates Retention Disposition Record with the chosen disposition state; emits `audit.retention-disposition.recorded` event; triggers downstream purge executor for `purge_eligible` (implementation-level).

**Steps (architectural):**

1. Receive disposition recommendation from Retention Policy Evaluation Workflow.
2. Verify Legal Hold check result from PR-D Workflow 7.
3. Create Retention Disposition Record with `disposition_state`, `disposition_reason`, `retention_policy_reference`, `legal_hold_check_result`, `legal_hold_reference` (if blocked), `proof_class`, `proof_reference`, `actor_reference` or `service_trigger_reference`, `company_scope_reference`, `audit_record_reference`.
4. Emit `audit.retention-disposition.recorded` event carrying `disposition_state` discriminator.
5. For `purge_eligible` state, signal downstream purge executor (implementation-level).
6. For `purged_reference_only` state, ensure File Tracking Record metadata + `file_hash_reference` preserved per File-Metadata-Outlives-Payload Rule (file-backed evidence).
7. Record `audit.evidence-access.recorded` if disposition required restricted-access evaluation.

**Inputs:** Disposition recommendation; Legal Hold check; Retention Policy Matrix.
**Outputs:** Retention Disposition Record; `audit.retention-disposition.recorded` event; (implementation-level) purge executor signal.
**Boundary:** Per Retention-Disposition-Append-Only Rule, append-only. Per Immutable-Evidence-Retention Rule, does NOT mutate Evidence Record.

### PR-D Workflow 4 - Redacted View Creation Workflow

**Purpose.** Creates a redacted view of raw evidence per the Redaction Policy Matrix; produces a Redaction Transformation Record; populates `redacted_view_reference` (PR-A) and `masked_payload_reference` (PR-B, for file-backed) on the parent Evidence Record / File Tracking Record.

**Steps (architectural):**

1. Source-module evidence emission triggers redacted view creation, OR retroactive policy-driven redaction triggers re-creation.
2. Look up Redaction Policy Matrix row for the evidence's (family, evidence_type, redaction_class).
3. Determine target audience(s) for redacted view (buyer / vendor / internal / audit_only).
4. For each target audience, apply redaction transformation per the policy matrix.
5. Create Redaction Transformation Record (append-only) with `raw_evidence_reference`, `redacted_view_reference`, `redaction_class`, `redaction_audience`, `redaction_reason`, `redaction_version`, `actor_reference` or `service_trigger_reference`, `transformation_timestamp`, `company_scope_reference`, `masked_payload_reference` (for file-backed), `policy_matrix_reference`, `audit_record_reference`.
6. Emit `audit.redaction.applied` event carrying `redaction_class`, `redaction_audience`, `redaction_version` discriminators.
7. Populate `redacted_view_reference` on Evidence Record (PR-A field).
8. For file-backed evidence, populate `masked_payload_reference` on File Tracking Record (PR-B field).

**Inputs:** Evidence Record with raw_evidence_reference; Redaction Policy Matrix.
**Outputs:** Redaction Transformation Record; `audit.redaction.applied` event; populated `redacted_view_reference` and (for file-backed) `masked_payload_reference`.
**Boundary:** Per Redaction-Never-Overwrites-Raw Rule, raw_evidence_reference NEVER mutated. Per Redaction-Transformation-Append-Only Rule, append-only. Per Redacted-Views-Per-Audience Rule, multiple records per Evidence Record allowed.

### PR-D Workflow 5 - Legal Hold Apply Workflow

**Purpose.** Creates Legal Hold Record with `status = applied`; emits `audit.legal-hold.applied` event; recorded via Audit Record (PR-A).

**Steps (architectural):**

1. Authority (Compliance / legal team; documented expectation; outside Logs & Audit) initiates legal hold.
2. Authority's actor passes Tenant Company `check_access` for hold-apply authority (Tenant Company owns; PR-D documents expectation).
3. Create Legal Hold Record with `hold_id`, `hold_scope_discriminators` (at least one populated; `company_scope_reference` REQUIRED single-tenant), `reason`, `authority`, `owner`, `applied_timestamp`, `applied_actor_reference` or `applied_service_trigger_reference`, `status = applied`, `hold_audit_reference`.
4. Emit `audit.legal-hold.applied` event.
5. (Evidence Records under scope are NOT mutated; scope match is evaluated at retention disposition time per Legal-Hold-Does-Not-Mutate-Evidence Rule.)

**Inputs:** Authority reference; scope discriminators; reason; owner.
**Outputs:** Legal Hold Record; `audit.legal-hold.applied` event.
**Boundary:** Per Legal-Hold-Does-Not-Mutate-Evidence Rule, NO Evidence Record mutation. Per Tenant-Company-Owns-Authority Rule, Tenant Company `check_access` evaluates authority.

### PR-D Workflow 6 - Legal Hold Release Workflow

**Purpose.** Transitions Legal Hold Record `status` from `applied` to `released` (or `lapsed`); populates `released_timestamp` and `release_authority`; emits `audit.legal-hold.released` event.

**Steps (architectural):**

1. Authority (Compliance / legal team) initiates hold release.
2. Authority's actor passes Tenant Company `check_access` for hold-release authority.
3. Transition Legal Hold Record: set `status = released` (or `lapsed`); populate `released_timestamp`, `release_authority`, `released_actor_reference` or `released_service_trigger_reference`.
4. Emit `audit.legal-hold.released` event.
5. (Downstream Retention Disposition Workflow may re-evaluate previously blocked dispositions; not part of this workflow.)

**Inputs:** Legal Hold Record reference; release authority; release reason (optional).
**Outputs:** Legal Hold Record (status transitioned); `audit.legal-hold.released` event.
**Boundary:** Per Legal-Hold-Apply-Append-Only Rule, release populates fields; does NOT delete or rewrite. Per Legal-Hold-Does-Not-Mutate-Evidence Rule, NO Evidence Record mutation.

### PR-D Workflow 7 - Legal Hold Blocks Purge Workflow

**Purpose.** Invoked by Retention Policy Evaluation Workflow (PR-D Workflow 2). Checks if Evidence Record (or batch) is under any active Legal Hold Record. If yes, disposition state is forced to `purge_blocked_by_hold`.

**Steps (architectural):**

1. Receive Evidence Record (or batch) reference and proposed disposition state.
2. Query Legal Hold Records with `status = applied` matching any of:
   - Evidence's `evidence_type` in `evidence_type_scope`.
   - Evidence's `evidence_family` in `evidence_family_scope`.
   - Evidence's `company_scope_reference` matches Legal Hold `company_scope_reference`.
   - File Tracking Record (for file-backed evidence) in `file_scope`.
   - Source module in `source_module_scope`.
3. If ANY active Legal Hold matches, return blocked = true; pass matching `legal_hold_reference` to caller.
4. If NO active Legal Hold matches, return blocked = false.

**Inputs:** Evidence Record reference; proposed disposition state.
**Outputs:** blocked (boolean); `legal_hold_reference` (if blocked).
**Boundary:** Per Legal-Hold-Overrides-Purge Rule, blocking is MANDATORY check before any purge-related disposition. Per Legal-Hold-Does-Not-Mutate-Evidence Rule, query only; no Evidence Record mutation.

### PR-D Workflow 8 - Evidence Access Recording Workflow

**Purpose.** Creates hardened Audit Access Record for every access decision (attempted / granted / denied); emits `audit.evidence-access.recorded` event.

**Steps (architectural):**

1. Access request reaches Logs & Audit (e.g., reviewer requests Evidence Record; service requests source evidence).
2. Determine `actor_reference` or `service_trigger_reference` (one populated).
3. Tenant Company `check_access` evaluates whether actor has authority for the evidence's `access_class` + `restricted_evidence` + `legal_hold_state`. Logs & Audit records the outcome; Tenant Company is canonical.
4. Logs & Audit Access Policy Matrix evaluates the access expectations per actor type, company relationship, parent / child scope, `access_policy_tier`.
5. Both must succeed for `access_result = granted`. Failure of EITHER -> `access_result = denied` with `denial_reason`.
6. If access in progress / partial / interrupted -> `access_result = attempted` (non-terminal).
7. Determine `view_type` (`raw` if requested AND granted with reason; `redacted` default).
8. Create hardened Audit Access Record with `access_result`, `view_type`, `access_reason_reference` (REQUIRED for raw / restricted / break-glass), `denial_reason` (if denied), `break_glass_flag`, `access_class_evaluated`, `redaction_class_evaluated`, `restricted_evidence_flag`, `legal_hold_state_at_access`, `redaction_transformation_reference` (if redacted view served), `actor_reference` or `service_trigger_reference`, `company_scope_reference`, `evidence_record_reference`, `audit_record_reference`.
9. Emit `audit.evidence-access.recorded` event carrying `access_result` and `view_type` discriminators.

**Inputs:** Access request; actor / service; evidence reference.
**Outputs:** Hardened Audit Access Record; `audit.evidence-access.recorded` event.
**Boundary:** Per All-Access-Logged Rule, every attempt logged. Per Service-Identity-Access-Logged Rule, service identity logged identically. Per Tenant-Company-Owns-Authority Rule, Tenant Company is canonical for authority. Per Raw-Evidence-Access-Exceptional Rule, raw access exceptional.

### PR-D Workflow 9 - Raw Evidence Access Workflow

**Purpose.** Handles requests for `raw_evidence_reference` access; requires `access_reason_reference`; sets `view_type = raw`; subjects to elevated authority check.

**Steps (architectural):**

1. Actor / service explicitly requests raw view (default is redacted per Redacted-View-Default Rule).
2. Verify `access_reason_reference` provided (REQUIRED for raw access).
3. Tenant Company `check_access` evaluates elevated authority requirement for raw access of this evidence's `access_class` + `restricted_evidence`.
4. Logs & Audit Access Policy Matrix evaluates raw access expectations per actor type.
5. If both succeed -> serve raw view; if either fails -> deny.
6. Invoke Evidence Access Recording Workflow (PR-D Workflow 8) with `view_type = raw`.

**Inputs:** Raw access request; access reason; actor / service.
**Outputs:** Raw view served (if granted) OR denial; hardened Audit Access Record via PR-D Workflow 8.
**Boundary:** Per Raw-Evidence-Access-Exceptional Rule, exceptional + logged + reason required.

### PR-D Workflow 10 - Restricted Evidence Access Workflow

**Purpose.** Handles access to Evidence Records with `restricted_evidence = true`. Evaluates elevated authority requirements per Access Policy Matrix; requires `access_reason_reference`; defaults to redacted view; raw escalation via PR-D Workflow 9.

**Steps (architectural):**

1. Access request reaches Logs & Audit for Evidence Record with `restricted_evidence = true`.
2. Verify `access_reason_reference` provided (REQUIRED for restricted access).
3. Tenant Company `check_access` evaluates elevated authority for `access_class` (typically `compliance_only` or `system_admin_only` per PR-D sensitivity mapping).
4. Logs & Audit Access Policy Matrix evaluates restricted access expectations.
5. If both succeed -> serve redacted view (or raw if explicitly requested via PR-D Workflow 9 with separate raw access governance).
6. Invoke Evidence Access Recording Workflow (PR-D Workflow 8) with `restricted_evidence_flag = true`.

**Inputs:** Restricted access request; access reason; actor / service.
**Outputs:** Redacted view served (if granted) OR raw view (via PR-D Workflow 9) OR denial.
**Boundary:** Per Tenant-Company-Owns-Authority Rule, Tenant Company canonical. Per All-Access-Logged Rule, attempt logged.

### PR-D Workflow 11 - Full Payload Exception Access Review Workflow

**Purpose.** Handles access to PR-B Full Payload Exception Record content. Defaults to masked / redacted view per Full Payload Exception Redaction Requirement. Raw full-payload access requires escalation per PR-D Workflow 9.

**Steps (architectural):**

1. Access request reaches Logs & Audit for evidence with Full Payload Exception Record (PR-B Workflow 10).
2. Default view: masked / redacted (per PR-B `masked_payload_reference` if file-backed; per `redacted_view_reference` otherwise).
3. If raw full-payload requested -> escalate to PR-D Workflow 9 (Raw Evidence Access).
4. Invoke Evidence Access Recording Workflow (PR-D Workflow 8) for every access decision.

**Inputs:** Full payload access request; actor / service.
**Outputs:** Masked / redacted view (default) OR raw escalation; hardened Audit Access Record.
**Boundary:** Per Redacted-View-Default Rule, masked / redacted default. Per Raw-Evidence-Access-Exceptional Rule, raw exceptional.

### PR-D Workflow 12 - File-Backed Evidence Governance Workflow

**Purpose.** Specializes retention / redaction / legal hold / access governance for file-backed evidence (Evidence Records with `file_tracking_record_reference`). Preserves hash, manages tombstone storage reference, defaults to masked payload.

**Steps (architectural):**

1. File-backed Evidence Record reaches retention review / disposition / redaction / access governance.
2. For retention disposition reaching `purged_reference_only`:
   - File Tracking Record metadata preserved.
   - `file_hash_reference` preserved (per File-Metadata-Outlives-Payload Rule).
   - `file_storage_reference` transitioned to tombstone state (payload deleted; reference preserved as tombstone marker).
3. For redaction (PR-D Workflow 4):
   - `masked_payload_reference` (PR-B File Tracking Record field) is the file-backed equivalent of `redacted_view_reference`.
   - Redaction Transformation Record links raw file payload -> masked payload.
4. For legal hold (PR-D Workflow 7):
   - Legal Hold Record `file_scope` blocks file payload purge; `purge_blocked_by_hold` recorded.
5. For access (PR-D Workflows 8 / 9 / 10):
   - Default access serves masked / redacted file payload via `masked_payload_reference`.
   - Raw file payload access requires escalation per PR-D Workflow 9.
   - Downloaded files are typically served views (redacted / masked); raw download requires escalation.

**Inputs:** File-backed Evidence Record; File Tracking Record reference (PR-B); Retention / Redaction / Legal Hold / Access governance signals.
**Outputs:** File-backed disposition; file-backed redaction; file-backed access decision.
**Boundary:** Per File-Metadata-Outlives-Payload Rule, hash preserved. Per Legal-Hold-Overrides-Purge Rule, hold blocks file purge. Per Redacted-View-Default Rule, masked payload default.

### PR-D Workflow 13 - Tenant / Parent / Child Evidence Access Evaluation Workflow

**Purpose.** Evaluates Access Policy Matrix against requester's actor type, company relationship, parent / child scope; defers permission grant to Tenant Company `check_access`; records via hardened Audit Access Record.

**Steps (architectural):**

1. Access request reaches Logs & Audit with requester identity.
2. Determine actor type (CIXCI System Admin, Compliance / Audit Reviewer, source-module service identity, buyer parent admin, buyer child admin, vendor parent admin, vendor child admin, device manufacturer, etc.).
3. Determine company relationship with evidence's `company_scope_reference` (self / parent / child / sibling / unrelated).
4. If unrelated (cross-tenant), deny by default per existing baseline.
5. If parent / child / sibling, evaluate Tenant Company hierarchy authority (Tenant Company owns hierarchy semantics; PR-D references but does NOT define).
6. Logs & Audit Access Policy Matrix evaluates per `access_class_evaluated`, `redaction_class_evaluated`, `restricted_evidence_flag`, `legal_hold_state_at_access`, `access_policy_tier`.
7. Tenant Company `check_access` evaluates permission grant.
8. Both must succeed for `access_result = granted`.
9. Invoke Evidence Access Recording Workflow (PR-D Workflow 8) to log the decision.

**Inputs:** Access request; requester identity; evidence reference.
**Outputs:** Access decision (granted / denied / attempted) via PR-D Workflow 8.
**Boundary:** Per Tenant-Company-Owns-Authority Rule, Tenant Company canonical. Per No-New-Tenant-Roles-In-PR-D Rule, PR-D does NOT define hierarchy / role semantics; references Tenant Company.

## PR-E Workflows - Search / Query / Review / Investigation / Audit Report Export

PR-E adds exactly **13 numbered workflows** on top of PR-A's 8 workflows, PR-B's 10 workflows, PR-C's 7 architectural usage patterns, and PR-D's 13 workflows. PR-E workflows formalize search / query / review / investigation / audit report export over PR-D-governed evidence. All existing workflows are preserved without rename.

### PR-E Workflow 1 - Evidence Search Session Creation Workflow

**Purpose.** Creates Evidence Search Session entity for a requested search; emits `audit.search.executed`; verifies tenant scope; verifies sensitive-filter purpose reference if applicable.

**Steps (architectural):**

1. Receive search request with query, filter set, and scope reference.
2. Verify `actor_reference` OR `service_trigger_reference` (one populated).
3. Verify `company_scope_reference` matches requester's tenant scope per PR-A discipline; cross-tenant denied.
4. Compute `sensitive_filter_used` boolean from filter set (per PR-E discriminator definition).
5. If `sensitive_filter_used = true`, verify `search_initiated_purpose_reference` is provided (REQUIRED per Sensitive-Search-Logged Rule); reject with `access_result = denied` via PR-D Workflow 8 if missing.
6. Create Evidence Search Session entity (append-only).
7. Emit `audit.search.executed` event carrying `query_target` and `sensitive_filter_used` discriminators.

**Inputs:** Search request; actor / service; scope reference; filter set.
**Outputs:** Evidence Search Session entity; `audit.search.executed` event.
**Boundary:** Per Search-Defers-To-PR-D-Access-Governance Rule, per-result access flows through PR-D Workflow 8 (not this workflow). Per Sensitive-Search-Logged Rule, purpose reference required for sensitive filters.

### PR-E Workflow 2 - Evidence Search Query Evaluation Workflow

**Purpose.** Evaluates the Evidence Search Query against the Search Index Projection per the Search Filter Set; produces a result candidate set; result candidates flow to per-result access evaluation.

**Steps (architectural):**

1. Receive Evidence Search Session reference.
2. Apply Search Filter Set against Search Index Projection (implementation-level query engine; PR-E does NOT lock).
3. Apply Search Scope Reference bounds (tenant scope; parent / child scope per PR-D Workflow 13).
4. Per Index-Stale-Acceptable Rule, result candidate set may be eventually consistent; freshness indicator MAY be returned (implementation).
5. Pass result candidate set to Search Result Access Evaluation Workflow (PR-E Workflow 4).

**Inputs:** Evidence Search Session.
**Outputs:** Result candidate set.
**Boundary:** Per Index-Is-Not-Source-of-Truth Rule, results derived; canonical records are PR-A / PR-B / PR-C / PR-D. Per No-Raw-Payload-Indexing Rule, raw payloads NEVER indexed.

### PR-E Workflow 3 - Search Scope / Filter Evaluation Workflow

**Purpose.** Evaluates the Search Scope Reference and Search Filter Set for authorization purposes; invokes Tenant Company `check_access` for the bounded scope; invokes PR-D Workflow 13 for parent / child scope evaluation.

**Steps (architectural):**

1. Receive Evidence Search Session reference.
2. Evaluate Search Scope Reference against requester's tenant authority via Tenant Company `check_access`.
3. Apply parent / child scope evaluation per PR-D Workflow 13.
4. Cross-tenant denied by default; if cross-tenant requested, reject with `access_result = denied` via PR-D Workflow 8.
5. Pass evaluated scope to Query Evaluation Workflow (PR-E Workflow 2).

**Inputs:** Evidence Search Session; Search Scope Reference.
**Outputs:** Authorized scope set OR denial via PR-D Workflow 8.
**Boundary:** Per Tenant-Company-Owns-Authority Rule (PR-D), Tenant Company canonical. Per Search-Not-Source-of-Truth Rule, scope filters on Evidence Records, NOT on source-module operational state.

### PR-E Workflow 4 - Search Result Access Evaluation Workflow

**Purpose.** For each result candidate, invokes PR-D Workflow 8 (Evidence Access Recording) to evaluate per-result access. Hidden-Denied-Result Rule applies: denied results hidden from result set and counts by default.

**Steps (architectural):**

1. Receive result candidate set from PR-E Workflow 2.
2. For each candidate Evidence Record:
   a. Invoke PR-D Workflow 8 with requester context; PR-D evaluates `access_class` + `restricted_evidence` + `legal_hold_state` + redaction audience match against Tenant Company `check_access` and PR-D Access Policy Matrix.
   b. PR-D Workflow 8 creates hardened Audit Access Record with `access_result` (granted / denied / attempted-non-terminal) and `view_type` (raw / redacted; default `redacted`).
   c. Hardened Audit Access Record references the Evidence Search Session via `search_session_reference` (or via correlation_reference if direct field unavailable).
3. For results with `access_result = granted`, include in result set with selected view (redacted by default).
4. For results with `access_result = denied`:
   - Hidden from result set and counts by default (Hidden-Denied-Result Rule).
   - If requester is authorized audit / compliance reviewer per PR-D Access Policy Matrix, MAY include in visible-denied metadata view per Visible-Denied-Metadata-Minimized Rule: minimized metadata only (presence-of-evidence indicator, `evidence_type`, `evidence_family`, `captured_at`, `redaction_class`, `access_class`, denial reason reference); MUST NOT expose raw content or restricted details.
5. Pass accessible result set to Redacted Search Result Rendering Workflow (PR-E Workflow 5).

**Inputs:** Result candidate set; requester context.
**Outputs:** Accessible (granted) result set + (when authorized) visible-denied minimized metadata set.
**Boundary:** Per Search-Defers-To-PR-D-Access-Governance Rule, every access via PR-D Workflow 8. Per Hidden-Denied-Result Rule, denied hidden by default including counts. Per Visible-Denied-Metadata-Minimized Rule, visible-denied minimized + reviewer-only.

### PR-E Workflow 5 - Redacted Search Result Rendering Workflow

**Purpose.** Renders each accessible result via the appropriate Redaction Transformation Record per requester's audience per Per-Audience-Result-Selection Rule and PR-D Redacted-Views-Per-Audience Rule.

**Steps (architectural):**

1. Receive accessible result set from PR-E Workflow 4.
2. For each result:
   a. Determine requester's audience (`buyer` / `vendor` / `internal` / `audit_only`) per PR-D Workflow 13 evaluation result.
   b. Select the appropriate PR-D Redaction Transformation Record for the requester's audience.
   c. Render result via `redacted_view_reference` (PR-A field) corresponding to the selected Redaction Transformation Record.
   d. For file-backed results, render via PR-B `masked_payload_reference` (per File-Backed-Result-Masked-Default Rule).
   e. For results matching `retention_disposition_state = purged_reference_only`, return metadata / hash / tombstone view only (per Purged-Reference-Only-Metadata-View Rule).
   f. For results matching `retention_disposition_state = archive`, indicate availability state (per Archived-Result-Availability-State Rule).
   g. For results with PR-B Full Payload Exception Record, default to masked view (per Full-Payload-Exception-No-Raw-Preview Rule).
   h. For source snapshot previews, render minimized content (per Source-Snapshot-Minimization-In-Preview Rule).
   i. For `restricted_evidence = true` results visible to authorized requesters, render redacted view; do NOT expose restricted_evidence indicator to unauthorized audiences (per Restricted-Flag-Visibility-Scoped Rule).
   j. For results under active Legal Hold, do NOT expose legal hold indicator to unauthorized audiences (per Legal-Hold-Flag-Visibility-Scoped Rule).
3. Return rendered result set.

**Inputs:** Accessible result set; requester audience.
**Outputs:** Rendered result set.
**Boundary:** Per Search-Result-Redacted-By-Default Rule, default to redacted. Per Per-Audience-Result-Selection Rule, audience-aware. Per multiple sensitivity rules, flag visibility scoped.

### PR-E Workflow 6 - Raw Evidence Retrieval Request Workflow

**Purpose.** Handles explicit raw evidence retrieval requests within a search session; invokes PR-D Workflow 9 (Raw Evidence Access); `access_reason_reference` REQUIRED per PR-D Raw-Evidence-Access-Exceptional Rule.

**Steps (architectural):**

1. Receive raw retrieval request from within an Evidence Search Session context.
2. Verify `access_reason_reference` provided (REQUIRED per PR-D Raw-Evidence-Access-Exceptional Rule); reject with `access_result = denied` if missing.
3. Invoke PR-D Workflow 9 (Raw Evidence Access) per request.
4. PR-D Workflow 9 evaluates elevated authority, populates `view_type = raw` on hardened Audit Access Record, and (if granted) serves raw view.

**Inputs:** Raw retrieval request; access reason reference.
**Outputs:** Raw view (if granted) OR denial via PR-D Workflow 9.
**Boundary:** Per Raw-Evidence-Access-Exceptional Rule (PR-D), exceptional. Search workflow defers to PR-D Workflow 9.

### PR-E Workflow 7 - File-Backed Evidence Retrieval Workflow

**Purpose.** Handles file-backed evidence retrieval within a search session; defaults to masked / redacted payload per File-Backed-Result-Masked-Default Rule; raw file payload requires PR-D Workflow 9 escalation; invokes PR-D Workflow 12 (File-Backed Evidence Governance).

**Steps (architectural):**

1. Receive file-backed retrieval request from within an Evidence Search Session context.
2. If default (masked / redacted) view requested:
   a. Serve via PR-B `masked_payload_reference`.
   b. Log access via PR-D Workflow 8.
3. If raw file payload requested:
   a. Escalate to PR-E Workflow 6 -> PR-D Workflow 9.
4. For file-backed evidence with `retention_disposition_state = purged_reference_only`, return file metadata + hash + tombstone storage reference (per File-Metadata-Outlives-Payload Rule and Purged-Reference-Only-Metadata-View Rule).

**Inputs:** File-backed retrieval request.
**Outputs:** Masked / redacted file view OR raw escalation OR tombstone metadata view.
**Boundary:** Per File-Backed-Result-Masked-Default Rule, masked default. Per PR-D Workflow 12, file-backed governance.

### PR-E Workflow 8 - Legal Hold Search Behavior Workflow

**Purpose.** Evaluates legal hold scope for each result; legal hold flags shown only to authorized audiences per Legal-Hold-Flag-Visibility-Scoped Rule; access proceeds normally per PR-D (legal hold blocks purge, NOT access).

**Steps (architectural):**

1. For each result in the result set, query active Legal Hold Records (via PR-D Workflow 7 scope-match logic) to determine `legal_hold_state_at_access`.
2. If requester is authorized audience (Compliance / Audit Reviewer, CIXCI System Admin, etc.) per PR-D Access Policy Matrix, expose legal hold indicator on result rendering.
3. If requester is unauthorized audience, do NOT expose legal hold indicator (per Legal-Hold-Flag-Visibility-Scoped Rule); the result is still accessible per PR-D access governance.
4. Hardened Audit Access Record records `legal_hold_state_at_access` regardless of indicator visibility.

**Inputs:** Result set; requester audience.
**Outputs:** Result set with audience-scoped legal hold indicator visibility.
**Boundary:** Per PR-D Legal-Hold-Does-Not-Mutate-Evidence Rule, no evidence mutation. Per Legal-Hold-Flag-Visibility-Scoped Rule, audience-scoped indicator visibility.

### PR-E Workflow 9 - Retention-Aware Search Result Handling Workflow

**Purpose.** Evaluates Retention Disposition Record state for each result; renders per `retention_disposition_state` (per Purged-Reference-Only-Metadata-View Rule, Archived-Result-Availability-State Rule); preserves PR-D File-Metadata-Outlives-Payload Rule discipline.

**Steps (architectural):**

1. For each result, query latest Retention Disposition Record state.
2. For `retain` or `preserved`: render normally.
3. For `archive`: render with availability-state indicator; retrieval-latency-expected.
4. For `purge_eligible` and `purge_blocked_by_hold`: render normally (payload still available; purge not yet executed OR blocked).
5. For `purged_reference_only`: render metadata / hash / tombstone view only; payload unavailable; file metadata + `file_hash_reference` preserved.
6. Hardened Audit Access Record records `retention_disposition_state` context (via existing PR-D fields where applicable).

**Inputs:** Result set; Retention Disposition Record references.
**Outputs:** Retention-aware rendered result set.
**Boundary:** Per Purged-Reference-Only-Metadata-View Rule and Archived-Result-Availability-State Rule. Per PR-D File-Metadata-Outlives-Payload Rule.

### PR-E Workflow 13 - Audit Report / Evidence Export Recording Workflow

**Purpose.** Creates Audit Report Export Record entity covering all 4 export_status values (generated / failed / canceled / metadata_only); generates export file via PR-B File Tracking Record ONLY when `export_status = generated`; applies PR-D Redaction Transformation Record per export audience; emits `audit.evidence-export.recorded` event with `export_status` discriminator.

**Steps (architectural):**

1. Receive export request with purpose reference, scope reference (Evidence Collection Record / Evidence Search Session / explicit Evidence Record list), and target audience.
2. Verify `export_purpose_reference` (REQUIRED).
3. Verify `actor_reference` OR `service_trigger_reference` has authority for export of the scoped evidence per PR-D Workflow 13 -> Tenant Company `check_access`.
4. For each evidence item in scope:
   a. Evaluate Legal Hold scope (per PR-D Workflow 7); evidence under hold MAY be exported by authorized actors (per OQ-EX-5 PR-E locked decision); export records hold state at export time.
   b. Apply PR-D Redaction Transformation Record per `export_redaction_audience`.
   c. For raw export items, require per-item PR-D Workflow 9 escalation with `access_reason_reference`.
5. Determine `export_status`:
   - `generated`: create PR-B File Tracking Record with `file_purpose = audit_export` (existing PR-B value); populate `export_file_tracking_record_reference` on Audit Report Export Record; generated file is the Evidence Export Package; subject to PR-D retention / redaction / legal hold governance.
   - `failed`: populate `export_failure_reason`; `export_file_tracking_record_reference` NULL.
   - `canceled`: populate `export_cancel_reason`; `export_file_tracking_record_reference` NULL.
   - `metadata_only`: NO PR-B File Tracking Record created; `export_file_tracking_record_reference` NULL; metadata-only preview / inventory listing.
6. Create Audit Report Export Record (append-only) with `export_status`.
7. Emit `audit.evidence-export.recorded` event carrying `export_status` and `export_redaction_audience` discriminators.
8. Downstream download / access of the generated artifact (when `export_status = generated`) is logged via PR-D Workflow 8 -> PR-D hardened Audit Access Record (per Export-Access-Logged-Via-PR-D Rule).

**Inputs:** Export request; scope reference; target audience; purpose reference.
**Outputs:** Audit Report Export Record entity; (when `export_status = generated`) PR-B File Tracking Record; `audit.evidence-export.recorded` event.
**Boundary:** Per Export-Default-Redacted Rule, redacted by default per audience. Per Export-File-Tracking-Only-When-Artifact-Exists Rule, File Tracking Record only when generated. Per Export-Access-Logged-Via-PR-D Rule, downloads logged via PR-D. Per Audit-Export-Not-Analytics Rule, NOT BI dashboard.
