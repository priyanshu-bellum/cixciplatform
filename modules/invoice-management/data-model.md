This document is proposal-level architecture. It defines Invoice Management entities without finalizing storage, accounting sync implementation, payment behavior, reconciliation ownership, tax handling, or ownership outside ADR-0011.

## Source Evidence Control Reference

A Source Evidence Control Reference is a proposal-level pattern for non-Pricing source references used by Invoice Eligibility Evidence, Invoice Lines, exports, reconciliation, and accounting handoff review.

Proposal-level fields/concepts:

- Source reference id.
- Source module.
- Source entity/reference.
- Source record version/hash.
- Source timestamp.
- Source freshness timestamp.
- Source expiration timestamp.
- Source disposition.
- Applied vs ignored state.
- Stale/missing/conflicting state.
- Supersession reference.
- Review-required state.
- Audit reference.

Apply this pattern to:

- Fulfillment/Returns shipment line evidence.
- Fulfillment/Returns delivered quantity evidence.
- Fulfillment/Returns return line disposition evidence.
- Fulfillment/Returns vendor-provided refund/adjustment evidence reference.
- Procurement PO reference.
- Procurement PO line reference.
- Procurement accepted-price evidence reference.
- Procurement received/completed evidence where applicable.
- Order Routing routed suborder reference.
- Order Routing routed suborder line reference.
- Order Routing routing snapshot reference.
- Product Catalog product reference.
- Product Catalog buyer-product relationship reference.
- Tenant Company access/scope evidence reference.

Bare source references are not enough for invoice immutability. Invoice Management must preserve enough evidence to prove which source state made an invoice line eligible. Missing, stale, superseded, ignored, expired, or conflicting source evidence blocks invoice generation or routes the line, report, export, or accounting sync request to review. Invoice Management validates source evidence bindability and disposition; it does not mutate upstream records.

## Ownership Summary

Invoice Management owns invoice records, invoice lines, invoice eligibility evaluation, preview, finalization, regeneration, supersession, invoice reports/exports, assigned invoice adjustment lifecycle, reconciliation detection/review, accounting sync request state and duplicate-posting safeguards for invoice handoff requests, invoice views, invoice status, and invoice events.

Invoice Management does not own Pricing calculation, fulfillment delivery truth, return operational disposition, refund execution, payment, accounting ledger posting, Procurement accepted-price truth, routing decisions, notification delivery, integration transport, Logs & Audit evidence, Analytics metrics, or AI recommendations.

## Retention Notes

Retention remains proposal-level for invoice records, invoice lines, eligibility evidence, source evidence control references, export batches, reconciliation uploads, accounting sync requests, access events, redaction decisions, and supersession history.

## Invoice Management Foundation Data Model

This section documents the Invoice Management data-model hardening. All existing Invoice Management baseline content is preserved by reference without modification. No adjacent module data-model entry is modified.

### Counterparty-scope discipline (invariant)

Every Invoice, Invoice Line, Invoice Adjustment, Invoice Exception Record (counterparty-scoped variants), Vendor Reconciliation Upload Job, Vendor Reconciliation Match Result, and all counterparty-scoped sub-structures carry the relevant scope keys:

- `buyer_reference` and/or `vendor_reference` (depending on `counterparty_role`).
- `company_scope_reference` (REQUIRED for counterparty-scoped records; supports parent / child scope per Tenant Company baseline).
- `billing_profile_reference` (where applicable; supports per-profile grouping).

Cross-counterparty reads / mutations are architecturally guarded by these scope keys. There is no cross-counterparty key.

### Snapshot immutability rules (locked)

- Invoice Line snapshots are IMMUTABLE once their parent Invoice is in any post-`draft` status.
- Upstream changes to source order / fulfillment / refund / pricing AFTER Invoice Line generation MUST NOT mutate the Invoice Line.
- Upstream changes produce Invoice Adjustment records in subsequent (or same) Invoice Runs.
- Re-runs use the same idempotency key (`invoice_period` + counterparty + `source_order_line_reference`) to avoid duplicate Invoice Line generation; on conflict, the existing Invoice Line is preserved and the new attempt produces an Invoice Adjustment or supersession.

### QuickBooks-derived state is REFERENCE-ONLY

- `quickbooks_payment_status_reference` and all other `quickbooks_*_reference` fields are EXTERNAL references.
- CIXCI does NOT own QuickBooks ledger truth, payment truth, or tax truth.
- Pulled-back QuickBooks state appears as `external_status_summary_reference` on Invoice, NOT as an Invoice Management-owned status.

### Cross-counterparty non-interference (architectural guarantee)

- Every Invoice, Invoice Line, Invoice Adjustment, Invoice Exception Record (counterparty-scoped), Vendor Reconciliation Upload Job, and Vendor Reconciliation Match Result carries the counterparty scope keys.
- There is no cross-counterparty key.
- Buyer 1's invoices cannot be read or mutated under Buyer 2's scope.
- Vendor 1's reconciliation uploads cannot affect Vendor 2's invoices.

### Vendor reconciliation file rule (data-model level)

- Vendor Reconciliation Upload Rows are READ-ONLY rows containing vendor-reported values.
- Upload Rows MUST NEVER mutate Order Routing, Fulfillment / Returns, Pricing, Product Catalog, Tenant Company, Logs & Audit, Integration Management, or QuickBooks records.
- Vendor-reported values appear in Match Results with explicit `vendor_reported_value_evidence` and `cixci_source_value_evidence` to make the comparison-evidence-not-source-truth boundary explicit at the data-model level.

### What this data-model section intentionally does NOT introduce

- Separate Buyer Invoice / Vendor Invoice / CIXCI Commission Invoice entity families.
- Invoice Generation Job / Item entities (workflow behavior only; observable via Invoice Run status).
- Invoice Reconciliation Record as a distinct entity (subsumed by Vendor Reconciliation Match Result + Exception).
- Settlement / Payout Reference entity.
- Payment Status Reference as internal truth.
- Tax Reference as CIXCI-owned truth.
- Invoice Approval Record as a separate entity (status history + workflow only).
- Automatic vendor payment execution record.
- New Tenant Company capabilities or role bundles.
- New Logs & Audit entities (new evidence kinds emitted via existing `service_identity.evidence_emit`).
- Concrete persistence schema. Implementation owns.
- Concrete query indexing strategy. Implementation owns.
- Concrete propagation latency. Implementation owns.
- Concrete UI / UX field shapes. Future UX / UI.
- Rename, deprecation, or replacement of any existing baseline data-model entry.
- `audit_export.*` capability references (PR #103 family is for compliance audit report exports; NOT used here).
