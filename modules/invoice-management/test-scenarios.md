# Invoice Management Test Scenarios

This document is proposal-level architecture. It lists validation scenarios without finalizing implementation tests.

## Ownership And Boundary

- Invoice generation consumes Pricing invoice-bindable evidence and does not recalculate commission or buyer-facing Wholesale Price.
- Invoice generation consumes delivered shipment line evidence and does not treat broad shipment/tracking status as invoice-eligible.
- Bulk PO invoice evaluation consumes Procurement accepted-price evidence and Pricing PO invoice-bindable evidence without recalculating accepted PO price.
- Reconciliation upload records mismatch review state without mutating Pricing, Fulfillment/Returns, Procurement, payment, accounting, or Integration records.
- Accounting sync request records Invoice request state and duplicate-posting safeguards while Integration Management owns delivery/receipt evidence.

## Source Evidence Versioning And Disposition

- Blocks invoice line when Fulfillment shipment line evidence lacks source record version/hash.
- Blocks invoice line when delivered quantity evidence is stale, superseded, or ignored.
- Blocks invoice line when Procurement accepted-price evidence is missing source disposition.
- Routes to review when routing snapshot version conflicts with routed suborder line version.
- Routes to review when Product Catalog buyer-product relationship evidence is superseded.
- Blocks export or generation when Tenant Company access/scope evidence is expired.
- Emits `invoice.source-evidence.stale`, `invoice.source-evidence.superseded`, or `invoice.source-evidence.ignored` where applicable.
- Emits `invoice.line.blocked.non-bindable-source-evidence` when source disposition is not invoice-bindable.

## Invoice Eligibility Evidence

- Creates `invoice.eligibility-evidence.created` when all required source evidence is fresh and invoice-bindable.
- Blocks invoice line when Pricing snapshot is missing.
- Blocks invoice line when Pricing snapshot is stale or superseded.
- Blocks invoice line when Pricing snapshot is non-invoice-bindable.
- Blocks invoice line when Tenant Company access/scope evidence is missing.
- Routes to review when source evidence conflicts across Pricing, Fulfillment/Returns, Procurement, or Product Catalog references.
- Emits `invoice.line.blocked.missing-evidence` with blocked reason.

## Pricing Snapshot Consumption

- Preserves Pricing snapshot id, version/hash, Pricing Channel, invoice-bindable status, commission component references, buyer-facing Wholesale Price output, override/exception references, and redaction evidence on invoice line evidence.
- Does not calculate commission from percentage fields.
- Does not calculate PO price from requested/accepted values.
- Does not calculate return/refund adjustment price without Pricing adjustment pricing evidence.

## Pricing Channel Split

- Online/DTC invoice line requires Online/DTC Pricing invoice-bindable snapshot.
- Bulk PO invoice line requires Pricing Channel = Bulk Purchase Order and Procurement PO evidence.
- Owned Channel / Kaseory invoice placeholder requires explicit Pricing exception evidence.
- Unsupported Buyer Storefront, Marketplace, or Retail POS channel routes to review until supported by Pricing evidence.

## Online Order Eligibility

- Delivered shipment line evidence creates invoiceable quantity.
- Shipped status without delivered quantity evidence blocks invoice generation.
- Tracking URL without delivered quantity evidence blocks invoice generation.
- Partial delivered quantity creates line-level invoiceable quantity only for delivered quantity.
- Stale, duplicate, or out-of-order shipment line evidence routes to review.

## Return / Adjustment Evidence

- Return line disposition evidence plus Pricing adjustment pricing evidence creates adjustment eligibility.
- Vendor-provided Return Refunded Amount is recorded as adjustment evidence but not accepted as final financial truth by itself.
- Vendor refund evidence variance emits `invoice.vendor-refund-evidence.variance-detected`.
- Invoice adjustment amount is blocked when vendor evidence conflicts with Pricing adjustment evidence or invoice rules.
- Missing return line disposition evidence blocks adjustment.
- Missing adjustment pricing evidence blocks adjustment.
- Partial return line evidence creates line-level adjustment quantity.
- Conflicting vendor refund evidence routes to review.

## Bulk PO Invoice Alignment

- PO invoice eligibility requires Procurement PO/line reference and Pricing PO invoice-bindable evidence.
- Missing Procurement accepted-price evidence blocks or routes to review where required.
- Pricing PO invoice-bindability blocked emits event.
- Accepted-price variance is referenced but not interpreted by Invoice Management.

## Export Governance

- Invoice export creates export batch with schema version, generated by/at, date-basis, source evidence basis, Tenant Company scope/version, redaction class/version, and Logs & Audit evidence reference.
- Export with missing redaction decision is blocked.
- Export with stale access scope requires recheck before download.
- Expired export emits `invoice.export.expired`.
- Revoked export emits `invoice.export.revoked`.
- Sensitive invoice access emits `invoice.sensitive.accessed`.

## Reconciliation

- Reconciliation upload follows preview/review before mismatch disposition.
- Invalid upload schema is rejected.
- Mismatch detection does not mutate source records.
- Reviewed upload emits `invoice.reconciliation-upload.reviewed`.

## Accounting Sync

- Accounting sync requested emits event with Integration delivery reference placeholder.
- Accounting sync request includes invoice version/hash, idempotency key, duplicate blocker, and provider request fingerprint.
- Reused idempotency key emits `invoice.accounting-sync.idempotency-key-reused` and blocks duplicate posting unless explicitly reviewed.
- Duplicate external reference emits `invoice.accounting-sync.duplicate-blocked`.
- Superseded invoice requires accounting sync supersession/correction reference before a new posting request.
- Accounting sync failure records Integration failure reference and Invoice review state.
- Invoice Management does not write external accounting ledger state.

## Tenant Visibility

- Buyer view hides vendor-only commission fields when redaction requires it.
- Vendor view hides unauthorized customer/accounting fields when redaction requires it.
- System admin view requires system admin scope and sensitive access event where applicable.
- Recheck-before-display blocks stale access evidence.
- Recheck-before-download blocks stale export access evidence.

## Invoice Management Foundation Test Scenarios

This section adds architecture-level acceptance scenarios for the Invoice Management foundation hardening. All existing Invoice Management baseline scenarios are preserved without modification. Scenarios assert observable state transitions, evidence emission, boundary enforcement, and consumer outcomes; concrete request / response shapes, error codes, and timings are out of scope.

---

### Invoice Run lifecycle

- CIXCI System Admin creates an Invoice Run for a billing period; Invoice Run at `invoice_run_status = requested`; event `invoice-run.requested` emitted; event `invoice-run.status-changed` with `invoice_run_status = requested` emitted; Evidence Record emitted with evidence kind `invoice_run`.
- Invoice Run transitions through `collecting_sources` -> `validating_reconciliation` (when reconciliation prerequisites apply) -> `generating` -> `generated` (no blocking exceptions) OR `completed_with_exceptions` (with exceptions).
- On `completed_with_exceptions`: Invoice Run transitions to `pending_review`; admin reviews exceptions per Workflow 8.
- On approval: Invoice Run transitions to `approved`.
- On QuickBooks handoff: transitions through `queued_for_sync` -> `sync_requested` -> `synced` (success) OR `sync_failed`.
- On supersession: prior Invoice Run transitions to `superseded`; new Invoice Run becomes current.
- On archival: transitions to `archived`.

### Invoice lifecycle

- Invoice transitions through `draft` -> `generated` -> `pending_review` -> `approved` -> `queued_for_sync` -> `sync_requested` -> `synced` (success) OR `sync_failed`.
- Invoice MAY transition to `adjustment_required` when a late refund or correction is recorded.
- Invoice MAY transition to `superseded` when a re-run produces a new Invoice for the same idempotency key in supersession path.
- Invoice MAY transition to `rejected`, `voided`, `canceled`, `archived` per admin action.
- Every status transition records a new entry in Invoice Status History.
- Every status transition emits `invoice.status-changed` with `invoice_status` discriminator.

### Delivered eligibility

- An order line with shipment / tracking status but NO Fulfillment / Returns delivery evidence: NOT invoice-eligible. **Shipment / tracking alone is NOT enough.**
- An order line with Fulfillment / Returns delivery evidence: invoice-eligible (subject to other rules).
- Vendor-reported delivered status WITHOUT CIXCI source delivery evidence: NOT automatically invoice-eligible. MAY produce a reconciliation Match Result for review.

### Return refund adjustment eligibility

- A return-refunded order line with source refund evidence AND Pricing adjustment evidence: adjustment-eligible; Invoice Adjustment with `adjustment_kind = refund` produced.
- A return-refunded order line WITHOUT source refund evidence: NOT automatically adjustment-eligible. MAY produce Invoice Exception Record at `exception_kind = missing_refund_evidence`.
- Vendor-reported refund WITHOUT CIXCI source refund evidence: MAY create adjustment CANDIDATE / Invoice Exception Record at `exception_kind = vendor_report_mismatch`; does NOT automatically reduce invoiceable amount.
- Return Rejected: reportable in vendor statement / payable package, but MUST NOT reduce invoiceable amount by default.

### Excluded order states

- Canceled, failed, pending, unfulfilled, undelivered, returned-not-refunded, rejected-without-refund orders: NOT included in invoice generation.

### Split / multi-vendor orders

- Order with line A from Vendor 1 and line B from Vendor 2: invoiced at line / suborder / vendor scope; Vendor 1's invoice contains line A only; Vendor 2's invoice contains line B only.

### Late refund handling

- Closed invoice period: invoice approved and synced to QuickBooks.
- Subsequent refund event after period close: creates later-period Invoice Adjustment with `adjustment_kind = late_refund`; `parent_invoice_line_reference` points to the original Invoice Line.
- **Original Invoice Line is NOT mutated.**

### Snapshot immutability

- Invoice Line generated at time T with source order line at version V1.
- Source order line subsequently corrected at version V2.
- The Invoice Line continues to reference V1 (via Invoice Line Source Reference snapshot); the Invoice Line is NOT mutated.
- The correction produces an Invoice Adjustment (Workflow 6 or Workflow 14) or Invoice Exception Record (`exception_kind = source_record_drift`).

### Idempotency

- Invoice Run re-run for the same period + counterparty + `source_order_line_reference`: produces NO duplicate Invoice Line for that key.
- On idempotency collision: Invoice Exception Record at `exception_kind = idempotency_collision` is recorded; route to Workflow 14 for supersession / adjustment.

### Re-run does not duplicate QuickBooks postings

- Prior Invoice Run synced to QuickBooks; admin re-runs for same period; re-run produces Invoice Adjustments for deltas (NOT duplicate Invoices); QuickBooks receives a bill / invoice adjustment (per `quickbooks_handoff_object_kind`), NOT a duplicate of the prior posting.

### Pricing / Commission snapshot binding

- Every Invoice Line carries `pricing_snapshot_reference` (REQUIRED) and `commission_snapshot_reference` (REQUIRED).
- Invoice Line snapshot values (`unit_price`, `quantity`, `vendor_wholesale_amount`, `buyer_wholesale_amount`, `buyer_side_commission_amount`, `vendor_side_commission_amount`, `invoiceable_amount`, `refund_adjustment_amount`, `net_amount`) populated from Pricing snapshot AS EVIDENCE.
- **Invoice Management does NOT recalculate Pricing-owned values.**
- Missing snapshot: Invoice Exception Record at `exception_kind = missing_pricing_snapshot` or `missing_commission_snapshot`.

### Buyer / Vendor Invoice separation

- Buyer monthly invoice: `invoice_type = buyer_invoice`, `counterparty_role = buyer`.
- Vendor monthly statement: `invoice_type = vendor_statement`, `counterparty_role = vendor`.
- Vendor payable package: `invoice_type = vendor_payable_package`, `counterparty_role = vendor`.
- CIXCI internal commission report: `invoice_type = commission_report`, `counterparty_role = cixci_internal`.
- Adjustment report: `invoice_type = adjustment_report`.
- Internal admin invoice report: `invoice_type = internal_admin_invoice_report`.
- **No separate entity family for buyer vs vendor vs commission invoices.**

### Vendor Reconciliation Upload (Workflow 16)

- Vendor uploads month-end file: Vendor Reconciliation Upload Job at `reconciliation_upload_status = uploaded`; event `vendor-reconciliation-upload.received` emitted; File Tracking Record reference via Logs & Audit PR-B.
- Validation runs: transitions to `validating` -> `validated` (success) OR `failed` (validation failure).
- File with duplicate rows: `failed` OR `review_required` per business policy; per-row Match Result at `match_result_status = duplicate_in_vendor_file`.
- File with missing required headers: `failed`.
- File with invalid order references: rows tagged; Match Results at `missing_in_cixci`.
- **Upload rows NEVER mutate Order Routing, Fulfillment / Returns, Pricing, Product Catalog, Tenant Company, Logs & Audit, Integration Management, or QuickBooks records.**

### Vendor Reconciliation Matching (Workflow 17)

- Matching runs against CIXCI source records: per-row Match Result with `match_result_status` discriminator.
- `matched`: no Invoice Exception Record.
- `missing_in_vendor_file`: CIXCI invoice-eligible row not present in vendor file; Invoice Exception Record at `exception_kind = vendor_report_mismatch`.
- `missing_in_cixci`: vendor file row not present in CIXCI; Invoice Exception Record.
- `duplicate_in_vendor_file`: vendor file has duplicate row; Invoice Exception Record.
- `status_mismatch` (e.g., vendor reports delivered, CIXCI shows undelivered): Invoice Exception Record.
- `amount_mismatch`, `quantity_mismatch`, `sku_upc_mismatch`, `tracking_mismatch`, `date_mismatch`, `refund_mismatch`, `rejected_reason_mismatch`: per-dimension Invoice Exception Records.
- `review_required`: indeterminate; routed to admin review.
- On matching completion: Vendor Reconciliation Upload Job transitions to `reconciled` (no exceptions) OR `reconciled_with_exceptions` OR `review_required`.
- **Vendor-reported values do NOT mutate CIXCI source records.**

### Vendor file as comparison evidence, NOT source truth

- Vendor reports a delivered date; CIXCI source shows no delivery evidence: Match Result `status_mismatch` OR `missing_in_cixci`; Invoice Exception Record; invoice eligibility NOT automatically created.
- Vendor reports a refund amount; CIXCI source shows no refund evidence: Match Result `refund_mismatch` OR `missing_in_cixci`; adjustment NOT automatically created.
- **Vendor file is comparison evidence only.**

### Vendor Payable Package Approval-Ready (Workflow 18)

- Vendor reconciliation `reconciled` AND all in-scope vendor Invoices `approved` AND no blocking Invoice Exception Records: vendor payable package marked approved-ready (data-level signal).
- If vendor reconciliation at `reconciled_with_exceptions` with exceptions still unresolved: payable package NOT marked approved-ready; held for review.

### Vendor Payment QuickBooks Handoff (Workflow 19)

- Approved-ready vendor payable package: QuickBooks Handoff Reference created with `quickbooks_handoff_object_kind = vendor_payable_package` (or `bill`); vendor Invoice transitions through `queued_for_sync` -> `sync_requested` -> `synced` or `sync_failed`.
- On success: `quickbooks_bill_reference`, `quickbooks_sync_reference`, `quickbooks_vendor_reference`, `quickbooks_last_synced_at` populated; event `quickbooks-handoff.reference-recorded` emitted.
- On failure: `quickbooks_error_reference` populated; Invoice transitions to `sync_failed`; Invoice Exception Record produced.
- **CIXCI does NOT auto-submit vendor payment.** Billing person submits payment from QuickBooks.
- QuickBooks-derived payment status (`paid`, `partially_paid`, `credited`, etc.) pulled back as `quickbooks_payment_status_reference` only.

### QuickBooks transport boundary

- Integration Management owns QuickBooks API transport.
- Invoice Management calls Integration Management to dispatch; Integration Management populates `integration_management_dispatch_reference`.
- Provider response, retry, external delivery evidence: owned by Integration Management.
- Invoice Management does NOT call QuickBooks API directly.

### QuickBooks payment status REFERENCE only

- QuickBooks reports `paid`: Invoice Management stores as `quickbooks_payment_status_reference`.
- Invoice's internal `invoice_status` REMAINS in the 14-value internal taxonomy (`synced`, etc.).
- External payment statuses (`sent`, `paid`, `partially_paid`, `credited`) are NOT in the internal `invoice_status` taxonomy.
- Invoice Management does NOT treat QuickBooks payment status as ledger truth.

### Invoice Adjustment vs Invoice Line mutation

- Source record change after Invoice Line generated: Invoice Adjustment created; Invoice Line NEVER mutated.
- Multiple adjustments accumulate as separate Invoice Adjustment records, each linked to the original Invoice Line.

### Invoice Exception Review (Workflow 8)

- Invoice Exception Record produced (any `exception_kind`): admin views via data-level signal (UI surface future).
- Admin acknowledges: `acknowledged_flag = true`, `acknowledged_timestamp`, `acknowledged_actor_reference` recorded.
- Admin resolves: `resolution_reference` recorded; MAY trigger Invoice Adjustment (Workflow 6) or supersession (Workflow 14).
- Blocking exceptions HOLD invoice approval per Workflow 9.

### Invoice Approval (Workflow 9)

- All blocking exceptions resolved: admin approves; Invoice transitions to `approved`.
- Missing QuickBooks customer / vendor mapping: blocking exception; approval HELD.
- Missing commission / pricing snapshot: blocking exception; approval HELD.
- Vendor reconciliation upload at `failed` (where required by business policy): approval HELD.

### Invoice Report Export (Workflow 15)

- Admin requests export of Invoice Report: Invoice Export / File Reference populated with `file_artifact_reference` (Logs & Audit File Tracking Record reference per PR-B).
- Event `invoice-report-export.recorded` emitted.

### Evidence emission (Workflow 13)

- Every Invoice Run, Invoice, Invoice Line, Invoice Adjustment, Invoice Exception Record, Vendor Reconciliation Upload Job, Vendor Reconciliation Match Result, QuickBooks Handoff Reference: emits Evidence Record with appropriate evidence kind (`invoice_run`, `invoice`, `invoice_line`, `invoice_adjustment`, `invoice_exception`, `vendor_reconciliation_upload`, `vendor_reconciliation_match`, `quickbooks_handoff`).
- All Evidence Records carry `correlation_reference`.
- Logs & Audit indexes per PR-A; retention applies per PR-D.

### Cross-counterparty non-interference

- Vendor 1's reconciliation upload does NOT affect Vendor 2's invoices.
- Buyer 1's invoice cannot be read or mutated under Buyer 2's scope.
- Cross-counterparty keys do not exist.

### `audit_export.*` non-use

- Invoice Run creation: existing CIXCI System Admin authority consulted; `audit_export.*` NOT consulted, NOT consumed.
- Vendor reconciliation upload: existing vendor + Tenant Company act-on-behalf authority consulted; `audit_export.*` NOT consulted.
- Invoice Report export: existing buyer / vendor / admin capability consulted; `audit_export.*` NOT consulted.

### Lifecycle blocking

- Suspended actor: cannot create Invoice Run, approve invoices, upload reconciliation, or request QuickBooks handoff.
- Pending Setup actor: cannot initiate invoice-affecting actions.
- Inactive target company in admin-on-behalf scenarios: blocked per existing PR #103 baseline.
- Active actor + active target: normal evaluation.

### Discriminator-based subscriber routing

- Subscriber to `invoice.status-changed` with `invoice_status = approved`: receives approval events.
- Subscriber to `vendor-reconciliation-match.completed` with `match_result_status = amount_mismatch`: receives only amount-mismatch events.
- Subscriber to `quickbooks-handoff.reference-recorded` with `quickbooks_handoff_object_kind = payment_status_update`: receives only payment status update events.
- Subscribers MUST handle unknown discriminator values gracefully (forward-compatibility).

### Existing PR #98-#105 baselines preserved

- Logs & Audit Evidence Record, File Tracking Record, retention (PR-D), audit access (PR-E): preserved by reference; no Logs & Audit file modified.
- Tenant Company `check_access`, capability registry, audit capability families (PR #103): preserved by reference; no Tenant Company file modified; `audit_export.*` NOT used.
- PR #104 Buyer Product Export Job / Selection Snapshot / dispatch reference: preserved by reference.
- PR #105 Buyer-Scoped Compatibility Projection / portfolio snapshot / impact records: preserved by reference; NOT consumed by Invoice Management.

- Concrete UI surfaces.

- Concrete idempotency cache shape, TTL.

- AI-Agent-initiated invoice / reconciliation scenarios (future PR).
