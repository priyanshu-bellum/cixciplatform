# Invoice Management Event Contracts

This document is proposal-level architecture. It describes event payload expectations without finalizing serialization, schema registry, or broker behavior.

## Shared Event Envelope

Invoice events should include:

- Event id.
- Event name.
- Event version.
- Occurred at timestamp.
- Actor/service reference.
- Tenant Company scope/version.
- Buyer/vendor/entity references where applicable.
- Invoice reference where applicable.
- Invoice line reference where applicable.
- Invoice period/date-basis where applicable.
- Pricing Channel where applicable.
- Source evidence references.
- Source evidence version/disposition references where applicable.
- Redaction decision version where applicable.
- Audit reference placeholder.

## Source Evidence Control Payload

Source evidence events should include:

- Source reference id.
- Source module.
- Source entity/reference.
- Source record version/hash.
- Source timestamp.
- Source freshness timestamp.
- Source expiration timestamp.
- Source disposition.
- Applied vs ignored state.
- Stale/missing/conflicting state.
- Supersession reference.
- Review-required state.
- Audit reference.

## Eligibility Payload

`invoice.eligibility-evidence.created` and `invoice.eligibility.blocked` should include:

- Invoice eligibility evidence reference.
- Pricing invoice-bindable snapshot evidence reference.
- Pricing snapshot id/version/hash.
- Vendor-side and buyer-side commission component references.
- Pricing rule/version/hash.
- Order Routing routed suborder/line Source Evidence Controls.
- Fulfillment shipment line Source Evidence Control.
- Delivered quantity Source Evidence Control.
- Return line disposition Source Evidence Control where applicable.
- Procurement PO/line and accepted-price Source Evidence Controls where applicable.
- Tenant Company access/scope Source Evidence Control.
- Product Catalog product/buyer relationship Source Evidence Control where applicable.
- Eligibility status.
- Blocked reason.
- Stale/missing/conflicting/ignored/superseded evidence state.
- Review-required state.

## Pricing Evidence Payload

Pricing-related invoice events should include references to Pricing-owned evidence rather than formula values:

- Pricing snapshot id.
- Snapshot version/hash.
- Pricing Channel.
- Invoice-bindable status.
- Vendor-side commission component reference.
- Buyer-side commission component reference.
- Buyer-facing Wholesale Price output reference.
- Buyer pricing mode/exception reference.
- Buyer-specific override reference.
- Owned-channel exception reference.
- Adjustment pricing evidence reference.
- PO invoice-bindable evidence reference.
- Pricing visibility/redaction evidence reference.

## Shipment And Return Evidence Payload

Fulfillment/return-related invoice events should include:

- Fulfillment/Returns shipment line evidence Source Evidence Control.
- Delivered quantity Source Evidence Control.
- Fulfillment/Returns return line disposition Source Evidence Control.
- Vendor-provided refund/adjustment evidence Source Evidence Control.
- Vendor evidence variance amount where detected.
- Return/refund evidence stale or missing state.
- Adjustment pricing evidence reference from Pricing where applicable.
- Adjustment amount source classification.
- Adjustment amount applied flag.
- Adjustment amount blocked reason.

Payloads must not imply that Invoice Management owns delivery truth, return operational disposition, or vendor refund financial truth.

## PO Evidence Payload

PO-related invoice events should include:

- Procurement PO Source Evidence Control.
- Procurement PO line Source Evidence Control.
- Procurement accepted-price Source Evidence Control.
- Procurement submitted/accepted/received/completed Source Evidence Controls where applicable.
- Pricing PO invoice-bindable evidence reference.
- PO invoice eligibility status.
- Blocked/review reason.

Payloads must not imply Invoice Management recalculates accepted PO price or owns Procurement accepted-price truth.

## Export Payload

Invoice export events should include:

- Invoice export batch/reference.
- Invoice export schema version.
- Generated by actor/service.
- Generated at timestamp.
- Invoice period/date-basis.
- Source evidence basis.
- Buyer/vendor/system admin view type.
- Tenant Company access scope/version.
- Redaction class/version.
- File/download reference placeholder.
- Expiration/revocation state.
- Supersession reference.
- Logs & Audit evidence reference.
- Integration delivery reference where exported externally.
- Notification delivery reference where scheduled/emailed.
- Sensitive access/export event reference where applicable.

## Reconciliation Payload

Reconciliation events should include:

- Reconciliation upload job reference.
- Upload schema version.
- Source file reference placeholder.
- Validation status.
- Preview/review state.
- Mismatch detection result.
- Source invoice/line references.
- Correction/supersession reference.
- Logs & Audit evidence reference.

## Accounting Sync Payload

Accounting sync events should include:

- Accounting sync request reference.
- Invoice reference.
- Invoice version/hash.
- Invoice export/reference.
- Accounting target/system reference.
- Idempotency key.
- Duplicate external reference blocker.
- Provider request fingerprint.
- External posting reference placeholder.
- Supersession/correction reference.
- Retry attempt reference.
- Duplicate-posting risk flag.
- Applied vs ignored state.
- Integration delivery/receipt evidence reference.
- Sync requested/accepted/failed state.
- Failure/retry/disposition reference.
- Review-required state.

Integration Management owns external delivery/receipt evidence and provider responses.

## Invoice Management Foundation Event Contracts

This section documents the payload contracts for the 11 new Invoice Management events. All payload extensions are reference-first per existing PR-A discipline. No concrete schema is locked; concrete payload schema is future API Governance Foundation PR work.

### Contract discipline

- All 11 new event payload contracts are reference-first per existing Invoice Management + PR-A discipline.
- No concrete schema is locked.
- Discriminator value catalogs documented per event below.
- Subscribers MUST handle unknown discriminator values gracefully (forward-compatibility).
- All existing payload envelope fields (`correlation_reference`, `trace_reference`, `idempotency_key`, `audit_record_reference`, schema version) preserved per existing baseline.

---

### Event 1 - `invoice-management.invoice-run.requested`

**Purpose:** indicate that an Invoice Run has been created.

**Payload context (reference-first):**

- `invoice_run_id`.
- `invoice_period_reference`.
- `requested_by_actor_reference`.
- `counterparty_scope_summary` (high-level summary; full scope retrievable via `invoice_run_id`).
- `requested_timestamp`.
- Existing baseline envelope: `correlation_reference`, `trace_reference`, `audit_record_reference`.

### Event 2 - `invoice-management.invoice-run.status-changed`

**Purpose:** indicate Invoice Run status transition.

**Discriminator:** `invoice_run_status` (string-enum; 15 values).

**Discriminator value catalog (15 values):**

| Value | Meaning |
|---|---|
| `requested` | Run created. |
| `collecting_sources` | Gathering eligible source records. |
| `validating_reconciliation` | Validating vendor reconciliation upload prerequisites. |
| `generating` | Generating Invoices / Lines / Adjustments. |
| `generated` | Run completed generation without blocking exceptions. |
| `completed_with_exceptions` | Run completed generation but has exceptions / holds / review items. |
| `pending_review` | Admin review state after generated output or exceptions. |
| `approved` | Admin approved the run output. |
| `queued_for_sync` | Queued for QuickBooks handoff. |
| `sync_requested` | QuickBooks handoff request issued to Integration Management. |
| `synced` | Integration Management reports successful QuickBooks sync. |
| `sync_failed` | QuickBooks / handoff sync failure (NOT source collection failure). |
| `canceled` | Run canceled. |
| `superseded` | Replaced by a newer Invoice Run. |
| `archived` | Archived for long-term retention. |

**Payload context (reference-first):**

- `invoice_run_id`.
- `prior_invoice_run_status`, `new_invoice_run_status` (the discriminator value).
- `transition_timestamp`.
- `reason_reference`.
- `triggering_actor_reference` OR `triggering_service_trigger_reference`.
- `triggering_invoice_exception_record_reference` (nullable).
- `triggering_vendor_reconciliation_upload_job_reference` (nullable).
- Existing baseline envelope.

### Event 3 - `invoice-management.invoice.generated`

**Purpose:** indicate an Invoice has been generated.

**Payload context (reference-first):**

- `invoice_id`.
- `invoice_run_reference`, `invoice_report_reference`.
- `invoice_type` (one of: `buyer_invoice`, `vendor_statement`, `vendor_payable_package`, `commission_report`, `adjustment_report`, `internal_admin_invoice_report`).
- `counterparty_role` (one of: `buyer`, `vendor`, `cixci_internal`).
- `buyer_reference` and/or `vendor_reference`.
- `company_scope_reference`, `billing_profile_reference`.
- `invoice_period_reference`.
- `invoice_total_snapshot_amount` (evidence value; not Pricing recalculation).
- `generation_timestamp`.
- Existing baseline envelope.

### Event 4 - `invoice-management.invoice.status-changed`

**Purpose:** indicate Invoice status transition.

**Discriminator:** `invoice_status` (string-enum; 14 internal values).

**Discriminator value catalog (14 values):**

`draft`, `generated`, `pending_review`, `approved`, `rejected`, `queued_for_sync`, `sync_requested`, `synced`, `sync_failed`, `adjustment_required`, `superseded`, `voided`, `canceled`, `archived`.

**Payload context (reference-first):**

- `invoice_id`.
- `prior_invoice_status`, `new_invoice_status`.
- `transition_timestamp`.
- `reason_reference`.
- `triggering_actor_reference` OR `triggering_service_trigger_reference`.
- `triggering_quickbooks_handoff_reference` (nullable; populated for sync-related transitions).
- `triggering_invoice_adjustment_reference` (nullable; populated for adjustment-driven transitions).
- Existing baseline envelope.

### Event 5 - `invoice-management.invoice-adjustment.recorded`

**Purpose:** indicate Invoice Adjustment creation.

**Discriminator:** `adjustment_kind` (string-enum).

**Discriminator value catalog (5 values):**

| Value | Meaning |
|---|---|
| `refund` | Same-period refund adjustment. |
| `correction` | Same-period correction. |
| `late_refund` | Later-period adjustment for refund occurring after a closed period. |
| `vendor_reconciliation_exception_resolution` | Adjustment driven by approved reconciliation exception resolution. |
| `manual_admin_adjustment` | CIXCI System Admin manual adjustment. |

**Payload context (reference-first):**

- `invoice_adjustment_id`.
- Parent `invoice_line_reference`, parent `invoice_reference`.
- `adjusting_invoice_run_reference`.
- `adjustment_amount`.
- `adjustment_reason_reference`.
- `triggering_return_refund_evidence_reference` (nullable).
- `triggering_vendor_reconciliation_match_result_reference` (nullable).
- Counterparty scope.
- Existing baseline envelope.

### Event 6 - `invoice-management.invoice-exception.recorded`

**Purpose:** indicate Invoice Exception Record creation.

**Discriminator:** `exception_kind` (string-enum; non-exhaustive list below).

**Discriminator value catalog (representative):**

`missing_quickbooks_mapping`, `missing_commission_snapshot`, `missing_pricing_snapshot`, `missing_delivery_evidence`, `missing_refund_evidence`, `vendor_report_mismatch`, `parent_company_scope_unresolved`, `idempotency_collision`, `source_record_drift`, `late_refund_candidate`, `external_payment_status_drift`.

**Payload context (reference-first):**

- `invoice_exception_record_id`.
- `invoice_run_reference`.
- `invoice_reference` (nullable), `invoice_line_reference` (nullable), `vendor_reconciliation_match_result_reference` (nullable).
- `exception_severity` (e.g., `blocking`, `review_required`, `advisory`).
- `recording_timestamp`.
- Existing baseline envelope.

### Event 7 - `invoice-management.invoice-report-export.recorded`

**Purpose:** indicate Invoice Report export file generation.

**Payload context (reference-first):**

- `invoice_report_id`.
- `invoice_export_file_reference` (sub-structure reference).
- `file_artifact_reference` (Logs & Audit File Tracking Record reference).
- `exported_by_actor_reference`.
- `export_timestamp`.
- Existing baseline envelope.

### Event 8 - `invoice-management.quickbooks-handoff.reference-recorded`

**Purpose:** indicate QuickBooks Handoff Reference sub-structure populated.

**Discriminator:** `quickbooks_handoff_object_kind` (string-enum; 4 values).

**Discriminator value catalog (4 values):**

| Value | Meaning |
|---|---|
| `invoice` | Buyer invoice handed off to QuickBooks. |
| `bill` | Vendor bill handed off to QuickBooks. |
| `vendor_payable_package` | Vendor payable package handed off. |
| `payment_status_update` | QuickBooks-derived payment status pulled back as reference. |

**Payload context (reference-first):**

- Parent `invoice_reference` OR `invoice_report_reference`.
- `quickbooks_handoff_object_kind` (the discriminator value).
- All `quickbooks_*_reference` fields populated where available.
- `integration_management_dispatch_reference`.
- `handoff_request_timestamp` (for outbound) OR `payment_status_pulled_at` (for `payment_status_update`).
- Existing baseline envelope.

**Boundary note:** Invoice Management owns the handoff REFERENCE; Integration Management owns the transport. QuickBooks-derived state stored as references only.

### Event 9 - `invoice-management.vendor-reconciliation-upload.received`

**Purpose:** indicate Vendor Reconciliation Upload Job created.

**Payload context (reference-first):**

- `vendor_reconciliation_upload_job_id`.
- `vendor_reference`, `company_scope_reference`, `invoice_period_reference`.
- `uploading_actor_reference`.
- `received_timestamp`.
- `file_evidence_reference` (Logs & Audit File Tracking Record reference).
- Existing baseline envelope.

### Event 10 - `invoice-management.vendor-reconciliation-upload.status-changed`

**Purpose:** indicate Vendor Reconciliation Upload Job status transition.

**Discriminator:** `reconciliation_upload_status` (string-enum; 10 values).

**Discriminator value catalog (10 values):**

`uploaded`, `validating`, `validated`, `matching`, `reconciled`, `reconciled_with_exceptions`, `failed`, `review_required`, `superseded`, `canceled`.

**Payload context (reference-first):**

- `vendor_reconciliation_upload_job_id`.
- `prior_reconciliation_upload_status`, `new_reconciliation_upload_status`.
- `transition_timestamp`.
- `reason_reference`.
- `match_result_summary` (counts by `match_result_status`; populated where the transition follows matching).
- `invoice_impact_summary` (counts of affected Invoice Lines / Adjustments / Exceptions).
- Existing baseline envelope.

### Event 11 - `invoice-management.vendor-reconciliation-match.completed`

**Purpose:** indicate Vendor Reconciliation Match Result creation.

**Discriminator:** `match_result_status` (string-enum; 13 values).

**Discriminator value catalog (13 values):**

| Value | Meaning |
|---|---|
| `matched` | Row matched to CIXCI source. |
| `missing_in_vendor_file` | CIXCI invoice-eligible row missing from vendor file. |
| `missing_in_cixci` | Row in vendor file does not exist in CIXCI. |
| `duplicate_in_vendor_file` | Duplicate row in vendor file. |
| `status_mismatch` | Status mismatch (delivered / return / rejected). |
| `amount_mismatch` | Amount mismatch (refund amount, etc.). |
| `quantity_mismatch` | Quantity mismatch. |
| `sku_upc_mismatch` | SKU or UPC mismatch. |
| `tracking_mismatch` | Tracking number mismatch. |
| `date_mismatch` | Delivered date mismatch. |
| `refund_mismatch` | Refund amount or status mismatch. |
| `rejected_reason_mismatch` | Rejected return reason mismatch. |
| `review_required` | Indeterminate; routed to admin review. |

**Payload context (reference-first):**

- `vendor_reconciliation_match_result_id`.
- `vendor_reconciliation_upload_job_reference`, `vendor_reconciliation_upload_row_reference`.
- Matched-side references where applicable (`source_order_reference`, `source_suborder_reference`, `source_order_line_reference`, `fulfillment_delivery_evidence_reference`, `return_refund_evidence_reference`).
- Mismatch details where applicable (`mismatch_field`, `vendor_reported_value_evidence`, `cixci_source_value_evidence`).
- `resulting_invoice_exception_record_reference` (nullable).
- `matching_timestamp`.
- Existing baseline envelope.

**Implementation note:** for high-volume runs, this event MAY be emitted as a batch / summary at the end of matching rather than per individual row; the discriminator and structure are preserved. Concrete batching is implementation.

---

### Discriminator extension discipline

- Discriminator values are case-sensitive lowercase identifiers per existing baseline.
- Discriminator values are stable across schema versions (additions are additive; removals require explicit deprecation per baseline).
- Subscribers MUST handle unknown discriminator values gracefully.
- Discriminator extensions do NOT change event ordering, retry, or idempotency semantics.

### Reference-first context discipline

Per PR-A reference-first discipline:

- No concrete payload field shape is locked.
- All payload context references are to existing entity / record identifiers.
- Implementation owns concrete payload schema.
- Future API Governance Foundation PR locks concrete payload schema.

### What this event-contracts section intentionally does NOT lock

- Concrete payload field schema. Future API Governance Foundation PR.
- Concrete subscriber routing / filtering implementation. Implementation.
- Concrete event delivery transport. Existing Integration Management baseline + future Integration coordination.
- Concrete schema versioning policy beyond existing Invoice Management baseline.
- Concrete subscriber rate limiting / backpressure. Implementation.
- Future discriminator value additions beyond the catalogs documented here (existing additive discipline applies).
- Concrete event ordering guarantees beyond existing baseline.

### Existing Invoice Management event contracts preserved (no edits)

All existing Invoice Management event contracts are preserved without modification. The 11 new event contracts are additive.
