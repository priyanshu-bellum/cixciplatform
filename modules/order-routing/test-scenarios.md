# Order Routing Test Scenarios

## Functional Scenarios

1. Route a single-line accessory order to one vendor suborder.
2. Route a multi-line parent order into multiple vendor suborders.
3. Preserve parent order linkage across split-order suborders.
4. Route a device line using Device Reference without owning canonical device data.
5. Route future branded merchandise using Product Type without owning Product Type validation.
6. Create routing decision record and immutable routing snapshot.
7. Create downstream fulfillment instruction placeholder.
8. Carry warranty registration required signal placeholder on eligible order line.
9. Emit routing completed and suborder created events.
10. Evaluate route dry-run without creating execution state.
11. Execute route after successful evaluation and create snapshot/suborders.

## Vendor Routed-Suborder Export Scenarios

1. Create vendor order export eligibility record for routed suborders assigned to one vendor.
2. Eligibility record preserves routed suborder reference, vendor reference, buyer/entity reference, routing snapshot reference, eligibility status/reason, exclusion reason, inclusion rule version, export window, prior export state, re-export allowed flag, review-required state, supersession/cancellation state, and audit reference.
3. Eligibility record does not imply vendor receipt, vendor acceptance, external delivery, scheduled email delivery, or file evidence.
4. Include only routed suborders assigned to the vendor receiving the export.
5. Include only routed suborders eligible for vendor processing.
6. Exclude cancelled routed suborders.
7. Exclude superseded routed suborders.
8. Exclude routed suborders already exported in the same batch unless explicit re-export is requested.
9. Exclude routed suborders assigned to another vendor.
10. Preserve export window and source event/version reference.
11. Route ambiguous eligibility to review-required state.
12. Detect export eligibility conflict when routing snapshot is missing or stale.
13. Create routed suborder export batch reference with schema version, inclusion rule version, export window, vendor reference, generated timestamp, generated by actor/service, batch idempotency key, and audit reference.
14. Create vendor order export content reference without creating Logs & Audit file evidence inside Order Routing.
15. Map export statuses to the shared export vocabulary while preserving Order Routing-specific substates.

## Export Batch Item Scenarios

1. Create export batch item for each routed suborder considered for a vendor export batch.
2. Export batch item preserves routed suborder reference, routing snapshot reference, eligibility record reference, included/excluded status, included/excluded reason, source event/version, and audit reference.
3. Export batch item preserves prior export membership reference where the suborder was previously exported.
4. Export batch item preserves re-export reason when included through an approved re-export.
5. Export batch item uses duplicate prevention key scoped to routed suborder, export batch, prior export membership, source event/version, and re-export reason.
6. Duplicate export is blocked at routed-suborder/batch-item level, not only at batch level.
7. Batch item can reference resulting fulfillment handoff request where handoff is requested.
8. Batch item enters review-required state for duplicate prevention conflict, stale source event/version, cancelled suborder, superseded suborder, or missing routing snapshot.

## Buyer / Retailer Split Export Scenarios

1. Split vendor export by buyer/retailer where configured.
2. Split reference preserves split reference id, vendor reference, buyer/entity reference, split rule version, export batch reference, routed suborder references, routing snapshot references, review-required state, and audit reference.
3. Ensure split-by-buyer behavior does not alter routing decisions or parent order linkage.
4. Route split conflicts to review when buyer/retailer grouping conflicts with routing snapshots or vendor grouping.

## CSV-Only Vendor Order Export Scenarios

1. CSV-only vendor export does not require Excel worksheet tab naming.
2. Export file metadata or naming includes buyer name or reference, export date, batch id, vendor reference, and schema version where applicable.
3. Header and field expectations reference the shared Import / Export / Validation Governance standard and Order Routing export contract.
4. UPC, SKU, order numbers, routed suborder references, and external identifiers are preserved as text identifiers.
5. CSV-only export content remains owned by Order Routing while file tracking remains owned by Logs & Audit.

## Manual Vendor Order Download Scenarios

1. Vendor manual download is allowed only when Tenant Company scope and permission evidence allow it.
2. Manual download reference preserves manual download id, export batch reference, actor/user reference, vendor/company/entity reference, download timestamp, download count, last downloaded by, last downloaded at, permission/scope reference, and audit reference.
3. Repeated manual downloads are auditable and do not create duplicate export batches unless re-export is explicitly requested.
4. Manual download does not bypass export eligibility, tenant scope, redaction, or file tracking.
5. Logs & Audit owns immutable download evidence.

## Re-Export Scenarios

1. Re-export request preserves original export batch reference, original export batch item references, requested routed suborders, reason, requester, approval/reference placeholder, duplicate processing risk flag, allowed/blocked state, generated replacement export reference, supersession reference, and audit reference.
2. Re-export must be explicit and auditable.
3. Re-export preserves prior export membership and reason on replacement batch items.
4. Re-export does not silently re-send all previously exported suborders.
5. Partial re-export includes only explicitly requested and allowed routed suborders.
6. Re-export blocked by duplicate processing risk emits review/blocking evidence without creating replacement export batch.

## Routing-To-Fulfillment Handoff Scenarios

1. Order Routing creates fulfillment handoff request after route completion where appropriate.
2. Order Routing creates fulfillment handoff request after export batch item inclusion where configured.
3. Handoff request preserves fulfillment handoff request id, routed suborder reference, export batch item reference, routing snapshot reference, handoff requested timestamp, handoff source version, handoff idempotency key, duplicate handoff blocker, review-required state, and audit reference.
4. `order.routing.fulfillment-handoff.requested` is a request, not proof that Fulfillment/Returns accepted execution.
5. Fulfillment/Returns records handoff acceptance or rejection.
6. Fulfillment/Returns disposition reference is recorded on the handoff request with Fulfillment/Returns source version, accepted/rejected/ignored state, and applied vs ignored state.
7. Missing, duplicate, stale, rejected, or ignored Fulfillment/Returns disposition is not treated as fulfilled, shipped, delivered, or accepted.
8. Duplicate fulfillment handoff is blocked by handoff idempotency key or duplicate handoff blocker.
9. Fulfillment/Returns records vendor fulfillment updates.
10. Order Routing does not update shipment status, delivered status, tracking number, tracking URL, return status, refund evidence, or fulfillment execution state.
11. Handoff failure produces downstream handoff exception/review state without mutating fulfillment execution.

6. Apply precedence ladder: exception blocker, tenant scope, price snapshot, Product Type, target selection, fulfillment target, vendor export eligibility, export batch item duplicate prevention, fulfillment handoff request, warranty requirement, manual override.

## Exception / Retry Scenarios

2. Eligibility/scope exception routes to Tenant Company or relationship review queue.

4. Target availability exception is retryable when target outage is temporary.
5. Vendor export eligibility exception routes to review when export scope is ambiguous.

7. Re-export exception blocks when requested scope is too broad or duplicate processing risk is not approved.
8. Fulfillment handoff exception routes to review for duplicate handoff, missing disposition, stale disposition, rejected disposition, ignored disposition, or applied-vs-ignored conflict.
9. Unsupported Product Type exception blocks until Routing Policy supports it.
10. Warranty registration exception is blocking or warning according to policy.
11. Downstream handoff exception uses retry budget.
12. Manual review exception requires approval before supersession.
13. Retry succeeds and creates new routing snapshot/export/handoff reference where allowed.
14. Retry exhausts and emits review-required event.
15. Retry storm prevention opens circuit breaker placeholder.

## Tenant Isolation Scenarios

1. Child entity routing does not expose sibling child entity order lines.
2. Vendor suborder view does not expose other vendor suborders.
3. Vendor export for one vendor does not expose another vendor's routed suborders.

## Scalability Scenarios

2. Duplicate route execution with same idempotency key returns existing result.
3. Duplicate routed suborder dedupe key does not create a second suborder.

8. Async retry queue respects retry budget by exception family.
9. Replay window prevents stale routing event from triggering new execution.

13. Audit retention captures snapshot, policy version, exception, retry, supersession, export eligibility, export batch, export batch item, re-export request, manual download, handoff request, and disposition reference.

## Regression Scenarios

## Vendor Export Schedule and Delivery Evidence Scenarios (PR-A)

Lightweight architecture / review scenarios for PR-A's six workflows and the entity/contract surface. These are state assertions and boundary-shape scenarios, not implementation test cases. Implementation tests for each workflow are deferred to developer-owned test suites.

### Schedule lifecycle

1. A CIXCI System Admin holding Export Schedule Authority creates a Vendor Export Schedule in `draft` state with vendor reference, timezone, daily send times, recipient references, delivery method reference, and split behavior. The Schedule is not yet generating Windows.

2. Pre-activation validation fails when the timezone is not a valid IANA timezone; Schedule remains in `draft`.

3. Pre-activation validation fails when no daily send times are configured; Schedule remains in `draft`.

4. Pre-activation validation fails when a recipient reference cannot be resolved through Tenant Company `check_access`; Schedule remains in `draft`.

5. A validated Schedule transitions `draft → active`. Workflow 2 begins generating Windows over the configured horizon.

6. Editing an active Schedule increments `schedule_version`. The new version is the active configuration going forward.

7. Editing an active Schedule does NOT retroactively rewrite Windows materialized from prior versions. Those Windows complete under their materialization version.

8. Pausing an active Schedule transitions `active → paused`. Windows in `scheduled` state at pause time transition to `superseded`.

9. Windows in `executing` state at pause time complete normally; their Delivery Evidences resolve per Workflow 3.

10. Resuming a paused Schedule transitions `paused → active`. Window generation resumes from current time forward (no backfill of skipped Windows).

11. Retiring a Schedule transitions to `retired`. Terminal. No new Windows.

12. Vendor self-service Schedule editing is not enabled. A non-System-Admin actor attempting to edit a Schedule is rejected for missing required authority (`export_schedule_authority`).

13. Buyer / retailer / MVNO accounts cannot edit vendor export Schedules through any surface.

### Window lifecycle

14. Workflow 2 materializes Windows for each `daily_send_times` entry in the Schedule, resolved through the configured `timezone`.

15. When `business_calendar_reference` resolves to a calendar and `holiday_weekend_behavior = skip_to_next_business_day`, a Window scheduled for a holiday is advanced to the next business day's send time.

16. When `business_calendar_reference` resolves to a calendar and `holiday_weekend_behavior = deliver_anyway`, a Window scheduled for a holiday is materialized for the original date.

17. When `business_calendar_reference` does not resolve (no calendar exists, reference is stale), `holiday_weekend_behavior` defaults to `deliver_anyway` and `business_day_classification` is `unknown_no_calendar`.

18. A Window's `scheduled_execution_at` is captured at materialization and is immutable after the Window enters `executing` state.

19. A Window transitions `scheduled → executing` at `scheduled_execution_at`.

20. A Window produces zero or more Export Batches per the Schedule's `buyer_retailer_split_behavior`. With `split_by_buyer`, multiple Export Batches may be produced per Window.

21. A Window transitions `executing → succeeded` when one or more Export Batches are produced.

22. A Window transitions `executing → failed` when no Export Batches are produced. The `failed` state may trigger Review-Required on a placeholder Delivery Evidence per anomaly detection.

23. A `scheduled` Window may be `superseded` by Schedule pause, retirement, or version change. Superseded Windows are retained for audit.

### Delivery Evidence capture

24. Workflow 3 creates a Vendor Export Delivery Evidence in `pending` state for each Export Batch produced by a `succeeded` Window.

25. Multiple Delivery Evidences may exist for a single Window when `buyer_retailer_split_behavior` produced multiple Export Batches. Each Delivery Evidence references its specific Export Batch.

26. A Delivery Evidence carries `delivery_method_reference` and `recipient_references` from the Schedule at the Window's materialization version. Changes to the Schedule after materialization do not retroactively rewrite the Delivery Evidence.

27. Delivery Evidence remains in `pending` while Integration Management performs transport.

28. On a successful Attempt, Delivery Evidence transitions `pending → confirmed`. `export_delivered_timestamp` is populated to the Attempt's timestamp.

29. Pre-execution events do not create Delivery Evidence. No speculative Delivery Evidence is created during Window `scheduled` state.

30. Pre-commit / pre-execution flows do not raise `order-routing.export-delivery-evidence.*` events.

### Manual Download evidence

31. For Delivery Method = Manual Download, the Delivery Evidence is created in `pending` state at Window `succeeded` and awaits pickup.

32. When an authorized actor downloads the export, a Delivery Attempt is created with `attempt_outcome = success` and the Delivery Evidence transitions `pending → confirmed`. `export_delivered_timestamp` = download timestamp.

33. Subsequent downloads of the same Export Batch produce additional Attempts but do not transition the Delivery Evidence out of `confirmed`. The first successful download is the canonical confirmation.

34. If the Schedule's `manual_download_expiration_window` elapses without any download, Delivery Evidence transitions `pending → unconfirmable` and Workflow 6 sets `export_review_required_state = review_required`.

35. Manual Download authorization is gated by existing Order Routing Manual Download authority (not by PR-A's Export Schedule Authority).

### Retry / failure evidence

36. Integration Management performs delivery attempts. Each attempt produces a Vendor Export Delivery Attempt under the parent Delivery Evidence.

37. The Attempt's `attempt_outcome` captures Integration Management's reported result: `success`, `transport_failure`, `recipient_bounce`, `timeout`, or `aborted`.

38. PR-A does not specify retry policy. Integration Management decides whether and when to retry.

39. On retry, a new Attempt is created with `attempt_sequence` incremented; the prior Attempt's `retry_after_reference` is populated.

40. When Integration Management signals retry exhaustion, Delivery Evidence transitions `pending → failed` and Workflow 6 sets `review_required`.

41. Partial-success outcomes (some recipients receive, others bounce) transition Delivery Evidence to `partial`. `export_delivered_timestamp` is populated to the first successful Attempt's timestamp.

42. `aborted` Attempts (Order-Routing-initiated cancellation) transition Delivery Evidence to `failed` immediately; no further retries are expected.

### Operational Review-Required workflow

43. `export_review_required_state` defaults to `not_required`. It transitions to `review_required` only on explicit trigger.

44. Triggers for `review_required`: retry exhaustion, expiration of Manual Download window, partial-success per anomaly detection, Window failure with anomaly, schedule-level anomaly, explicit System Admin action.

45. A System Admin holding Export Schedule Authority transitions `review_required → under_review` to begin review.

46. Correction actions during `under_review` produce audit history but do not transition the state out of `under_review`. The state transitions only via `resolved` or reopening.

47. Resolution via re-export uses existing Order Routing re-export controls. The Review-Required state does not silently generate a replacement export.

48. Resolution via audit-evidenced acceptance (without re-export) requires complete evidence: actor reference, reason reference, affected entity reference, timestamp, audit reference. Missing evidence triggers validation failure `REVIEW_RESOLUTION_AUDIT_EVIDENCE_MISSING`.

49. Reopening (`resolved → review_required`) preserves full history.

50. The Review-Required state does not connect to Notification Platform Service. PR-A does not notify anyone that review is required.

### Authority and permission discipline

51. Export Schedule Authority is required for Schedule create / edit / pause / resume / retire actions.

52. Export Schedule Authority is required for explicit Review-Required state transitions.

53. Export Schedule Authority resolves through Tenant Company `check_access`.

54. Authority failure produces a different validation outcome than override-audit-evidence failure. Authority-missing is `EXPORT_SCHEDULE_AUTHORITY_REQUIRED` (rejected for missing required authority); audit-evidence-missing is `REVIEW_RESOLUTION_AUDIT_EVIDENCE_MISSING` (rejected for incomplete audit). These conditions are not mixed.

55. Existing Order Routing authority classes (Manual Download authority, re-export authority, handoff approval) are unchanged by PR-A and continue to gate their respective actions.

56. Fulfillment / Returns actors cannot transition Vendor Export Schedule, Window, Delivery Evidence, Delivery Attempt, or Review-Required state through any path.

57. Integration Management cannot directly modify Order Routing state. Integration Management reports outcomes; Order Routing records.

58. Buyers cannot perform any of the above actions.

### Event family discipline

59. Exactly 12 PR-A event names are introduced. Four Schedule lifecycle, three Window lifecycle, three Delivery Evidence, two Operational Review.

60. All PR-A event names follow `order-routing.<entity>.<verb-past-tense>`.

61. PR-A introduces no SLA-related events (no `late-fulfillment-import`, no `fulfillment-response-deadline`, no `same-day-cutoff-elapsed`).

62. PR-A introduces no shipment / tracking / delivery-of-physical-goods events.

63. Legacy Order Routing event names are preserved unchanged.

64. PR-A events are state-change events with categorical context. No event carries a `consumer_action_hint`-style command-shaped field.

### Event-contract discipline

65. PR-A event payloads are reference-first. No full Schedule or Delivery Evidence snapshot is embedded.

66. PR-A event payloads do not carry recipient PII. Recipient references resolve through Tenant Company.

67. PR-A event payloads do not carry raw export content. Export Batches are referenced.

68. PR-A event payloads do not carry buyer / retailer portfolio data.

69. PR-A event payloads do not carry tenant-specific eligibility data.

70. Every PR-A event has `eventId`, `eventType`, `eventVersion`, `occurredAt`, `redaction_class`, `audit_reference`, `vendor_reference`.

71. Default `redaction_class` for PR-A events is `tenant_scoped`.

72. Consumers must handle event replay idempotently (same `eventId` produces same downstream state).

### Boundary discipline

73. Order Routing produces audit references; Logs & Audit retains the immutable record.

74. Order Routing requests delivery; Integration Management performs delivery. Order Routing does not implement transport.

75. Order Routing references Tenant-Company-owned recipient scope; Order Routing does not validate recipient identity.

76. Order Routing references a business calendar; Order Routing does not own or define a business calendar entity.

77. Order Routing carries `same_day_cutoff_reference` and `after_hours_handling_reference` on Schedule; Order Routing does not evaluate cutoffs.

78. Order Routing does not mutate Product Catalog, Fulfillment / Returns, Tenant Company, Integration Management, Notification Platform Service, Logs & Audit, or Analytics / Reporting state under any path.

79. Fulfillment / Returns will read Delivery Evidence read-only (future Fulfillment / Returns PR-A). Fulfillment / Returns cannot mutate Order Routing state.

80. Analytics / Reporting will read Review-Required state read-only (future Cross-Module Summary Email PR). Analytics / Reporting cannot mutate Order Routing state.

### API surface discipline

81. PR-A's API placeholders are read-only. No PR-A API mutates Schedule, Window, Delivery Evidence, Delivery Attempt, or Review-Required state.

82. PR-A does not introduce OpenAPI schemas.

83. PR-A does not introduce concrete URL paths beyond Order Routing's existing API style.

84. PR-A does not introduce mutating endpoints. Mutations occur through internal CIXCI System Admin surface, existing Order Routing workflows, or system actors at execution time.

94. PR-A does not own return state.

99. PR-A does not modify any Integration Management, Notification Platform Service, Tenant Company, Logs & Audit, Analytics / Reporting, Product Catalog, or Device Catalog file.

### Confirmed delivery state does not assert operational acceptance

101. `delivery_confirmation_state = confirmed` asserts only that the Delivery Method produced confirmed delivery evidence. It does **not** assert that the vendor acknowledged the export.

102. `delivery_confirmation_state = confirmed` does **not** assert that the vendor opened or viewed the export.

103. `delivery_confirmation_state = confirmed` does **not** assert that the vendor processed the export.

104. `delivery_confirmation_state = confirmed` does **not** assert that the vendor accepted operational responsibility for the export.

105. `delivery_confirmation_state = confirmed` does **not** assert that fulfillment execution was accepted.

106. For Delivery Method = scheduled email: email delivered (per Integration Management / Notification Platform receipt) ≠ email opened. PR-A captures the delivery receipt only.

107. For Delivery Method = SFTP push: file transfer confirmed ≠ file consumption by vendor. PR-A captures the transfer confirmation only.

108. For Delivery Method = API push: HTTP response confirming receipt ≠ vendor system processed the payload. PR-A captures the transport-layer receipt only.

109. For Delivery Method = manual download: download successful ≠ operational acceptance. PR-A captures the download fact only.

110. Operational fulfillment acceptance state is owned by Fulfillment / Returns (future PR) and the Boundary/Handoff PR; PR-A does not model it.

## Cross-Module Handoff Publication Scenarios (Boundary/Handoff PR)

Producer-side scenarios for the Order Routing -> Fulfillment / Returns handoff. Each scenario describes what Order Routing commits to as the producer; consumer-side scenarios (idempotency, replay, eligibility) live in `modules/fulfillment-returns/test-scenarios.md`.

### Publication on `confirmed` state transition

1. A Vendor Export Delivery Evidence transitions to `confirmed`. Order Routing publishes `order-routing.export-delivery-evidence.confirmed` with the stable payload references per PR #91. Publication is at-least-once; duplicates may occur at the transport layer.

2. The `eventVersion = v1` is unchanged. The Boundary/Handoff PR does not bump versions.

3. Payload references (`vendor_export_delivery_evidence_reference`, `vendor_reference`, `vendor_export_window_reference`, `export_delivered_timestamp`, `delivery_method_reference`) are stable post-emission. Consumers may rely on them for idempotency key derivation.

### Publication on `failed` state transition

4. A Vendor Export Delivery Evidence transitions to `failed`. Order Routing publishes `order-routing.export-delivery-evidence.failed`. Same commitment profile as `.confirmed`.

5. The `export_delivered_timestamp` on `.failed` carries the latest Attempt timestamp per PR #91.

### Source state machine ordering

6. Within a single Vendor Export Delivery Evidence's lifecycle, the event order is naturally `created -> confirmed` or `created -> failed` per PR #91. Transport may reorder events across different source records; ordering within a single source record is preserved at the source.

7. PR #91's terminal-once-confirmed invariant: once a Delivery Evidence is `confirmed`, the source state does not transition. Out-of-order `failed` after `confirmed` for the same source record is impossible at the source.

### Re-export semantics

8. A re-export per PR #91 existing re-export controls produces a **new** Vendor Export Delivery Evidence with a new `vendor_export_delivery_evidence_reference`. The new record produces its own publish stream (`.created`, then `.confirmed` or `.failed`). The prior record's terminal state is preserved; no retroactive mutation.

9. Consumers receive two distinct publish streams for two distinct source records. Each produces its own consumer-side Handoff Record per Workflow A.

### No retroactive mutation

10. PR #91's invariant: `confirmed` and `failed` are terminal source states. Order Routing does not edit terminal source records. Corrections go through re-export.

11. Consumer-side replay-time eligibility invariant (Fulfillment / Returns side) depends on this: the original eligibility decision captured on a consumer Handoff Record can be honored across replays only because the source record (and therefore the payload references) does not change.

### No producer-side consumer state

12. Order Routing does not track whether any consumer has observed any published event. Consumer Handoff Records are Fulfillment / Returns-owned. Order Routing reads no Fulfillment / Returns state.

13. Producer-side observability of consumer handoff outcomes happens via cross-reference (consumer's SLA Evaluation Record carries `vendor_export_delivery_evidence_reference` back; querying Fulfillment / Returns by source reference shows handoff outcome). Order Routing does not aggregate.

### No acceptance implication

14. `delivery_confirmation_state = confirmed` is a transport-layer fact. It does not assert vendor acknowledged, opened, processed, or accepted the export, and does not assert fulfillment execution was accepted. PR #91's clarification is reaffirmed by this PR.

15. Email delivered != email opened. SFTP push confirmed != file consumed. Manual download != operational acceptance. API push confirmed != vendor system processed.

### What this PR does NOT add to Order Routing

16. No new Order Routing entity. The Cross-Module Handoff Record is Fulfillment / Returns-owned; this is consistent with Order Routing not owning consumer state.

17. No new Order Routing event. PR #91's 12 events are unchanged.

18. No new Order Routing workflow. PR #91 Workflow 3 (Vendor Export Delivery Evidence Capture) remains the canonical producer flow.

19. No new Order Routing permission or authority class.

20. No new Order Routing API contract placeholder.

21. No modification of `modules/order-routing/data-model.md`, `modules/order-routing/workflows.md`, `modules/order-routing/events.md`, `modules/order-routing/permissions.md`, or `modules/order-routing/api-contracts.md`.

22. `modules/order-routing/vendor-export-fulfillment-handoff-governance.md` is modified with a light cross-reference only (<=15 lines added, 0 removed, no restructuring, no ownership changes, no SLA logic, no new event names, no new workflow detail).
